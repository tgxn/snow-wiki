1 Overview

Configuration automation extensions allow you to customize a ServiceNow instance to support configuration automation providers in addition to those offered by ServiceNow. Change manager extensions scripting is available to users with the Chef Administrator [chef_admin] or Puppet Administrator [puppet_admin] role.

Note: A configuration template was called a node definition in earlier versions of the system.

To support a new provider, you need to define:

  • The change manager script
  • A difference by CI extension script
  • A difference by node definition script
  • Workflows, if needed
  • The change manager
  • The provider

Configuration automation extensions are available starting with the Eureka release.

2 Defining the Change Manager Script

The change manager script is responsible for the logical change process and exposes customization hooks for provider implementations. which are scripts that determine how the assets are handled. For reference information on this script include, see Change Manager Script Include.

To define a new change manager script:

  1. Create a new script that extends the CfgAutoChangeManager script include.
  2. Implement the following mandatory functions:
    • isAuthorized(GlideRecord record)
    • isChangeRequired()
    • getChangeId(GlideRecord current)
    • setChangeId(GlideRecord record, String changeId)
    • prepare(GlideRecord record)
    • finalizeChange(GlideRecord record)
    • getWorkflowVariables(String workflowName, GlideRecord record)
    • getPushWorkflowName()
    • cleanupAfterCancelChange(GlideRecord record)
    For detailed information on each function, see Change Manager Script Include.
  3. Implement the optional getPushWorkflowName() override function if it applies to the provider.

2.1 Example Change Manager Script

var ChefNodeDefinitionChangeManager = Class.create();
ChefNodeDefinitionChangeManager.prototype = Object.extendsObject(CfgAutoChangeManager, {
    // Code body omitted for brevity
});

3 Defining a Difference by CI Extension Script

The Difference by CI Extension script generates the differences between the current CI definition and upcoming changes. These differences are inserted into the change request detail. Extend the ConfigAutomationDiffBase script include to create this script and implement the process(String configuration_automation_changeId) method, which is called to generate the diff message.

4 Defining a Difference by Node Definition Script

The Difference by Node Definition script generates the differences between the current configuration template and upcoming changes. These differences are inserted into the change request detail. Extend the ConfigAutomationDiffBase script include and implement the processString draft_node_definition_Id) method, which is called to generate the diff message.

5 Defining Workflows

Workflows are optional depending on the requirements of the new configuration automation provider. For example, Puppet does not use workflows; instead, the Puppet Master pulls the changes from the instance. Chef, however, uses the following workflows to pull changes:

Workflow Description
Push CI Change to Chef Server Pushes the configuration automation change to the Chef server.
Push Chef Changes By Node Definition Pushes configuration automation changes triggered by a configuration template change.

If the new configuration automation provider requires workflows, register the workflows in Configuration Automation Provider form, and make sure to implement the getPushWorkflowName() and getWorkflowVariables(String workflowName, GlideRecord record) method in the change manager script.

6 Defining the Change Manager

To define a change manager:

  1. Define the script.
  2. Register the script.
  3. Configure the change manager in the Change Manager module.

The change manager is a script that manages changes between ServiceNow configuration templates or CIs and the definitions on the configuration automation provider server. Use the CfgAutoChangeManager script include as your starting point, and implement the required functions so that they provide the necessary information. Register your script in the Config Automation Provider [config_automation_provider] table.

  1. Navigate to Configuration Automation > Extension > Change Managers.
  2. Click New.
  3. Fill in the fields, as appropriate (see table).
  4. Click Submit.
Configuration Automation Change Manager

Field Description
Provider The name of the configuration automation provider.
Change Manager The script that defines how to manage the configuration information for this provider. This is the script defined in Defining the Change Manager Script.
Table The table that contains the configuration automation information.

7 Defining the Provider

The Configuration Automation Provider form gathers all the the script and table information required by the configuration automation provider.

  1. Navigate to Configuration Automation > Extension > Providers.
  2. Click New.
  3. Fill in the fields, as appropriate (see table).
  4. Click Submit.
Configuration Automation Provider

Field Description
Name The name of the new provider.
Description A description for the new provider.
Difference Extension by CI The script that defines the difference extension by CI. This is the script defined in Defining a Difference by CI Extension Script.
CI Push Subflow [Optional] The sub workflow the defines how changes are pushed for a CI, if required by the configuration automation provider.
Difference Extension by Node Definition The script that defines the difference extension by configuration template, created in Defining a Difference by Node Definition Script.
Node Definition Push Subflow [Optional] The sub workflow, if required by the configuration automation provider, that defines the changes pushed to a configuration template.
