1 Overview

A set of six user management activities enables organizations to automate their on-boarding/off-boarding processes with auditable, self-documenting workflows that save time and eliminate mistakes. The Orchestration Active Directory activities in this release are designed to manage user accounts and reset user passwords. The following activities cover the most common use cases administrators encounter when managing Active Directory user accounts:

  • Create AD User Account
  • Update AD User Account
  • Remove AD User Account
  • Disable AD User Account
  • Query AD
  • Reset AD User Password.

These activities share a common design, have complementary functionality, and share a common set of parameters. They can be used singly or together to create consistent workflows for provisioning and de-provisioning user accounts.

2 Example

An organization plans to make their ServiceNow instance the single system of record for user account data and wants to update Active Directory with the latest changes. Their solution is to create an Orchestration workflow that pushes changes from the ServiceNow user record down to the Active Directory to create a new user record or update an existing record. This is accomplished by creating an Orchestration workflow that can create and update records in Active Directory based on the data in the ServiceNow User [sys_user] table. However, if you want the workflow to create an Active Directory group or add users to an Active Directory group, you can create a custom activity. For information on assembling a workflow see Creating a Workflow.

Note: This example workflow assumes that ServiceNow is configured for LDAP and an LDAP server is configured to accept the new user accounts. The Active Directory user management activities are not dependent on LDAP, but the presence of LDAP makes this example workflow much easier. You must provide the domain controller's IP address to the workflow, either by hardcoding it, adding another workflow input, or using a script to look it up from the CMDB.



  1. Navigate to Workflow > Workflow Editor.
  2. Create a new workflow using the following variables:
    • Name: Snyc AD User
    • Table: Global [global]
  3. Click Submit.
    A basic workflow with a start and end point appears on the canvas.
  4. Click the arrow beside the gear icon in the upper left corner of the canvas and select Edit Inputs from the menu.


  5. In the Variables record list, click New, and create a new variable with the following values:
    • Column name: u_user
    • Label: User
    • Type: Reference
    • Reference: sys_user.
    • Attributes: allow_invalid=true
  6. Click Submit.
  7. Expand the folder for Orchestration - Active Directory in the Activities pane.
  8. Drag and drop the Update AD User Account activity onto the transition line between the Begin and End points of the new workflow.
    This action automatically links the activity with the end points and opens the Activity Properties form.
  9. Complete the properties form as follows:
    • Name: Enter a logical name such as Update user data.
    • Domain Controller: The ServiceNow LDAP integration adds a reference to the LDAP server to which the user's account is linked. To identify the LDAP server, enter the following statement:
    ${workflow.inputs.u_user.ldap_server.server_url}
    • Username: This example assumes that the ServiceNow user name matches the Active Directory sAMAccountName. Enter the following:
    ${workflow.inputs.u_user.user_name}
    • User Data: Set the first name, last name, and title of the user. The following expression pulls the necessary data directly from the ServiceNow user record and writes it to a JSON object.
    {
    "givenName" : "${workflow.inputs.u_user.first_name}",
    "SN" : "${workflow.inputs.u_user.last_name}",
    "title" : "${workflow.inputs.u_user.last_name}"
    }
  10. Click Submit.
    The workflow looks like this:


  11. Attach both activity outcomes (Success and Failure) to the end point.
    At this point, the workflow takes a ServiceNow user record as input and updates the First Name, Last Name, and Title of the corresponding Active Directory account. If the account does not exist in Active Directory, the workflow fails.
    Note: In a normal workflow, some type of alternate action is desirable upon failure. For example, you might send an email notification if the workflow failed to update the record.
  12. To prevent the workflow from failing, add a Create AD User activity to the transition lines between Begin and the Update AD User Account activity.
  13. Complete the properties form as follows:
    • Name: Enter a logical name such as Create user data.
    • Domain Controller: Same as for the update activity. ${workflow.inputs.u_user.ldap_server.server_url}
    • Username: Same as for the update activity. ${workflow.inputs.u_user.user_name}
    • User Data: Leave this field empty for this example. This creates an account with only the user name in it.
  14. Click Submit.
  15. Connect the Failure outcome of the Create AD User activity to End.
    For this example, we are ignoring errors. The workflow now looks like this:


    This procedure builds a simple workflow that creates a bare-bones Active Directory account consisting of a user name only. The workflow then updates that account with additional information provided by the ServiceNow User [sys_user] table. However, we do not want to execute the Create AD User activity if the account already exists. The best practice is to query Active Directory for matching user records and then branch the workflow based on the results of that query. If an account already exists, then the workflow updates the account. If the account does not exist, then the workflow creates the account in Active Directory.
  16. Drag and drop the Query AD activity onto the transition between Begin and Create AD User Account.
  17. Complete the properties form as follows:
    • Name: Enter a logical name such as Search for existing account.
    • Domain Controller: ${workflow.inputs.u_user.ldap_server.server_url}
    • Properties: A comma-separated list of Active Directory properties to return. For example, givenName, SN, title. If the parameter field is blank, then all properties are returned. In this workflow, we leave the field blank.
    • Search Filter: An LDAP filter string that defines the search parameters. Use any valid LDAP filtering criteria. To find user accounts matching the input record, we use:
    (samaccountname=${workflow.inputs.u_user.user_name})
  18. Click Submit.
  19. Connect the Failure outcome for the query activity to the End point.
    Remember that we are ignoring errors in this workflow.
  20. Connect the Success outcome of the query activity to the Update AD User Account activity.
    The workflow now looks like this:


    The Query AD activity returns its results as a JSON string in the workflow scratchpad (as workflow.scratchpad.queryAD). This JSON string is always an array of objects. Each object corresponds to an Active Directory entry that matched the query. Our workflow should branch, whether that array is empty or not.
  21. Drag a standard If activity from the Conditions folder and drop it onto the transition between Query AD and Update AD User Account.
  22. Complete the properties form as follows:
    • Name: Enter a logical name such as If this account exists.
    • Advanced: Select this check box to open the Script field.
    • Script: For the If activity to work correctly, we must return a yes or no in the answer variable (var) which corresponds to the Yes and No outcomes of the query activity. Line 1 converts the Query AD results from a JSON string into a Javascript array called queryResults. Line 2 checks the length of that array. If the array is more than 0, a match to the account was found, and we set our answer to yes. Otherwise, the answer is no.
    var queryResults=new JSON().decode(workflow.scratchpad.queryAD);
    answer = ( queryResults.length>0 ? 'yes' : 'no' );
  23. Click Submit.
  24. In the canvas, create a transition from the Yes outcome of the If activity to the Update AD User Account activity.
  25. Create a transition from the No outcome of the If activity to the Create AD User Account activity.
    This is the final step. This workflow will query Active Directory to determine if an account already exists. If an account exists, the workflow updates that account. If an account does not exist, the workflow creates the account and then updates the Active Directory with the configured User Data set.
