1 Overview

 Note: Before reading this page, be familiar with the ServiceNow rollback activities detailed on the Approval and Rollback Activities page.


When conditions in a workflow trigger a Rollback or Rollback To activity, the workflow moves processing backward to a specified activity in the workflow and resets certain activities that have already executed back to their original state. This is useful when handling an unexpected failure or as part of a programmed logical flow. When an activity is reset during a workflow rollback, the following things happen:

  • Approvals are reset to Not Requested.
  • Tasks are reset to either Open or Pending. A rollback workflow path cannot create new tasks.

Activities that perform external system operations such as deleting a file or sending an email are not rolled back. Only approval and task activity states are reset.

The Rollback activity is deprecated starting with the Eureka release. Use the Rollback To activity instead.

2 Comparing Rollbacks

A workflow can contain a single rollback, multiple rollbacks, or nested rollbacks in more complex workflows. ServiceNow workflows offer two rollback activities, Rollback and Rollback To, for use in different situations. Both rollback activities define the same rollback target, but reset different activities. The Rollback activity updates all activities that have executed between the rollback and the transitioned to activity since the most recent Rollback execution or, if no Rollback has executed, since the Begin. This can include those activities not directly joined by transition lines. Depending on various factors, including slow approvals by users, execution times can vary, causing Rollback to reset the state of more activities than anticipated. The Rollback To activity eliminates this uncertainty by resetting activities based on the actual workflow sequence (transition line attachments) of activities between itself and the transitioned to activity, rather than using the execution order to determine where processing should restart.

2.1 Rollback Behavior

The Rollback activity should be used only in workflows in which a single rollback is used. Activities are reset once, and then processing begins again, giving all tasks and approvals the opportunity to be resolved before the workflow completes. If Rollback is used to trigger multiple or nested rollbacks, some reset activities are left unresolved by repeated rollbacks which cannot include activities that were reset by previous rollbacks in the execution order.

In the example here, the execution timeline is:

(0) Begin
(1) Log Message
(2) Task 1
(3) Task 3
(4) Task 2
(5) Timer
(6) Approval 1
(7) Roll back to first task
(8) Task 1
(9) Task 2
(10) Approval 1
(11) Log approval
(12) Join
(14) Send email
(15) End


2.1.1 Transition History

This workflow resets activities outside the transition lines and rolls back to the beginning. Notice that Task 3 is reset but can remain in a Pending state even after the workflow has reached the end. It is not necessary to close Task 3 to complete the workflow, which means that it might be overlooked. To see what activities were rolled back, select the Workflow Transition History Related List and look at the Rolled back column.

The Rollback activity (7) resets the state of the following activities:

(2) Task 1: reset to Open
(3) Task 3: reset to Pending
(4) Approval 1: reset to Not Yet Requested
(5) Task 2: reset to Pending


2.2 Rollback To Behavior

The Rollback To activity transitions directly to the activity specified by the transition line arrow and has more predictable results than the Rollback activity. Use the Rollback To activity for all workflows that use multiple or nested rollbacks. Rollback To resets the targeted task (the direct transition) to Open. All tasks that have executed between the Rollback To activity and the targeted task (rolled back task) are set to Pending. For sample workflows that use Rollback To, see Designing Workflows with Rollback To.

In the example here, the execution timeline is the same as with the Rollback activity:

(0) Begin
(1) Log Message
(2) Task 1
(3) Task 3
(4) Task 2
(5) Timer
(6) Approval 1
(7) Roll back to first task
(8) Task 1
(9) Task 2
(10) Approval 1
(11) Log approval
(12) Join
(14) Send email
(15) End


2.2.1 Transition History

The state of (3) Task 3 does not change, since this activity does not directly transition from the rollback target activity. Contrast this with the Rollback activity, whose execution path returns to (0) Begin. To see what activities were rolled back, select the Workflow Transition History Related List and look at the Rolled back column.

The Rollback To activity (7) updates the following activities:

(8) Task 1: reset to Open
(9) Task 2: reset to Pending
(10) Approval 1: reset to Not Yet Requested


3 Resetting Activities with Go To Transitions

A third option for transitioning backward in a workflow is to use a simple connector arrow (Go to) which returns to a previous point in the flow and resets to Open only the activity to which it connects directly. Use this type of transition instead of a rollback activity to avoid unwanted resets. The Go to connector resets the state of an existing task or creates a new task if one does not already exist. All task and approval states that have executed between the activity from which the Go to transition originates and the target activity remain unchanged.

In this example, the workflow progresses through four stages of acquisition: Research, Purchasing, Receiving, and Inspection. A rejected approval for any of the first three stages results in a rollback to the top of that flow (the If activity) and resets the task and approval for that stage. If the Inspection approval is rejected, the workflow execution returns directly to the Purchasing Task with a Go to connector that resets the task. No other tasks or approvals are reset. If the workflow uses a rollback activity instead of a Go to connector, all the tasks and approvals between the Inspection approval and the Purchasing Task are reset and left in an unresolved state when the workflow ends.
