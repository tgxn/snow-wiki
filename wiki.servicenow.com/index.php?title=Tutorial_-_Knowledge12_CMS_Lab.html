1 Overview

Presenter: Mark O'Donnell, Chris Hahn, Tyler Jones

Audience: CMS Administrators, Webmasters

Abstract: Learn best practices by creating a site template within the Content Management System (CMS) based on the ESS Portal. The ESS Portal will be used as a reference model, and content is aimed at system administrators or web developers with hands-on ServiceNow experience and strong HTML and CSS skills.


Welcome to the wiki companion to the Content Management System (CMS) mini-class, first presented at Knowledge12 in New Orleans. We will be using the integration points page as an extended reference for all of the exercises provided in this class. The first exercises will address page template and layout structure of the site. Then focus on navigation and content by static and dynamic linking into the knowledge base, the service request catalog, and the business service portfolio. Once page structure, page template use, navigation, and content population are covered, we will address design techniques and long-term maintenance of the site.

2.1 CMS Language Support

Making translation strings work seamlessly in CMS has been a hot topic for the past year, for Berlin we took the opportunity to look at how languages were rendering within the CMS and make optimizations wherever possible.

  • goto: yoursystem/ - Open Plugins under the "System Definition" application, in the "Go to" filter set to "Name" type in i18n, make sure the "I18N: Japanese Translations" and "I18N: Knowledge Management Internationalization Plugin v2" plugins are active. In the Language drop-down menu in the upper right hand side of the system toolbar select "Japanise" then refresh the tab that has the ESS Portal. Browse the Portal sections and see language strings get pushed to their proper places.

For those who are on Aspen what does this mean and why is this unique and worthy of discussion because we already have translations in our CMS site? Localizing a CMS site was a manual process and based on the nature of the menu blocks, and we needed to consider the content along with where the strings were being stored. A big change on the database level moves the majority of the strings that you would associate with the site wireframe (menu section names, menu item names, site breadcrumb names, link names, footer menu links) to a table that allows you to define a translation string in a single place (translated_field table) versus generating a unique string (translated_text table) for each menu item or menu section. Knowing that site menus provide redundant linking to the most important pages or sections in your site, it made sense to limit the times you had to translate the strings involved.

  • translated_text - is used for a unique string translation that may not be used in other parts of the site or system
  • translated_field - should be used for strings that are shared and commonly used within the site. For example concepts such as "Save" or in the ESS Portal "Knowledge" and "Order Things"


When working, if you have any questions about where specific translation are stored, right click on the label and click the "Show - field_name" at the the bottom the context menu.

4.3 Service Catalog

Instead of building out a Page for every Category we want to build a set of pages which use the existing Catalog data model (Specifically Categories and Catalog Items).

We will require the following pages:

  • Catalog Entry Page
  • Catalog Category Page
  • Catalog Item Page

4.3.1 Catalog Entry Page

The first page needed for a Service Catalog is an entry page, to allow the user to drill-down further into the Categories.

The page should include the following blocks:

  • Header - The standard site header
  • New Items List - A list of recently created Catalog Items
  • Category List - A list of Categories which have been flagged to be shown in the CMS
4.3.1.1 New Items List

Create a new List, with the following attributes:

  • Name: New Items
  • Type: list_simple
  • Table: Catalog Item [sc_cat_item]
  • Order: Created
  • Order direction: Descending
  • Condition: Class is Catalog Item and Active is true
4.3.1.2 Category List

Create a new List, with the following attributes:

  • Name: Categories
  • Type: list_simple
  • Table: Category [sc_category]
  • Order: Title
  • Condition: Show in CMS is true and Active is true
4.3.1.3 Create the page
  • Copy the page from an existing Page belonging to the same site, set the following attributes:
    • Name: Catalog Entry

answer = getItemsForCurrentCategory()

function getItemsForCurrentCategory() {
   
   var documentKey = RP.getParameterValue('sysparm_document_key');
   documentKey = documentKey.split(',');

   var documentTableName = documentKey[0]
   var categoryID = documentKey[1];
   
   var catalogItems = new GlideRecord('sc_cat_item');
   catalogItems.addQuery('category', categoryID);
   catalogItems.addActiveQuery();
   catalogItems.orderBy('name');
   catalogItems.query();

   return catalogItems;
}

4.3.4 An enhancement

The Product Owner has reviewed the first iteration of the Catalog pages, and would like the List of Catalog Items shown on a Category Detail page to also contain the Short Description.

To do this, we will need to use a custom List Definition which does not render the Items using the Summary Template in the Content Type. Instead we will put the markup directly in the List Definition.

  • Enter the following details:

    • XML:
<?xml version="1.0" encoding="utf-8" ?>
<j:jelly trim="false" xmlns:j="jelly:core" xmlns:g="glide" xmlns:j2="null" xmlns:g2="null">
   <div class="sc_cat_item_list">
      <g:for_each_record file="${current}" max="${jvar_max_entries}">
         <div>
            <img class="icon" height="16" src="images/icons/catalog_item.gifx" width="16" />
            <span>
               <a href="catalog.do?uri=com.glideapp.servicecatalog_cat_item_view.do%3Fsysparm_id%3D${current.sys_id}%26sysparm_no_search%3Dtrue&amp;sysparm_document_key=sc_cat_item,${current.sys_id}">
                  ${current.name.getDisplayValue()}
               </a>
            </span>
            <div class="short_description"><!-- Replace this comment with your code --></div>
         </div>
      </g:for_each_record>
   </div>
</j:jelly>
