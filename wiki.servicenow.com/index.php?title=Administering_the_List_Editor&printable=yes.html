1 Overview

The list editor allows users to edit field values directly from a list without navigating to a form. Administrators can manage this feature by using the following options:

  • Configure global properties
  • Configure list control settings for a table
  • Configure contextual security rules
  • Manage user preferences


 Warning: Currently, Client Scripts and UI Policy run on forms only (client-side) and do not apply to list editing. Allowing list editing with Client Scripts running on fields in a form can result in incorrect data being saved to the record. For systems in which Client Scripts or UI Policies apply to forms, either disable list editing or create appropriate Business Rules or Access Control to control the setting of values in the list editor. See Setting Field Values in Forms and Lists for information on managing form and list editing.

2 Configuring List Editor Properties

To configure list editor properties, navigate to System Properties > UI Properties.

Name Description
glide.ui.list_edit Enable list editing. Default value is true, which globally enables the list editor. To disable the list editor globally, set this value to false.
glide.ui.list_edit_ignore_types List of element types (comma-separated) that cannot be edited in the list editor. In addition to field types you specify with this property, certain fields are not editable from a list.


2.1 Fields that are not editable from a list

The following fields are not editable from the list editor:

  • Fields of type:
    • Conditions [conditions]
    • Currency [currency]
    • Document ID [document_id]
    • Field List [field_list]
    • HTML [html]
    • Image [user_image]
    • List [glide_list]
    • Price [price]
    • Template Value [template_value]
    • Time [glide_time]
    • User Roles [user_roles]

3 Configuring List Control Settings for the List Editor

List control settings customize the behavior of list functions for a table.

To configure the list control settings that affect the list editor:

  1. Navigate to a list view for the desired table.
  2. Right-click the header bar and select the appropriate option for your version:
    • Fuji or later: Configure > List Control
    • Eureka or earlier: Personalize > List Control
  3. Select the desired settings (see table).
  4. Click Update or Submit.
Name Description
List edit type Controls the ability of a user to edit values directly in individual cells in a list. The options are:
  • Save immediately (cell edit mode): enables cell editing. The entire row is saved as soon as the user enters a new value.
  • Save data by rows: enables cell editing. The row is saved only when the user navigates away from the row or clicks the Save () button. This mode allows the user to modify multiple values before saving a record.
  • Disable list editing: prevents users from editing cells in the list.
List edit insert row Controls the ability of a user to create new records in list view. Select the check box to enable or clear the check box to disable this ability. When it is enabled, an empty row appears at the bottom of the list.
Insert a new row (record)

4 Configuring Contextual Security for the List Editor

The list editor enforces existing access control rules (ACLs) and additional security controls to restrict editing from a list.

The list_edit security operation specifically controls the ability to edit information in a list. Apply this operation in the same manner as the write operation to limit list editing for fields that require the user to navigate to the form. Reasons you may require a user to edit a field in a form include complex UI policy constraints or other policies you have in place.

To use the list editor for a particular field, the user must have the following access rights:

  • write and list_edit access to the field
  • write and list_edit access to any dependent fields of the field
  • write and list_edit access to any fields that are dependent on the field being edited

The following examples use the list_edit security operation to restrict list editing in certain contexts. To configure access controls, navigate to System Security > Access Controls.


4.1 Example 1. Restrict a Table

This access control prevents everyone from editing all fields in the Incident table in a list.

  • Type: record
  • Operation: list_edit
  • Name: Incident [incident]
  • Admin overrides: Clear the check box.
  • Script: 
answer = false;
Restrict the incident table


4.2 Example 2. Restrict a Field

This access control prevents everyone except an administrator from editing the Short Description field of an incident record in a list.

  • Type: record
  • Operation: list_edit
  • Name: Incident [incident], Short Description
  • Admin overrides: Select the check box.
  • Script: 
answer = false;
Restrict the incident short description


4.3 Example 3. Restrict a Field with a Condition

This access control prevents everyone from editing a Critical Incident in a list. It is defined by a condition.

  • Type: record
  • Operation: list_edit
  • Name: Incident [incident]
  • Admin overrides: Clear the check box.
  • Condition: Priority is not 1 - Critical
Restrict critical incidents


4.4 Example 4. Restrict a Field with a Script

This access control prevents everyone from editing an incident with a category of Software in a list. It is defined by a script.

  • Type: record
  • Operation: list_edit
  • Name: Incident [incident]
  • Admin overrides: Clear the check box.
  • Script: 
if (current.category == 'software')
answer = false;
else
answer = true;
Restrict software incidents


5 Managing User Preferences

Users can set user preferences for list editing by personalizing a list (available when the list mechanic is activated).

To view and manage list editor user preferences, navigate to User Administration > User Preferences.

Name Description
list_edit_enable Set the value to true to enable or false to disable list editing. Default value is true.
list_edit_double Set the value to true for double-click or false for single-click. Default value is true.
