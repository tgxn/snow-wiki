Legacy:Logica (Remedy) Integration

1 Overview

The service management integration between ServiceNow and the Remedy system at Logica is a two-way web service integration.

Note: The following integration is not part of the supported platform. The integration may require additional customization, typically performed by an experienced administrator or consultant, prior to operating correctly.


2 Business Communication Flow

The following diagram outlines the relationship and communication flow between components of the system.

This integration involves only two of the systems listed below:

  • Client Integration Tools – this the integration set of components on the ServiceNow side
  • Logica SMI Hub Integration Layer (BAO - BMC Atrium Orchestrator) – this is the set of messages and scripts used to map and handle communication between Remedy and ServiceNow at the Logica side.

3 Communication Messages

The integration is based around the concept of messages between Logica and ServiceNow. The messages are built around business cases and agreements between the partner and Logica.

Some messages can originate in either side (Remedy or ServiceNow). Other messages can only originate from one side.

The following diagram below outlines those messages.

Due to the fact that this integration is bi-directional, we handle both outgoing and incoming messages in this integration.

4 Basic Architecture

The service-management record types that we support in this integration currently are:

  1. Standard Incidents
  2. Request Tasks

For messages that are outgoing from ServiceNow, we use a number of Business Rules. For the most part, the business rules monitor the “Task” table in ServiceNow. This table handles all types of service-management records (eg. Incident, problem, change, task, etc). In some cases, however, we have business rules specifically for Incident since incident has a few unqiue fields that are important to the integration.

For messages coming into ServiceNow from the Logica SMI Hub, we use a Web Service Import Set. This import set employs a number of Mapping Tables with Transform Scripts. The maps query the “MessageType” field in the incoming XML message to determine which map to use. Each message uses its own Transform Map/Script.

5 The Integration

The entire integration for the Logica SMI hub is encapsulated in an application called: “Integration – Logica SMI”. The application contains a number of modules that each play a role in the integration. Each module will be explained below.

Properties – This module contains the settings for the integration

Script Includes – These are the libraries used by the entire integration. There are four principle libraries:

  • LogicaSoapEnvelope – This library assists in the creation of the outbound SOAP messages being sent from the integration
  • LogicaSoapMessageHeader – this class is consumed by the LogicaSoapEnvelop class and assists in the SOAP Message creation tasks
  • LogicaLogger – This is a special logger class specific to the integration. It provides log messages to the log with a source of “Logica”. Some of its options are configurable in the “Properties” module inside of the application.
  • LogicaHelper – The class contains a variety of helper methods that are often common across libraries, business rules, and transform scripts.

Businesss Rules – These are the business rules associated with the integration. They handled the outbound messages being sent to Logica. They are principally triggered from actions on the task table. However, for fields that are specific to individual incidents, etc, there are a few that are triggered from those tables directly.

Incoming Logica Message – This is the incoming SOAP web service for the integration. It contains a number of transform scripts. Each script looks for a message identifier field inside of an incoming message to determine which script(s) will be executed. The Logica SMI Hub points to this web service for their outbound messages to ServiceNow.

Network Connectivity – this is a tool that can be used to determine if the instance can communicate with specific IP Addresses or URL’s. It is helpful when trying to ensure that the given instance is able to communicate with the Logica Hub Servers. This module requires the use of a Local MID Server on the instance.

Check SNC Network Address – This is a very small tool that shows how external web pages view the instance. This is helpful in determining what IP Address is used for communication between the instance and the general internet.

6 Requirements

The integration requires the following plug-ins:

  • System Web Services
  • Web Service Consumer
  • Web Service Import Set Tables
  • Web Service Import Sets
  • Web Service Provider – Common

7 General Notes

Soap Attachments – your instance must have some form of a 2010 release of ServiceNow installed in order to take advantage of sending attachments to ServiceNow through a SOAP interface.

Large Soap Attachments – If you discover that small attachments come through into the ServiceNow instance properly, but larger attachments appear to be corrupted, you may need to change a system property. The property, glide.soapprocessor.large_field_patch_max, is an integer that represents a SOAP message size in Bytes. When a SOAP Message size exceeds this setting, then the message is stored as an attachment in the system. This action conflicts with the current implementation of accepting attachments through SOAP. You can easily raise that setting to a higher number. However, be aware that in some rare situations this may affect the performance for your instance.
