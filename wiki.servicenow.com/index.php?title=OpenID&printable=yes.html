1 Overview

The OpenID integration enables single sign-on by exchanging URL parameters with an external OpenID Provider (OP). The OP authenticates the user and passes an OpenID parameter to ServiceNow containing some portion of the user's credentials (typically, the user name or email address). After verifying the identity of the OP, ServiceNow searches for a user with matching credentials, and if it finds such a user logs in the user.

The ServiceNow Open ID integration use a stateless process that requires posting back to the IdP for signature verification.



  1. User requests a resource from ServiceNow without first authenticating with the OpenID Provider (The request does not contain any OpenID URL parameters).
  2. ServiceNow redirects the request to the OP with URL parameters requesting authorization, including the post back URL to ServiceNow after the OP validates authentication.
  3. The OP challenges the user to provide credentials.
  4. The user enters his/her OpenID credentials and authenticates successfully.
  5. The OP posts the request, including the required OpenID URL parameters back to ServiceNow.
  6. ServiceNow validates the request.
    a. Verifies the "return_to" parameter matches the instance's URL.
    b. Inspects the "response_nonce" parameter and ensures that it is within the time frame allowed.
    c. Posts back to the signature verification service at the OpenID provider, ensuring that the signed parameters are valid.
  7. ServiceNow logs in the user specified by the "user_key" parameter, and responds to the user with the session cookies.

2 OpenID Integration Requirements

The OpenID single sign-on integration requires:

  • Activating the OpenID plugin
  • Access to an OpenID Provider
    • Must follow the OpenID specification
    • Must provide a list of valid OpenID URL parameters
    • Must handle requests from stateless Service Providers

3 OpenID Setup

Perform these steps to enable single sign-on with OpenID.

  1. Request the OpenID plugin.
  2. Enter OpenID system properties.
  3. Enter Single Sign-On system properties.
  4. Add appropriate values to the User table field.
  5. Test the integration.

3.1 Step 1. Activate the OpenID Plugin

Click the plus to expand instructions for activating a plugin.

If you have the admin role, use the following steps to activate the plugin.

  1. Navigate to System Definition > Plugins.
  2. Right-click the plugin name on the list and select Activate/Upgrade.
    If the plugin depends on other plugins, these plugins are listed along with their activation status.
  3. [Optional] If available, select the Load demo data check box.
    Some plugins include demo data—sample records that are designed to illustrate plugin features for common use cases. Loading demo data is a good policy when you first activate the plugin on a development or test instance. You can load demo data after the plugin is activated by repeating this process and selecting the check box.
  4. Click Activate.

The plugin installs a new application called OpenID Single Sign-on.

3.2 Step 2. Enter OpenID System Properties

Navigate to OpenID Single Sign-on > Properties to enable single sign-on with an OpenID provider. Set these system properties.

Field Description
Enable external authentication. Set to Yes to allow an OpenID provider to pre-authenticate users.
The URL of the OpenID provider Enter the URL to the OpenID provider server. For example, https://www.google.com/accounts/o8/id
The Open ID prefix used for session validation. This value is used to prefix the incoming Open ID URL parameters for example: "openid.ns", "openid.return_to", "openid.response_nonce", "openid.signed" and others Enter the prefix the OpenID provider uses to identify OpenID URL parameters For example, openid.
The name of the parameter that contains the user identifying information. Do not include the global prefix (ie. use 'ext1.value.email' not 'openid.ext1.value.email') Enter the URL parameter that contains user credential information. This parameter uniquely identifies a user in your ServiceNow instance. Do not include the global prefix. For example, ext.value.email.
Service-now.com field name from the User's table used to match against the incoming header field value Enter the name of the ServiceNow User table field that contains matching values to the user credentials contained in the URL parameter. Typically, this is either user_name or email fields.
Use HTTP POST to verify the signed URL parameters with the Open ID provider. Determines whether to use a GET or POST (more secure) when verifying the OpenID tokens. POST is used if property is checked. Set to Yes to use the more secure POST method to verify URL parameters. If the OpenID server only supports verification via the GET method, set this property to No.
Enable verbose logging. Set to Yes to enable expanded logging (larger, more informative log records).

3.3 Step 3. Enter Single Sign-On System Properties

Navigate to System Properties > Single Sign-On to set redirection behaviors when using OpenID for single sign-on.

Field Description
Failed Requirement Redirection Enter the URL to the OpenID provider with authentication request and redirection URL parameters. The OP should provide you with the URL parameters required for this URL.
Redirect users on a failed authentication Enter the URL to redirect the user if the OpenID authentication request fails. An authentication request might fail for any of these reasons:
  • A user successfully logs into the OP, but does not have a user record in ServiceNow.
  • The security check fails because the return_to value does not match the instance name.
  • The nonce time does not match the time allowed by the ServiceNow instance (within 60 seconds).
  • There is any other error.
External Logout Redirect Location Enter the URL to redirect the user when the user logs out of OP. The instance logs the user out of the ServiceNow session before redirecting the user to this URL.
Disable Local Login Set to yes to prevent users from seeing the ServiceNow login page. Instead, the instance redirects users to the URL specified in the Failed Requirement Redirect property.

3.4 Step 4. Add Appropriate Values to the User Table Field

Ensure that the integration's User table field contains appropriate matching values. For example, if the OpenID Provider uses an email address as the user credentials in the URL parameter, ensure that ServiceNow lists the same email address as the OP. The integration fails to authenticate any user who does not have a User Table value that matches a URL parameter value.

3.5 Step 5. Test the Integration

Here is a list of common login errors and their solutions.

Error or Symptom Solution
The user successfully logs into the OpenID provider, but cannot log into ServiceNow. Verify that the user has a valid user record in ServiceNow and that the user's OpenID credentials match a field value in the User table.
The OpenID provider never returns a response. Verify the URLs and URL attributes are properly set in the OpenID Single Sign-On properties.
