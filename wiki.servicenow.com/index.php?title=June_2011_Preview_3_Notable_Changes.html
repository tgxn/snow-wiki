June 2011 Preview 3 Notable Changes

From ServiceNow Wiki

The following are notable changes you may encounter with our June 2011 Preview 3 release. These changes are not necessarily problems or major features, but rather new-elements or behaviors you may notice being different than before. For additional changes, see June 2011 Preview 3 Fixes.

1 Changes for June 2011 Preview 3

1.1 Preventing orphan workflow stages

Workflow-specific Workflow Stages cannot be added from the Stages reference icon list view. To add new stage, navigate to the Graphical Workflow Editor and select the Workflow Actions menu and select Edit Stages.

1.2 Chat

Users can now specify a Room avatar (picture) for chat rooms. The room avatar appears beside the room name in the buddy list and in the list of all rooms.

1.3 Scripted Web Services

The Web Services Provider - Scripted plugin is now installed automatically on all instances.

1.4 ACLs on system fields for new instances

New instances are created with ACLs preventing write access to the following fields on all tables: sys_id, sys_mod_count, sys_created_by, sys_created_on, sys_updated_by, sys_updated_on.

1.5 Currency field list decoration change

For currency fields in a list, if the user's session currency is the same as that of the value in the list cell, the toggle globe decoration will not be displayed for that cell.

1.6 Workflow field type filter operators

Workflow fields now get the same filter operators as choice fields.

1.7 Field style for metric_definition.name

There is a new Field Style for metric_definition.name, increasing the width.

1.8 can_group field attribute

With a field attribute of can_group=true or can_group=false, it is now possible to override the system's default determination of whether a field can be grouped on.

1.9 "Reject" UI Action action name

The "Reject" UI Action form button for the sysapproval_approver table has been given an action name of "reject" for consistency with the "Approve" UI Action form button.

1.10 Android mobile UI choice

Android devices can now easily switch between desktop version and mobile version like iOS devices can.

1.11 closed and closed_at field populated for all tasks

Added a high order Business Rule to the task table, "Set Closure Fields", that sets closed and closed_at fields when active changes to false if not already populated.

1.12 Austere and Visionary CMS theme plugins no longer available

The legacy Austere and Visionary CMS theme plugins have been removed from the codebase and are no longer available for installation. Customers who have already installed these plugins will not notice any change.

1.13 Calculated Planned Task field decoration

Planned task fields that are automatically calculated from values in child records are identified with a field decoration:

1.14 Wizard and Catalog display width increased

Wizard panel and Catalog item display width was previously hardcoded at 550 pixels. This restriction has been removed, so the display should expand to the width of the user's screen.

1.15 Field styles apply to multi-line fields in forms

Field styles are now applied to multi-line fields in forms, consistent with single line fields.

1.16 Workflow properties module

A Properties module has been added to the Workflow application for easy access to relevant properties.

1.17 Line Charts supported in Custom Charting

Line charts can now be used by custom chart generation scripts.

1.18 Certification Task group assignment

Certification Tasks can now be automatically assigned to groups in addition to users. Requires the Certification plugin.

1.19 service_knowledge and service_news tables no longer shipped

The service_knowledge and service_news tables have been cleaned from the code base. These predated and were made obsolete by the kb_knowledge table. Existing customers who already have these tables will not notice a change.

1.20 Real time planned start calculation

Update to planned task and project date logic. Primary change is to always calculate Planned end and successor Planned start dates from any actuals (work_start, work_end) that have values. This makes for a real time project instead of calculating everything on original planned values even when actuals exist and are different. Also display actuals instead of planned in project timelines.

1.21 Module role requirement changes

  • The System UI -> Videos module now requires the image_admin role
  • The System UI -> UI Context Menus module now requires the admin role
  • The Reports -> Administration module now requires the report_admin role, not admin
  • The Reports -> Report Ranges module now requires the report_admin role
  • The Configuration -> Relationships module now requires the ecmdb_admin role
  • The Configuration -> Baseline module now requires the ecmdb_admin role

1.22 "Transform" UI Action role restriction

The "Transform" UI Action form link is restricted to import_admin, import_transformer roles.

1.23 Dictionary related lists

Two Relationship related lists have been added to the Dictionary form, Choices and Labels. They show any choice list options for a field and any labels for a field or table. Lists are omitted if empty. New button is omitted. Customers who have customized their Dictionary form in any way may not see this change.

1.24 Client Transaction Timings plugin active for new instances

New instances have the Client Transaction Timings plugin activated by default.

1.25 Embedded List Control change

A new table has been created to store list control information for embedded lists, sys_ui_list_control_embedded. An update script is automatically run to handle this change.

1.26 System Changes

1.26.1 Dirty form support enabled for new instances

Dirty form support is enabled by default for new instances. Existing instances will not see a change in behavior. Also, Chrome now supports the dirty form property.

1.26.2 Tabbed forms enabled for new instances

The use of tabbed forms is enabled by default for new instances. Existing instances will not see a change in behavior.

1.26.3 Update on iterate disabled for new instances

Update on iterate (which determines whether a record is automatically updated when navigating to the next/previous record using the blue up/down arrows on a form) is disabled by default for new instances. Existing instances will not see a change in behavior.

1.26.4 Enter Submits Form disabled for new instances

The behavior of the Enter key automatically submitting the form is disabled by default for new instances. This is a System user preference that can be changed on a system or user basis as desired. Existing instances will not see a change in behavior.

1.26.5 Modules activated by default

The following modules have been activated. Customers who have never modified these modules will now see them in the navigation frame:

  • User Administration > User Roles - (admin) sys_user_has_role list grouped by role
  • User Administration > Group Roles - (admin) sys_user_group_has_role list grouped by role
  • System Definition > Deleted Records - (admin) sys_audit_delete list showing deletion details for records deleted from audited tables
  • Reports > Administration - (report_admin) sys_report list showing underlying records defining reports
  • Reports > Report Ranges - (report_admin) sys_report_range list showing data value ranges that appear as one grouping in reports

1.26.6 "Run as" field added to Scheduled Import form

The "Run as" field (reference to sys_user) has been added to the Scheduled Import form. Customers who have never modified this form will now see it.

1.26.7 "Referenced value field name" field added to Field Map form

The "Referenced value field name" field has been added to the Field Map form for Transform Maps. Customers who have never modified this form will now see it.

1.26.8 "Show insert" and "Show update" fields added to UI Action form

The "Show insert" and "Show update" fields have been added to the UI Action form. Customers who have never modified this form will now see them.

1.26.9 Applications/Modules deactivated for new instances

The following applications/modules are not active for new instances. They can be easily activated if desired. Existing instances will not see a change in behavior.

  • Release v1 application
  • Service Catalog > Maintain Bundles module
  • Service Catalog > Wizards module

1.26.10 SSO Redirection Properties added to property page

The four existing Single Sign-on redirection properties are added to the Single Sign-on properties page, and they are created in an instance if they do not exist.

1.26.11 Security Jump Start plugin removes write access to system fields

Six new ACLs have been added to the Security Jump Start plugin preventing write access to system fields. Customers who have installed the Security Jump Start plugin (also installed automatically for instances created since March 2009) will get these ACLs:

  • *.sys_id - nobody can write
  • *.sys_created_on - nobody can write
  • *.sys_created_by - nobody can write
  • *.sys_updated_on - nobody can write
  • *.sys_updated_by - nobody can write
  • *.sys_mod_count - nobody can write

1.26.12 Transform Script to deactivate LDAP users for new instances

An inactive onBefore Transform Script has been added to the LDAP User Import transform map, which can be used to deactivate users imported from LDAP when appropriate. Existing instances will not see a change in behavior.

1.26.13 Warning comment in LDAP User Import transform map script against mapping Manager field

A comment has been added to the LDAP User Import transform map script warning against mapping the Manager field. The 'ldapUtils' scripts take care of this already. Customers who have never modified this transform map will now see the comment.

1.26.14 Group import transform script removed for new instances

An inactive onAfter transform script that imported groups from LDAP based on group membership has been removed from the LDAP User Import transform map. Almost without exception, it resulted in numerous undesired groups being created. Existing instances will not see a change in behavior.

1.26.15 Script Logs module added

System Logs > Script Logs module has been added, showing the list of log entries created by gs.log(message) statements in scripts.

1.26.16 schedule_admin role (added to all instances) protects Schedule tables with Security Jump Start Plugin

A schedule_admin role has been created. Instances with the Security Jump Start plugin activated get new ACLs restricting create/write/delete access to the cmn_schedule and cmn_schedule_span tables to the schedule_admin role.

1.26.17 Update Set "Ignore" state

A new state "Ignore" has been added to Update Sets. If an Update Set has a state of Ignore, it will be excluded when retrieving completed update sets.

1.26.18 Update Set requires Admin to Read

A new ACL requires users to be administrators to read update sets.

1.26.19 "Updated" field added to many administrative lists

The Updated date/time field has been added to over a hundred administrative lists where it might be useful. Customers who have not customized a particular list will see the Updated field as the last column.

1.27 Task Management Changes

1.27.1 Simultaneous update alert for new instances

A new client script "Simultaneous Update Alert" has been added to the task table for new instances. It alerts a user, when submitting a form, if the record has been updated since it was opened in their browser. The user is given the opportunity to cancel or proceed with the form submission. Existing instances will not see a change in behavior.

1.27.2 Write access to five task fields restricted to admin for new instances

When a new instance is created, ACLs are automatically created restricting write access to admin for the following task fields: Number, Opened, Opened by, Closed, Closed by. Existing instances will not see a change in behavior.

1.27.3 User reference autocomplete uses contains query for new instances

A new user preference named "sys_user.autocomplete.contains" has been added to new instances setting the autocomplete query to a "contains" query (vs. a "starts with" query) for reference fields against the User table. e.g., typing "Good" will return "Don Goodliffe" in the autocomplete matches. Existing instances will not see a change in behavior.

1.27.4 Only admins can delete tasks for new instances

A row-level Delete ACL has been added to the task table requiring the admin role to delete task records. Existing instances will not see a change in behavior.

1.27.5 Reassignment count field and business rule moved from Incident to Task

The reassignment_count field is promoted to the task table from the incident table. All instances will see this change. The "reassignment count" Business Rule has had the Table value changed from task to incident. Customers who have not customized this rule will see this change.

1.27.6 Field Styles for Additional comments and Work notes journal fields for new instances

To distinguish them from each other, Field Styles have been added to the Additional comments and Work notes journal fields for new instances, to color the inputs the same as they are colored by default in the activity formatter. Existing instances will not see a change in behavior.

1.27.7 Additional comments field label change for ITIL users for new instances

Users with the itil role will see "(Customer visible)" appended to the Additional comments field label. This is done with the "Modify Comments Label" client script for new instances. Existing instances will not see a change in behavior.

1.27.8 New button removed from task form related lists for new instances

Where it didn't make sense to have them, the New and Edit buttons have been removed from related lists for various task forms for new instances. This is done using List Control and can be customized as desired. Existing instances will not see a change in behavior.


1.28 Incident Management Changes

1.28.1 Incident Overview homepage reworked for new instances

The Incident Overview homepage has been modified for consistency and relevancy for new instances. Existing instances will not see a change in behavior.

1.28.2 Email client activated by default

The email_client=true attribute has been added to the Incident table's dictionary record, activating the email client by default for all instances. It can be deactivated by setting the attribute to false.

1.28.3 Incident state changes to Active when assigned for new instances

An Incident in the New state will be changed to the Active state when assigned. This happens with the "Incident State Active on Assignment" business rule for new instances. Existing instances will not see a change in behavior.

1.28.4 Reopen count integer field added for new instances

A new integer field, Reopen count, has been added to the Incident table for new instances. The "Reopen Count" business rule increments this value when the Incident's state is changed from Closed or Resolved to another state. Existing instances will not see a change in behavior.

1.28.5 Parent Incident reference field added for new instances

A new reference field, Parent Incident, has been added to the Incident table. It can be used to group Incidents. Additional comments and Work notes to a parent incident are copied to its unresolved child incidents. Resolving a parent incident automatically resolves its unresolved children. Close notes and Close code is copied from a parent incident to its children if these fields are empty on the child. Existing instances will not see a change in behavior.

1.28.6 Child Incidents integer field added for new instances

A new integer field, Child Incidents, has been added to the Incident table for new instances. When the Parent Incident field value changes for an incident, a business rule automatically updates the Child Incidents value. Existing instances will not see a change in behavior.

1.28.7 VIP indicator for Caller field in lists for the Best Practice - Incident Resolution Workflow plugin

The field style on incident.caller_id has been modified in the Best Practice - Incident Resolution Workflow plugin to add a small VIP indicator to the Caller field in a list if the referenced user is a VIP. This indicator is consistent with the one already added to the Caller field label on the form by the Highlight VIP Caller client script. Customers with the Best Practice - Incident Resolution Workflow plugin activated who have not customized this field style will see this change.

1.28.8 Knowledge article Source field populated

The Incident Create Knowledge and Problem Create Knowledge business rules have been modified to populate the Source reference field of a new article when it is automatically created from an Incident or Problem. Customers who have not customized these rules will see this change.

1.28.9 Self Service users can resolve/close their own Incidents for the Best Practice - Incident Resolution Workflow plugin

The Close Incident and Resolve Incident UI Actions have been modified in the Best Practice - Incident Resolution Workflow plugin to allow Self Service users to resolve and close Incidents where they are the Caller. Customers with the Best Practice - Incident Resolution Workflow plugin activated who have not customized these UI Actions will see this change.

1.28.10 Incident form reworked for new instances

The Incident form has been modified for new instances. It now has multiple sections (main, Notes, Related Records, and Closure Information), and fields have been added/removed for relevance and consistency. Existing instances will not see a change in behavior.


1.29 Change Management Changes

1.29.1 Change Overview homepage reworked for new instances

The Change Overview homepage has been modified for consistency and relevancy for new instances. Existing instances will not see a change in behavior.

1.29.2 Change Schedule module and timeline page

The Planning Timeline module has had its title changed to Change Schedule for consistency with ITIL v3 nomenclature. The Planned Changes timeline page has similarly been retitled to Change Schedule. Customers who have not customized this module or timeline page will see this change.


1.30 Problem Management Changes

1.30.1 Problem Overview homepage reworked for new instances

The Problem Overview homepage has been modified for consistency and relevancy for new instances. Existing instances will not see a change in behavior.

1.30.2 New Problem Task table for new instances

A new table, Problem Task, has been added for new instances similar to the Change Task table. The "problem_task_events" business rule fires events based on Problem Task record activity. The Problem Tasks related list has been added to the Problem form. Existing instances will not see a change in behavior.

1.30.3 Known error boolean field added for new instances

A new boolean field, Known error, has been added for new instances. This allows a Problem to be designated as a known error independent of the Problem's state. The filter on the Known Error module has been changed to reflect this change. Existing instances will not see a change in behavior.

1.30.4 Related Incidents integer field added for new instances

A new integer field, Related Incidents, has been added to the Problem table for new instances. When the Problem field value changes for an incident, a business rule automatically updates the Related Incidents value. Existing instances will not see a change in behavior.

1.30.5 Related Incident changes based on Problem activity for new instances

Related Incidents are no longer automatically resolved for new instances when a Problem closes. Related incident Work notes are updated with Problem close notes, and their state is set to Active if it is currently Awaiting Problem. Existing instances will not see a change in behavior.

1.30.6 Workaround field type changed from journal to journal_input for new instances

The Workaround field's internal type has been changed to journal_input for new instances. Existing instances will not see a change in behavior.

1.30.7 Activity formatter added to Problem form

The Activity formatter has been added to the Problem form. Customers who have not customized the Problem form will see this change.


1.31 Service Catalog Changes

1.31.1 Service Catalog list and form changes

Request, Requested Item and Catalog Task lists and forms have been reworked for consistency and relevance. Customers will see changes to these lists and forms they have not customized.

1.31.2 New assignment and work notes email notifications for Catalog Task table for new instances

New email notifications have been added for the Catalog Task table for new instances when the Catalog Task gets assigned or worknoted. Existing instances will not see a change in behavior.

1.31.3 Request Item email notifications for new instances

When a Request Item is commented, there are new email notifications for new instances to notify the Requester and the Assigned to individuals for all Catalog Tasks for that Request Item. Existing instances will not see a change in behavior.


1.32 Approvals Changes

1.32.1 Approve/Reject button coloring for new instances

The Approve and Reject buttons are now colored green and red for new instances. Existing instances will not see a change in behavior.

1.32.2 Activity formatter added to Approval form

An Activity formatter has been added for the approval table (shows history plus sent/received emails). Customers who have not customized the Approval form will see this change.

1.32.3 Comments field type changed from journal to journal_input for new instances

The Comments field's internal type has been changed to journal_input for new instances. Existing instances will not see a change in behavior.

1.32.4 Comments mandatory on rejection for new instances

Comments are now mandatory on approval rejection for new instances. Existing instances will not see a change in behavior.


1.33 Knowledge Changes

1.33.1 Source field added to Knowledge form

The Source reference field has been added to the Knowledge form. Customers who have not customized this form will see this change.

1.33.2 New Knowledge modules added for new instances

The following new modules have been added to the Knowledge application for new instances. Existing instances will not see a change in behavior.

  • Draft/Review
  • Published
  • Retired


1.34 Self Service Changes

1.34.1 New modules added to Self Service for new instances

The following modules have been added to Self Service for new instances. Existing instances will not see a change in behavior.

  • Watched Incidents
  • Watched Requested Items

1.34.2 Self Service homepage reworked for new instances

The Self Service homepage has been modified for consistency and relevancy for new instances. Existing instances will not see a change in behavior.

1.34.3 New button removed from Incident list for Self Service for new instances

The New button has been removed from the Incident list for Self Service users for new instances. Those users should go through the Catalog to create a new Incident. Existing instances will not see a change in behavior.
