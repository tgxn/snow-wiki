1 Overview

The Service Management (SM) application designer allows you to create custom applications that use a similar service management workflow.

2 SM Application Types

Use the SM application designer to create basic, intermediate, or advanced applications, each with a different set of capabilities and default configurations.

Type Description Default Configuration
Basic Basic applications use a request driven process. This process allows parent records, such as work orders, facilities requests, and HR cases, that contain a single task to streamline the request process by displaying information from the parent and task records on the same form.
  • Requests are assigned via auto-assignment.
  • Requests are created using a dedicated catalog.
  • Templates automatically create catalog items.
  • A dedicated knowledge base is available
Intermediate Intermediate applications use a task driven process that is useful for requests that contain multiple tasks.
  • Approvals are required for new requests, and the Approval Workflow for SM is used.
  • Tasks are assigned via auto-assignment.
  • Requests are created using a dedicated catalog.
  • Templates automatically create catalog items.
  • A dedicated knowledge base is available.
  • Managed documents are used.
Advanced Advanced applications use a task driven process that is useful for requests that contain multiple tasks.
  • Approvals are required for new requests, and the Approval Workflow for SM is used..
  • Tasks are assigned via auto-assignment.
  • After tasks are assigned, they must be accepted or rejected by agents.
  • Part requirements are needed by agents.
  • Requests are created using a dedicated catalog.
  • Templates automatically create catalog items.
  • A dedicated knowledge base is available.
  • Managed documents are used.
  • Time tracking is used.
  • The new application uses dispatch queue.

You can modify the configuration prior to the application generation process or after. You can even convert a basic application to an intermediate or advanced application, or create a hybrid application.

3 Creating SM Applications

You can create SM applications by:

  • Generating an application using the default settings. This method involves simply entering an application name and scope, and clicking a single button.
  • Configuring an application prior to generating it. This allows you to tailor the application to your needs.

3.1 Generating an SM Application With the Default Settings

  1. Navigate to System Applications > Applications.
  2. In the Develop tab, click New. The Getting Started screen opens.
  3. Next to Start from a template, click Create.
  4. Enter a unique Application name for this application. The application name can be a maximum of 20 characters. Only letters, digits, spaces, and underscores (_) are valid.
    If you enter the name of an existing application, you will be asked to select a different name.
  5. The Application scope field defaults to an x prefix that identifies this as a custom application, an optional instance vendor prefix that identifies your company, followed by the application name. You can accept the default or change it, if needed.
    If you enter the application scope of an existing application, you will be asked to select a different name after you click Create.
  6. Click Create in the section that identifies the type of application you want to create.
    A confirmation box opens and displays the default configuration settings that the new application will inherit. The screen below is for a Basic application.
  7. Click OK.
    A confirmation box appears.
  8. Click View App.
    All of the files you need to run the application, including tables, roles, properties, ACLs, and navigation menus are created, and the new application appears.
Note: Click the annotation icon () in the header to view on-screen descriptions of each of the fields.


3.2 Configuring an SM Application Prior to Generation

  1. Navigate to System Applications > Create Application.
  2. Next to Start from a template, click Create.
  3. Enter a unique Application name for this application. The application name can be a maximum of 20 characters. Only letters, digits, spaces, and underscores (_) are valid.
    If you enter the name of an existing application, you will be asked to select a different name.
  4. The Application scope field defaults to the application name preceded by an x prefix, which identifies this as a custom application, and a prefix that identifies your company. You can accept the default or change it, if needed.
  5. Click Configure in the section that identifies the type of applications you want to create.
    The default configuration settings for the specified application type appear (see table for field descriptions). The following screen shows the settings for a Basic application.
  6. Change any of the default settings, as needed (see table).
  7. When you are done, click Create.
    A confirmation box opens and displays your settings.
  8. Click OK.
    A confirmation box appears.
  9. Click View App.
    All of the files you need to run the application, including tables, roles, properties, ACLs, and navigation menus are created, and the new application appears.
Application settings
Field Description Configuration Defaults by Type
Application scope The application scope determines where an application runs and what resources from other applications it can access. This field displays the application name you entered on the Create Application from a Template screen with an sn prefix. All types
Request number prefix The prefix applied to all request numbers created using your application. This field defaults to a prefix based on the application name, followed by an R for request. For example, if your application name is Legal Matters, the request number prefix will be LMR. If you have another application that already uses this prefix, you must change it. This prefix will be applied to all request numbers created using your application. All types
Task number prefix The prefix applied to all task numbers created using your application. This field defaults to a prefix based on the application name, followed by a T for task. For example, if your application name is Legal Matters, the request number prefix will be LMT. If you have another application that already uses this prefix for tasks, you must change it. This prefix will be applied to all task numbers created using your application. All types
User role prefix The prefix applied to all user roles defined for the application. This field defaults to the application name. If you have another application that already uses this prefix for user roles, you must change it. All types
Create CMS pages Creates an icon for your application to be added to the Shared Service CMS Portal. This switch appears only if you have first activated the CMS User Interface - Service Management Core plugin. If you do not activate the plugin, the switch will not be displayed and the icon for your application will not appear on the CMS portal. All types
Business Process
Request lifecycle is Select request driven if you want tasks to be assigned to users in an assignment group. Users can still add tasks to requests; however, closing all tasks will not automatically close the request. Select task driven if you do not want to require tasks to fulfill requests. All types set to request driven
Tasks are assigned Select the assignment method:
  • via auto-assignment: For more information, see Using Auto-Assignment.
  • via a workflow: You will be prompted to select the workflow you want to use for dispatching.
  • manually: For more information, see Dispatching Tasks from the Queue.
 All types set to via auto-assignment
Use this workflow to dispatch tasks Click the link and select the workflow you want to use for dispatching orders. This field is displayed if you select via a workflow as the method for dispatching tasks.

If you do not select a workflow, the default workflow,Dispatch Workflow for SM will be used. This example workflow evaluates each agent in the assignment group for the task and assigns the task to the agent that has the most matching skills.

Approval is required or not required for new requests Select required to require requests to be approved before moving to the next state. If this option is set to not required, any requests in the Awaiting Approval state will be automatically approved. All types set to not required
Use this workflow for approvals Click the select a workflow link and select the approval workflow you want to use for requests. This field is displayed if you select Approval is required for new requests.

If you do not select a workflow, the default workflow, Approval Workflow for SM, will be used. This workflow requests an approval from all users with an approver role.

An approver must provide one of the following approval responses before requests can move to the next state:

  • Approved
  • Rejected
  • More information required
  • Duplicate
After tasks are assigned, they must be or cannot be accepted or rejected by agents Select cannot be accepted or rejected to have agents automatically accept tasks. When tasks must be accepted or rejected, agents must click Accept on tasks to begin work or click Reject to return the task to the Pending Dispatch or Pending Assignment state.

Note: If tasks are dispatched via a workflow or via auto-assignment, rejected tasks will be automatically re-assigned.

Advanced type set to cannot
Agent travel time will be or will not be tracked Select will be tracked to enable time cards on tasks. Advanced type set to will not
Additional options
Part requirements are needed or are not needed by agents Select are not needed to hide modules, related lists, and other UI components related to part requirements. Select are needed to enable parts and part sourcing. Advanced type set to are not needed
Requests are created using the New Application Select regular form only to deactivate the catalog and disable automatic publishing of templates to the catalog. Select catalog or regular form to activate the work management catalog and enable automatic publishing of templates to the catalog. All types set to catalog or regular form
Templates will or will not create a dedicated catalog item Select will to activate automatic publishing of catalog items for the new application. All types set to will
A dedicated knowledge base is available or is not available for searching Select is available to activate the knowledge base for the new application. All types set to is available
Managed Documents Select will be used to add a related list to managed documents. This feature requires Managed Documents. All types set to will not be used do not require users to manage documents.
New Application will or will not use the dispatch queue This field is displayed if you select the Request lifecycle is task driven option. Select will to require the new application to use the dispatch queue. A user with a qualifier role must provide a dispatch group for tasks before they can be ready for assigned or dispatched.

If will not is selected, all dispatch-related modules will be removed from the navigation pane, and all dispatch-related fields will be removed from the new application's screens.

Task work notes will or will not update the request work notes Select will to synchronize task work notes with the work notes on the request. So when work notes in the task are added, the same work notes appear in the request. All types set to will not
Assignment options
Selection of groups will or will not consider group's covered location Select will to limit the selection of groups from the Dispatch group, Qualification group, and Assignment group fields to groups that cover the location of the task. Intermediate and Advanced types set to will
Auto-selection of agents will or will not consider agent or task schedules This field is displayed if you select the Request lifecycle is task driven option. Select will not if you do not want auto-assignment to use the schedules of the agents or tasks to determine who can be assigned to each task. Additionally, the Estimated work duration field is not mandatory when this configuration option is set to will not.
Auto-selection of agents will or will not consider location of agents Select will to use the agent and location when determining who to assign the task to. Agents closer to the task location get preference. Intermediate and Advanced types set to will
Auto-selection of agents requires them to have all or some of the required skills for the task This option determines the degree to which skills must be matched to a task when determining auto-assignment.
  • Select all to require that an assigned agent must have all of the skills to perform the task. An agent who lacks any of the skills is eliminated.
  • Select some if you want agents who have most of the skills needed to perform the task.
  • Select none if you want to auto-assign agents without taking skills into account.
Intermediate and Advanced types set to all
Note: For any option that is not already set to its default, you can click [Reset to default] to return the option to its default setting.


4 Customizing an SM Application

After you have generated an SM application, you can continue to customize the application by navigating to (Application name) > Administration > Configuration. The options displayed are the same ones that are displayed when you are creating an Advanced application. This gives you the ability to turn a Basic or Intermediate application into an Advanced application by turning on additional options.

To customize a generated application:

  1. Navigate to the menus and modules for your new SM application.
  2. Click Configuration.
  3. Make changes to the configuration, as needed. For field descriptions, see Configuring an SM Application Prior to Generation.
  4. When you are done, click Save.
