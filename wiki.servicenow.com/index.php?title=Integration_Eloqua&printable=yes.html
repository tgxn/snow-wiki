1 Overview

1.1 Purpose:

Eloqua is a third-party system that allows its clients to design forms for web pages and email. When customers interact with those forms, Eloqua tracks that information and shares it with ServiceNow. This integration transfers form data and form activity from Eloqua's database to a ServiceNow instance.

1.2 The Application

The Eloqua Integration provides an extensive application module in the index frame of your ServiceNow instance. Here are some of the breakdowns of the modules available through the application.

  • Overview and Instructions - displays this wiki page
  • Settings / Properties - this is where you configure the integration to fit your own environment
  • Business Rules - For every Eloqua form, you will create a business rule to handle the information coming in from that form. If forms are not getting processed, this is probably where you would go to debug why a particular form is not processing correctly.
  • Scheduled Jobs - Since Eloqua is unable to send SOAP requests to a third-party, we have to periodically bring in form and activity information from Eloqua. These scheduled jobs will handle this. Also, there are scheduled jobs for both cleanup of old data that has been processed as well as jobs that monitor the system for possible problems or outages.
  • Script Includes - This module holds the scripting libraries for the integration
  • SOAP: Eloqua SOAP Messages - this is the parent SOAP request object for the subsequent SOAP messages listed below. It contains the WSDL format and connection details. If Eloqua changes their WSDL / SOAP API format, this is likely where you will go to fix that.
  • SOAP: Eloqua DescribeEntityType - this is one of the messages listed in the Eloqua SOAP Messages object. This is used to query what forms are listed in Eloqua at this time.
  • SOAP: Eloqa Query - this is the other message we use from the Eloqua SOAP Messages object. This queries the submissions for a given form.
  • Error Handling: Email Notifications - These are the various email notifications that are set up to be sent out for different alerts
  • Error Handling: Email Templates - Not currently utilized, but you can change them if you want more customized email notifications
  • Error Handling: Event Registry - The list of events that are thrown by the Eloqua Integration
  • Eloqua Tables: Eloqua Form - A table of all of the forms you have in your Eloqua system. You can mark these as active or inactive. They are brought into the table automatically.
  • Eloqua Tables: Eloqua Form Submission - A table of the different submissions for the various forms. The integration brings these in automatically on a scheduled job and they are processed by the various business rules.
  • Eloqua Tables: Transform Maps - These are the transform maps used to bring in data from Eloqua and set it into the various tables used by the integration.


1.3 How it Works:

1.3.1 Scheduled Jobs

This integration features a number of scheduled jobs that run constantly.

EloquaCheckSubmissionImports - This job will check the Eloqua Form table to see if there are any forms that have not been successfully queried within the grace period that is set in the "Settings / Properties" module of the integration under "Query History". If any forms are exceeding that grade period, and event is triggered which sends out an email alert prompting action

EloquaGetForms - This job will query Eloqua for all of the forms in the system. If there are new forms it will add them for the "Eloqua Form" table and list that form as "Active"

EloquaProcessFormSubmission - This job takes the "Eloqua Form" entry that has the oldest "Processed" date for an Active form. It will query Eloqua and take all new form submissions and enter it into the "Eloqua Form Submission" table.

'EloquaCheckForUnprocessedSubmissions' - Periodically we will check the Eloqua Form Submission table for submissions that are not being processed. This is a warning to us that maybe a business rule is failing or does not exist for that particular Form or submission data. An email will be sent if there are any unprocessed items.

'EloquaCleanOldProcessedSubmissions' - When form submissions are processed, they are left in the Eloqua Form Submission table with a processed flag of true. This job will periodically clean up the old processed submissions from the table.

'EloquaCleanImportSetTable' - If your system has a lot of activity, the the imports could be bringing in a lot of data, making your import set table pretty large. This job will clean up the import set table at a much quicker rate than the ordinary methods in the system.

1.3.2 Tables

There are also a couple of important tables in the integration:

Eloqua Form Submission - table acts as a type of queue of form submissions that need to be processed. Every active form in Eloqua should have a Business Rule set up in the integration. This business rule should monitor the inserts on the "Eloqua Form Submission" table. When a new submission is listed in the table, the appropriate business rule should process it according to its own logic.

Eloqua Form - This table receives a list of forms that are currently handled or have been handled in Eloqua. This list will be automatically populated by the EloquaGetForms scheduled job mentioned above. (One form record is not automatically populated - see the "Things to Note" section of this document with regard to the ProcessedActivity form.) If a form is no longer in use by the system, it is important to mark the active flag for that record as "false". This will speed up the time between data updates for the other tables.

1.4 Integrating a new Eloqua Form:

When creating a new form in Eloqua, you must create a corresponding Business Rule for that form. Name the business rule using the following naming convention: Eloqua[FormName]. Therefore, if you created a form named, "June Webinar Feedback", you would create a business rule called "EloquaJuneWebinarFeedback".

Your business rule should be monitoring the "Eloqua Form Submission" table for any new inserts where the form name is the name of your form. Also make sure to set your condition to only check for u_processed rows that are "false".


1.5 Things to note

  • There is one form that must be entered into the "Eloqua Form" table after a new Eloqua Integration implementation. While all forms are typically automatically generated through a web service call on a scheduled basis, there is one that you need to create manually. To do this, go to the "Eloqua Form" table and create a new record with two pieces of data: 1) Form ID: 0; 2) Form Name: ProcessedActivity. This table stores all of the activity detail from Eloqua.
