1 Overview

A cluster is a type of complex license that acts as a parent record to related infrastructure licenses and enables an administrator to evaluate compliance for license counts in the cluster instead of the licenses. Related licenses in a cluster record are different instances of a single software package in the same locality. Examples of this might be groups of licenses for a Development department that were purchased at different times with different license keys or expiration dates.

Note: A significantly improved and updated application named Software Asset Management was included in the Berlin release. For information about upgrading from Software License Management to Software Asset Management, see Upgrading to Software Asset Management.


2 How Clusters Work

Cluster licenses support infrastructure licenses only. When an infrastructure license is related to a cluster license, the following form elements are not available:

  • Related Lists for displaying licensed and unlicensed CIs.
  • License checker
  • Used field for displaying the actual license count.

License compliance is calculated and displayed in the cluster license only and is based strictly on license count. The cluster license displays licensed and unlicensed configuration items, and calculates compliance based the number of licensed CIs compared with the number of entitled CIs.

3 Configuring Cluster Licenses

  1. Navigate to Software License Mangement > Complex Software Licenses.
  2. Click New.
  3. Complete the complex software license with all the pertinent data (locality, entitled value, software package, etc.)
  4. In the License type field, select Cluster.
  5. In the Compliance type field select License count to check for compliance calculated by comparing actual usage against the entitled license count.
  6. Right-click in the record's header bar and select Save from the context menu.
  7. Related Lists appear on the form.
  8. Select the Related License Related List.
  9. Click New to create a related infrastructure license, or click Edit to select an existing license.
  10. Ensure that the related license describes the same locality and software package as the cluster license.
  11. Include all unique information for this instance of the software package, such as entitled usage, license key, and expiration date.
  12. Ensure that the cluster record number appears in the Parent field, and then save the record.
    The Related Lists are hidden, as are the fields for actual license count (Used). These totals are now calculated by the cluster license.


  13. Continue adding related licenses for all the permutations of this software package.
    The completed cluster license displays the entitled license count and the actual usage. The license is In compliance if the number of CIs using the license is less than the entitlement, regardless of whether the CIs are licensed to use the software package.


3.1 Example

The Acme Corporation wants to check the compliance of their Adobe Acrobat 7 licenses at all their remote locations. The administrator creates a complex license for the corporate level and then creates a hierarchy of related complex licenses for the company's regions and city offices. He discovers that there are licenses for Acrobat that were purchased at different times. There are a number of these licenses scattered around the organization, and each one has a different license key. He decides that the best way to deal with the different licenses is to use cluster licenses to calculate compliance in one place for each locality. He creates a hierarchy that enables him to roll up the license count from each remote office and check compliance at each locality upward.
