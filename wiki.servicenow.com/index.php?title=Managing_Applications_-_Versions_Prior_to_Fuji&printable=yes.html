1 Overview

The system offers administrators several ways to manage applications.

Working on applications has changed significantly starting with the Fuji release. For the current version, see Managing Applications.

2 Select the Current Application

Application developers can use the application picker to select the application to which any new records and updates apply.

  1. Click the gear icon in the banner frame (starting with the Eureka release).
  2. From Application, select the application where you want to save changes.
    Applications picker

3 Update the Application Record

Application developers and administrators can update a custom application record to add new features or change application functionality.

  1. Navigate to System Applications > Custom Applications.
  2. Click the application name for the application you want to update.
  3. Enter or select new record values (see table).
    Sample custom application record

  4. Click Update.
Field Description
Name [Required] Enter a label for the application. Changing the name does not change any other field value derived from the application name such as the Menu or User role fields.
Short description Enter a description of the application purpose or usage.
Active Select whether this application is available to users. See Deactivate Application.
Menu Select the application menu where you want to display modules. For more information about menus and modules, see Administering Application Menus and Modules.
Version Enter any version information for the application.
User role Select the user role required to access the application menu. For more information about user roles, see Creating Roles.
Related lists
Tables View or create application data tables.
Roles View or create user roles for the application.
Menus View or create menus for the application.
All Files View configuration records associated with this application in platform feature tables.

3.1 Add Tables

Application designers can add new tables to store application data.

  1. Navigate to System Applications > Custom Applications.
  2. Click the application name for the application to which you want to add a table.
  3. From the Tables related list, click New.
  4. Complete the table form. See Creating a Custom Table.
    The system automatically associates the new table to the application.

3.2 Add Roles

Application designers can add new user roles to control access to application menus and data.

  1. Navigate to System Applications > Custom Applications.
  2. Click the application name for the application to which you want to add a role.
  3. From the Roles related list, click New.
  4. Complete the User Role form. See Creating Roles.
    The system automatically associates the new user role to the application.

3.3 Add Menus

Application designers can add new menus to organize application modules.

  1. Navigate to System Applications > Custom Applications.
  2. Click the application name for the application for which you want to add a menu.
  3. From the Menus related list, click New.
  4. Complete the Application Menu form. See Creating Application Menus.
    The system automatically associates the new menu to the application.

3.4 Add Update Sets

Application designers can save a version of their application in an update set if they want to share it with other instances or need an external copy of the application. See Publishing Applications - Versions Prior to Fuji.

4 Set the Application State

Administrators can deactivate, activate, and delete applications from the custom application record.

4.1 Deactivate Application

Administrators can deactivate applications they no longer need to remove user interface elements but preserve all application data.

Note: Administrators can still access application data tables from the application filter or direct URLs to the tables.


To deactivate an application:

  1. Navigate to System Applications > Custom Applications.
  2. Select the application you want to deactivate.
  3. Clear the Active check box.
  4. Click Update.
    • The system deactivates all associated menus, modules, and access control rules.
    • They system prevents non-admin users from accessing the application tables from the application navigator.
    • The system allows you to create new menus, modules, and access control rules for this application, but prevents you from making them active.

4.2 Activate Application

Administrators can reactivate a deactivated application to restore it to normal service. You can selectively deactivate application menus, modules, and access control rules after you activate the application.

To activate an application:

  1. Navigate to System Applications > Custom Applications.
  2. Click All in the breadcrumbs to remove the default active = true filter condition and display both active and inactive applications.
  3. Select the application you want to activate.
  4. Select the Active check box.
  5. Click Update.
    The system activates all associated menus, modules, and access control rules.

4.3 Delete Application

Application developers can delete applications they no longer need. Typically, application developers only delete applications that have never been shared with other instances. This ensures they can support any published applications.

By default, deleting an application also deletes all the records that are associated with the application, such as tables, business rules, and menus. Alternatively, you can deactivate an application that is no longer in use to preserve all application data.

Note: You can always reinstall an application from a backup update set.


To delete an application:

  1. [Recommended] Back up the application by publishing it to an update set. See Publishing Applications - Versions Prior to Fuji.
  2. Navigate to System Applications > Custom Applications.
  3. Open the application record.
  4. Click Delete. The system displays a confirmation dialog box indicating the number of application files in the application to be deleted.
    Confirm deletion

  5. [Optional] Click Show files to cancel deletion and navigate to a list of the application files in the application.
  6. [Optional] Clear the check box to delete only the application record and not the rest of the records. You may want to keep some files if you plan on reinstalling the application later.
  7. Click OK.
  8. In the confirmation dialog box, enter delete and click OK.
    • The custom application record is deleted.
    • If you selected the check box in step 6, the system deletes all records from application tables. However, the system does not delete tables extended in another applications (starting with the Dublin release).

4.4 Restoring Deleted Records

In some cases, you can restore deleted configuration records.

Limitations for restoring records include:

  • You cannot restore a record if its parent record is deleted. For example, you cannot restore a field on a deleted table. You must restore the parent record first.
  • You cannot restore a table if it extends a deleted table.
  • When a record is deleted, the application file stores a reference to the current record version. You can only restore records when this reference and version record exist.

To restore an application that you just deleted:

  1. On the deleted application confirmation page, click the Show Application File link.
  2. Under Related Links, click Restore Deleted Application.
    Restore all application records

    The application and all records that were deleted with the application are restored.

To restore an older application or another configuration record:

  1. Navigate to System Applications > Restore Application Files.
  2. Locate the application file by filtering on the source table of the configuration record. For example:
    • For an application, filter on Source Table is Application and look for the application name.
    • For a table, filter on Source Table is Table and look for the table name.
    • For a field, filter on Source Table is Dictionary Entry and look for the field name.
  3. Open the application file.
  4. Under Related Links, click the link to restore the record.
    • The record is restored.
    • If the record is an application, table, or field, all descendant records that were deleted with it are restored. For example, restoring an application also restores all records that were deleted with the application. Restoring a table also restores all the fields and labels that were deleted with the table.

5 Assign Items to Applications

Application designers can move some or all items between applications directly but may produce conflicts that need to be resolved. Application designers can also choose what update set to assign changes to as well as include homepages and UI pages in the update set.

5.1 Manually Moving Application Files

The most efficient way to build applications is to select an application in the application picker so that new application files are automatically assigned to it and you are notified of potential conflicts. However, you can manually assign application files to applications by:

  • Moving an application file with all of its descendant records (recommended). For example, when you move a UI policy, also move its UI policy actions. Similarly, when you move a table, also move all the fields and labels on the table.
  • Moving a single application file without affecting its descendants. Use this method with caution; if you move an application file without its descendants, items may be missing when you publish the application.

Move files manually to assign unchanged application files to an application or to move application files from one application to another.


To move an application file with all of its descendant records (recommended):

  1. Navigate to the record in a list or form view. For example, System Definition > UI Policies.
  2. For a list, select the check box beside each appropriate row, and then select Move to Application in the Actions choice list.
  3. For a form, right-click the header and select Move to Application.
  4. Select an application from the list. The list defaults to the last application you selected.
    Move application files to an application

  5. Click Add.
    The application file and all of its descendant records are moved to the selected application.


To move a single application file without affecting its descendants (use with caution):

  1. Navigate to the record.
  2. Right-click the header and select Show Application File.
  3. In the Application field, select the desired application.
  4. Click Update.
    Update the application file record

5.2 Resolving Conflicts

A conflict may exist when application files that normally belong together are not assigned to the same application. A notification appears when the application file you changed or its parent is not assigned to the application selected in the application picker. For example, a conflict is detected in this scenario.

  • Table A is associated with Application A.
  • Application B is selected in the application picker.
  • You create a new field on Table A.


To resolve conflicts, choose an action for each conflict that is detected.

  • To move the application files to the recommended application, click Move Files.
  • To leave the application files where they are, click Leave Alone.
Resolve each conflict

5.3 Add Homepages and Content Pages to Applications

Administrators must manually add homepages and content pages to applications and update sets prior to distribution. This process only stores the version of the page as it existed at the time of inclusion. Any further changes to these pages requires an administrator to manually add them again.

To manually add the current version of a page to an application and update set:

  1. Navigate to Homepage Admin (or Content Management) > Pages.
  2. Right-click a homepage or content page record.
  3. Select Unload Portal Page.
    The current version of the page is added to the current application (starting with the Dublin release) and to the current update set.
