1 Overview

The Security Assertion Markup Language (SAML) is an XML-based standard for exchanging authentication and authorization data between security domains. SAML exchanges security information between an identity provider (a producer of assertions) and a service provider (a consumer of assertions). SAML is a product of the OASIS Security Services Technical Committee. When implemented correctly, SAML is one of the most secure methods of single sign-on available. However, it is also one of the most difficult to implement. To use SAML, you should understand the standard as defined by OASIS. More information regarding SAML 1.1 can be found at wikipedia.

Browser/POST Profile Detailed Processing

The SAML 1.1 Browser/POST profile typically has this workflow.

  1. User accesses local IdP site.
  2. IdP challenges user for credentials.
  3. User provides credentials and logs into IdP. The user typically has access to an application portal.
  4. Application portal displays remote application links from service providers.
  5. User selects remote application from service provider (for example, ServiceNow).
  6. Inter-site transfer service creates an HTTP form including the SAML response and authentication assertions.
  7. Browser posts the HTTP form to the assertion consumer.
  8. Assertion consumer creates a cookie and redirects the user to the service provider (for example, ServiceNow).
 Note: New customers are encouraged to implement a SAML 2.0 integration instead of SAML 1.1 integration. Existing customers are also encouraged to migrate to a SAML 2.0 implementation to take advantage of increased security features. Customers who choose to remain on SAML 1.1 should install update 1 to add the audience parameter to their integration.


2 Integration Requirements

The SAML 1.1 single sign-on integration requires:

  • Activating the latest version of the SAML 1.1 plugin (update 1)
  • Access to a SAML 1.1 Identity Provider (IdP)
    • Must implement SAML over SOAP over HTTP
    • Must use the Browser/POST SAML Profile
      • Must authenticate users with a header, parameter, or cookie
      • Must POST a SAML value in an HTTP header
      • Must base64-encode the HTTP header
    • Must provide an inter-site transfer service
    • Must have a valid certificate


 Note: At this time, the integration can only use the Browser/POST profile, not the Browser/Artifact profile.


3 SAML 1.1 Setup

Here is an overview of tasks to enable single sign-on with SAML 1.1.

  1. Request the SAML 1.1 update 1 plugin.
  2. Create a local SAML 1.1 Identity Provider (IdP).
  3. Set IdP metadata.
    1. Specify HTTP header that contains SAML communications.
    2. Specify XML element that contains login information.
  4. Set the instance URL.
  5. Set the User table field that matches the SAML token.
  6. Install the IdP certificate.
  7. Test the integration.

3.1 Step 1. Request the SAML 1.1 Plugin

Contact customer support to request the SAML 1.1 Single Sign-On Update 1 plugin. The plugin installs a new application called SAML Single Sign-on.

3.2 Step 2. Create a Local SAML 1.1 Identity Provider (IdP)

The SAML 1.1 integration requires a software developer to create a local SAML 1.1 IdP. The local SAML 1.1 IdP authenticates users and sends authentication assertions to one or more service providers. The local SAML 1.1 IdP must meet these criteria:

  • Conforms to the SAML 1.1 specification.
  • Authenticates local users and passes authentication assertions.
    • Parses incoming HTTP requests to identify the targeted resource at the Service Provider (in this case the targeted ServiceNow instance).
    • Passes authentication assertions to the inter-site transfer service.
      		Note: The IdP developer should decide what XML key to pass in the authentication assertion. This key must eventually match a value in a ServiceNow User table field. Typically, the key matches to values in the user_name or email field.
  • Publishes an inter-site transfer service.
    • Passes an HTML form that contains the SAML response and authentication assertions to the service provider.
    • Specifies the URL of assertion consumer (the URL to ServiceNow instance)
 Note: It typically takes a developer 6-8 weeks to create and configure a local IdP. During that time, it is recommended that you implement another SSO method.


3.2.1 Add XML Key Values to ServiceNow Instance

After the IdP developer determines the XML key the integration uses, ensure that the ServiceNow User table field contains appropriate matching values. For example, if the integration uses the email field as the XML key, then the administrator must ensure that the ServiceNow system lists the same email address as the IdP. The integration fails to authenticate any user who does not have a matching value for the XML key.

3.3 Step 3. Set IdP Metadata

The SAML integration needs to know how the local IdP authenticates users.

  1. Navigate to SAML Single Sign-on > Properties.
  2. Enter the IdP metadata for these properties. See the table of SAML properties.
    • HTTP header name that contains the base64-encoded SAML value.
    • The name of the XML element that contains the login information.
    • If the XML element appears more than once, this value represents which of these XML elements to use. This is a 0-based index so the first occurrence would be offset 0, the second occurrence is 1, etc.
  3. Continue to Step 4. Set the Instance URL.



3.4 Step 4. Set the Instance URL

SAML 1.1 requires the service provider (SP) to publish an Assertion Consumer service URL. In this case, the Assertion Consumption service URL is the URL of the ServiceNow instance.

  1. Enter the IdP metadata for the property The intended audience to receive the SAML response (saml:Audience). See the table of SAML properties.
  2. Continue to Step 5. Set the User Table Field that Matches the SAML Token.

3.5 Step 5. Set the User Table Field that Matches the SAML Token

The XML key in the SAML response must match a existing value in a ServiceNow User table field. The XML Key is the XML element in the SAMLResponse that contains the user's authentication information. The XML key is typically a unique value for the user such as a user name or email address. The IdP developer determines what the XML Key is when creating the local IdP.

  1. Enter the IdP metadata for the property What ServiceNow user field to coalesce against the XML value corresponding to the XML key. See the table of SAML properties.
  2. Click Save.

3.5.1 SAML Properties

Set these SAML properties in SAML Single Sign-on > Properties.

Property Description
Enable external authentication Select this check box to require all users to use SAML 1.1 single sign-on. If anyone tries to access ServiceNow in an unauthenticated state, the instance automatically sends an authentication request to the IdP and redirects the user to the SAML IdP authentication page.
HTTP header name that contains the base64 encoded SAML value Typically, this property does not need to be changed unless your local IdP posts the response using a POST variable other than SAMLResponse.
The intended audience to receive the SAML response (saml:Audience) [Only available with update 1] Enter the name of the ServiceNow instance. This is the URL for the Assertion Consumer service.
The name of the XML element that contains the login information Enter the XML element that contains the login information. For example, if the XML contained <SM:Username>ajandersen </SM:Username>, then SM:Username would be the XML element. If the XML contains <saml:Attribute>AttributeName="SMContent"AttributeNamespace=...</saml:attribute>, then the XML element would be saml:Attribute.
If the XML element appears more than once, this value represents which of these XML elements to use. This is a 0-based index so the first occurrence would be offset 0, the second occurrence is 1, etc. If there is more than one XML element that has the name as defined in the last property, specify which one to use. The value of 0 means that the first occurrence of the XML tag is the one to use.
What ServiceNow user field to coalesce against the XML value corresponding to the XML key. Enter the field name from the User table where the integration can find a value to match the XML key. Typically, this value is either from the user_name or email field.
Enable replay cache. Activating this property will ensure that the same assertion cannot be used more than one time. For security purposes, set this field value to Yes.
Validate assertion conditions. Disabling this property is a security risk and should only be disabled for debugging purposes. For security purposes, set this field value to Yes. Disable this property only to debug the SAML integration or when configuring it for the first time. When disabled, the instance ignores the reply cache, valid time window, and other security checks, however it still evaluates the certificate signature to ensure it matches the signature in the x509 Certificate module.
Enable verbose logging Select this check box to enable expanded logging (larger, more informative log records).

3.6 Step 6. Install the IdP Certificate

Identity providers use a certificate to verify communications with the service provider. The IdP's certificate is located within the IdP's metadata. The IdP developer determines where the certificate metadata resides when creating the local IdP.

To install the certificate:

  1. Navigate to SAML Single Sign-on > x509 Certificate.
  2. Do not change the Name entry. The name of the X.509 certificate must be SAML 1.1 in order for the integration to use it.
  3. In the PEM Certificate field, enter the value of the X509Certificate.
  4. Click Save.

 Note: The integration does not currently sign the certificate in communications between ServiceNow and the IdP.


3.6.1 Replacing a Missing Certificate

If the ServiceNow SAML Single Sign-on > Certificate module displays a blank page, someone has deleted the SAML 2.0 certificate record.

To fix the missing certificate:


If an administrator clicks on the x509 Certificate module and sees a blank page, someone has deleted the SAML 1.1 certificate record. To fix the missing certificate:

  1. Navigate to System Definition > Certificates.
  2. Create a new record called SAML 1.1.
  3. Navigate to SAML Single Sign-on > x509 Certificate.
  4. In the PEM Certificate field, enter the value of the X509Certificate.
  5. Click Save.

3.7 Step 7. Test the Integration

After completing all other setup tasks, test the integration.

  1. Log in to the instance as user with the admin role.
  2. Navigate to SAML Single Sign-on > Properties.
  3. In the Enable external authentication property, select Yes.
    		Note: Enabling external authentication requires all users to use SAML 1.1 single sign-on. If anyone tries to access ServiceNow in an unauthenticated state, the instance automatically sends an authentication request to the IdP and redirects the user to the SAML IdP authentication page.
  4. Click Save.
  5. Log out of the instance.
  6. Browse to the instance URL. If the integration is functioning properly, the IdP should ask for the user credentials.
  7. Provide the IdP user credentials. If the integration is functioning properly, the IdP should forward the user to the instance as the appropriate user.

3.7.1 Troubleshooting Login Errors

Here is a list of common login errors and their solutions.

Error or Symptom Solution
Authentication fails and the login request generates an infinite loop between ServiceNow and the IdP. Set (or create) the system property glide.authenticate.failed_redirect to redirect failed authentication requests to a URL. Typically the URL endpoint is an error page or logout page.
Login requests generate an infinite loop between ServiceNow and the IdP when High Security is active. The High Security plugin's Rotating Session feature can cause problems with the SAML 1.1 authentication process. Disable the Rotating Session feature when using SAML 1.1 for authentication. SAML 1.1 needs to redirect URLs to an IdP. When Rotating Sessions are enabled, any redirection causes the session to rotate and forces the instance to query the IdP again. This causes an endless loop of new sessions between the instance and the IdP.
The IdP successfully authenticates the user, but ServiceNow cannot find a matching token. Verify that the integration properties (SAML Single Sign-on > Properties) contain the correct User table field to match XML key values against. If the instance properties contains the correct User table field, review the External Authentication and SAMLEndPoint login scripts. Verify that these script pass the correct XML key value. The scripts may contain hard-coded values for the user_name or email fields.

4 Upgrading an Existing SAML 1.1 Integration

Contact customer support for an upgrade. Activating Multiple Provider Single Sign-On automatically activates SAML 2 Update 1, starting with the Fuji release.
