1 Overview

The Field Service Management Plugin includes an inventory management feature, whose goal is to coordinate part selection and delivery to the Field Service Agent.


Inventory Management tracks the following information pertaining to parts:

  • Location (example: warehouse)
  • Status (example: reserved)
  • Movement
  • Status levels
 Note: The information on this page explains Inventory Management v1. For information about Inventory Management v2, see Inventory Management v2.


1.1 Parts

The Parts module in the Field Service Management plugin corresponds to the current ServiceNow Configuration Item model. This model is also used in the Enterprise CMDB Plugin. The Field Service Management plugin adds the concepts of consumable to trace quantity and part number to facilitate selection. For more details, see Enterprise CMDB Plugin.

There are two types of parts:

  • Consumable Parts
Consumable parts are traced by the system quantitatively. When the part is used or installed at the service order location, the quantity of that part is updated in the system. Note that attributes, such as where the consumable part is currently installed, are not tracked.
  • Non-consumable Parts
Non-consumable parts correspond to configuration items defined up-front in the system. The complete life cycle of configuration items is tracked by monitoring changes of status and location. A configuration item created or modified in the system becomes an inventoried part only if it fulfills the following conditions:
  • CI location is a warehouse currently defined in the system
  • Model ID is a part currently defined in the system

The creation, update, or deletion of a configuration item automatically updates the quantity of the non-consumable part available in the system. For more detailed information, see Inventory Items.

1.2 Warehouse Types

To optimize part selection, the warehouse type has two significant characteristics:
  • Priority: suggests the order of warehouses the parts should be sourced from. Example: if the “Field Agent Vehicle” of the assigned agent contains the required part, it should be preferred to the “Central Warehouse” because the part in the Agent's vehicle will not require delivery
  • Shipment Required(true/false): informs the system if “transfer order” creation will be necessary if the part is sourced from the warehouse of the given type. Example: the part in the “Field Agent Vehicle” will not require shipment so no transfer order needs to be created.


Currently, the Field Service Management plugin defines the following types of warehouses:

Value Name Priority Shipment Required Description Comment
1 On Site 1 False Warehouse on customer site Close to the users, does not require shipping.
2 FSA Car 3 False Corresponds to the Field Service Agent car. Treated as a small warehouse with set of parts.
3 FSL 4 True Forward Shipping Location Small warehouses where the parts can typically be shipped via overnight delivery.
4 PUDO 5 True Pick Up/Drop Off location This type is called sometimes a "by-box”. Can be a postal box where the parts are delivered for the Field Service Agent and where parts can also be returned to the warehouse. Often in close proximity to major customer sites.
5 Warehouse 6 True This is a standard warehouse. There is usually one per region.
6 Central Warehouse 7 True This is a central warehouse. Usually a large facility from which many parts are shipped.
7 Field Agent 2 False Virtual Warehouse linked with Field Service Agent directly, used for delivery. Mainly used to indicate to the system that the part has been delivered and is with Field Service Agent.

1.3 Field Service Agent (FSA) Warehouses

The Field Service Management plugin creates a new many-to-many relationship between Field Service Agents and warehouses. This relationship functions three ways:
  • To determine part delivery location once the FSA is assigned to the service order.
  • To link the vehicle or warehouse to an FSA.
  • To handle the delivery of a part to the FSA.
Note regarding the third function: every time the role “fsm_agent” is assigned to a user, a personal warehouse of the type “Field Agent” (value 7) is automatically created. When a part is assigned the location of “Field Agent” warehouse, the system records that part as having been delivered. This allows the transfer order to be closed. The personal warehouse is deleted as soon as the role is withdrawn from the user.

2 Inventory Items

The Inventory Item in the Field Service Management plugin is created by the association among four different data points:

  • Warehouse
  • Part
  • Status
  • Sub-Status

If any of the four data points change, a new record is generated.

Examples:

Inventory Number Part Warehouse Status Quantity
INV01000101 USB10 USB Cable Southern California Warehouse In Stock 45
INV01000102 USB10 USB Cable Southern California Warehouse Reserved 2
INV01000103 USB10 USB Cable San Diego Warehouse In Stock 10

2.1 Inventory Item: Consumables

Inventory Items corresponding to a part tagged consumable are managed only in terms of quantity and have no objects related to them.

The movements of these items between warehouses and different statuses or sub-statuses are tracked during different parts movement linked to their life cycle. They can be transferred from one warehouse to another, consumed during in the Service Order, installed, decommissioned, or sent out to the location outside the company warehouse system. This is a prevalent action during “part sourcing” and “execution” (status changes) as well as during the transfer of the parts (warehouse and status changes).

Change of status or warehouse location of a transferred part implies:

  • The creation of a new target inventory item (if necessary).
  • The addition of transferred quantity to the quantity of already existing inventory item (the same combination of warehouse, part and status).
  • The subtraction of the transferred quantity from the quantity of the original Inventory Item.

2.2 Inventory Items: Configuration Items

Inventory Items corresponding to a part which is not tagged as consumable are seen as configuration items located in the one of the warehouses. Management of these items is performed at two levels: the Inventory Item level and Configuration Item level.

The movements of these items between different warehouses or statuses are performed the same way as for the “consumable” inventory items described above. Changes to configuration items are reflected at the Inventory Item level. This includes the modification or creation of a corresponding Inventory Item Line and a quantity. The opposite is also true: any changes on CI level will automatically modify or create the corresponding Inventory Item and its quantity.

2.3 Inventory Item Status

As Inventory Items or CIs are created or updated throughout their life cycle, two sets of status have to be mapped. Inventory Item lines are created or updated as changes occur to the status and sub-status of a consumable part; the same is done for CIs. However, the CIs can exist outside the Inventory Management System. The result is two sets of status and sub-status values, one for CIs and another for Inventory Items.

3 Transfer Orders

The Inventory Management add-on allows to create a new transfer order for all the part movement inside the company warehouse system and outside it. This is done through the dedicated module called Create Transfer Order.

For the automated creation of the Transfer Order during the part-sourcing phase, see Field Service Management plugin for more details.

3.3 Create “Return Material Authorization” Transfer Order

In some cases, especially for material under warranty, it is important to get the vendor authorization for the transfer of parts. This authorization is often accompanied by the RMA Number. This type of Transfer Order is called an RMA Transfer Order and requires approval before shipment.



This type of TO triggers the default Workflow called Transfer Order RMA approval shown below:



The special User Group “RMA Approvers” has been created for the purpose of the approval of RMAs.


If the Transfer Order is of type RMA, the change of stage won’t be possible until it’s status change to approved and the RMA Number is provided. This can be done via the normal approval mechanism or directly into Transfer Order if the RMA is obtained by phone and the RMA number is known and entered.

3.4 Transfer Order Deletion

After the transfer order created it’s still possible to delete it. However, this operation is possible under one condition. The Tranfer Order has to be in one of the two following stages: Awaiting Dispatch or Shipment Preparation. If this a case, the deletion will trigger the following steps:

  1. The original Inventory Item line is incremented by the quantity recorded in Transfer Order and the Inventory Item line corresponding to the status of “Pending Transfer” or “In Transit” is decreased by the quantity from Transfer Order.
  2. The CIs original status is restored. This is deduced from CI substatus as its value is unique and doesn’t change across the Transfer Order life-cycle.
  3. The TO is finally deleted from the system.

4 Activating the Plugin

Click the plus to expand instructions for activating a plugin.

If you have the admin role, use the following steps to activate the plugin.

  1. Navigate to System Definition > Plugins.
  2. Right-click the plugin name on the list and select Activate/Upgrade.
    If the plugin depends on other plugins, these plugins are listed along with their activation status.
  3. [Optional] If available, select the Load demo data check box.
    Some plugins include demo data—sample records that are designed to illustrate plugin features for common use cases. Loading demo data is a good policy when you first activate the plugin on a development or test instance. You can load demo data after the plugin is activated by repeating this process and selecting the check box.
  4. Click Activate.
