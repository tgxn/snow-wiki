Winter 2009 Notable Changes

From ServiceNow Wiki
Winter 2009 Notable Changes
Jump to: navigation, search

The following are notable changes you may encounter with our Winter 2009 release. Not necessarily problems, not necessarily major features, just things or behaviors you may notice being different than before.

1 Removal of IR Relevancy scores from Catalog and Knowledge searches

Results can still be sorted by IR Relevancy, but the actual score will no longer be visible. Each individual score itself had no practical meaning (what did 14% mean on a search result, other than that it was a lower score than 21%?) and was confusing. Now, search results will simply indicate how they are sorted (on view count or relevancy).


2 Wiki navigation add-on for Knowledge Base

Addition of "ServiceNow wiki" navigation add-on for Knowledge Base, allowing wiki searches from the Knowledge Search widget. Can be disabled if desired at Knowledge Base --> Navigation Add-ons


3 Reduced Logging from Upgrade Client

Unless you have explicitly disabled this feature, your ServiceNow instance checks in with our management servers on an hourly basis to see if it needs to upgrade itself. Most of the time it does not need to upgrade.

In builds prior to Summer 2008 Stable 2, the upgrade client would nonetheless write the following entry to the system log: "Already running desired war file, upgrade not required". You would see these messages every hour in System Diagnostics --> Upgrade Log.

Now the upgrade client will only log a message if it needs to upgrade the instance or encounters an unexpected situation.


4 Discovery enhancements

  • Added "chunky ping", automatically breaking up pings of large IP address ranges into smaller chunks. This increases overall Discovery performance plus provides faster feedback.
  • SNMP probes now have a related list for OID specifications, instead of one large text field
  • Added a simple way to use SNMP System OIDs for classification
  • Added events that are thrown when a MID server goes up or down, and a sample email notifier on the MID server down event
  • Added CI Consolidation, useful when Discovery is combined with either imported or manually entered CMDB data
  • We now capture ALL serial numbers present in a device (in a new Serial Number table)
  • Added new probe report, useful for watching Discoveries in progress, analyzing their performance, or troubleshooting problems
  • Added support for OS/X servers (Mac); they are now classified as servers, not workstations
  • Added a MID server UI action (list context/form link) to get the latest stats for any MID server with a single click
  • Enhancement to Export all capability to improve subsequent import success
  • Network adapters and IP addresses are now discovered for UPSs

5 Administration links added to Reports page

"Administration" and "Scheduled Reports" links have been added to the top of the View/Run Reports page for admins who want an easy way to get to those lists. Administration takes you to the sys_report list where reports are defined, and Scheduled Reports takes you to the list of reports scheduled to be emailed. Users with the report_admin role will see these links. If this role does not exist in your instance, you can create it.

6 Default Boilerplate on Common Administrative Forms

If you create a new Script Include, UI Page, or UI Macro, the system will pre-populate the script or Jelly field with appropriate boilerplate for your item.

For example, a script include would default to: 

gs.include("PrototypeServer");

var NewClass = Class.create();

NewClass.prototype = {
  initialize : function() {
  }
}

Whereas a UI Macro would default to: 

<?xml version="1.0" encoding="utf-8" ?>
<j:jelly trim="false" xmlns:j="jelly:core" xmlns:g="glide" xmlns:j2="null" xmlns:g2="null">

</j:jelly>

7 .form and .list navigation in nav filter

Using the Navigator filter to search for the name of a table followed by .list or .form will take you to the list or form for that table. This is the same capability that already existed if a user typed tablename_list.do or tablename.do into the browser's address bar.

e.g., incident.list will take you to the incident list, incident.form will take you to a blank incident form.

8 "since" operator changed to "since baseline"

The "since" operator has been changed to "since baseline", since that is what it is.

9 Label height consistency

The label heights of some field types have been adjusted to make them consistent across the board and across browsers. (e.g., in IE, the field label height for a choice list field was 1 pixel less than the same label for an input field.)

10 Catalog Category Loops

You are no longer allowed to created a Service Catalog category loop with parentage.

11 Domain Multiple Updates

For multiple update the domain field will now be empty rather than defaulting to "global"

12 "Reply to" override for Email Client

You can now override the Reply to value for the email client. Previously you could only override the From value.

13 Delegate notifications

A distinction is now made for delegate notifications between Email Notifications and Meeting Invitations. Another checkbox has been added to the Delegate form for Meeting Invitations.

14 Perspective information

If a user has a perspective other than "All", we provide a visual indication in the form of a subtle "Show all applications" link just beneath the navigator controls.

15 Catalog and Wizard client scripts converted to UI Policies

Catalog and Wizard client scripts have been converted to UI Policies instead.

16 Link URL moved to bottom of notification

Due to the fact that handheld devices show the complete "Link" URL for email notifications, it has been placed at the bottom all notifications to ensure legibility both in email clients and handhelds.

17 Parent exposed on related list UI Actions

The parent record is now exposed on related list UI Actions. For example, if you wanted to disable Affected CIs New and Edit buttons for closed changes you could copy the global m2m UI Actions to the task_ci table and add a condition of parent.active.



18 Scrolling News Moved to Gadgets

The scrolling news homepage widget used to be in its own content category "Scrollers". For Winter 2009, it has been moved into the gadgets category and the "Scrollers" category (which had only one thing in it) has been removed. Existing homepages will not be affected, and new home pages can, of course, continue to use the news scroller, it just has to be selected from the gadgets category now.

19 Reload Form Context Menu

Reload form can be access from the form context menu. It will reload the current form, clearing all unsaved changes. If Dirty Form Support is enabled you will still be prompted to confirm the reload.

20 Personal List "cog" removed from reference lookups

The personal list "cog" (allowing a user to define a personal set of columns for their list) has been removed from reference lookup lists (i.e., the list you see when clicking a magnifying glass next to a reference field).

21 Email Watermark format changed

The watermark in an email notification has changed from something that looks like Service-Now.incident.c9fe63344656747200010de35e52b63g- to a more friendly format that looks like Ref:MSG0025442. The new format allows for effective matching of email replies back to the record that triggered the original email.

22 Some Form Sections are not displayed

Previously, if the "CustomView" (for example) view of a form had multiple form sections defined for it, but there was not an actual multi-section form defined for that view, the application would display all form sections for that view in alphabetical order. With Winter 2009, this behavior changed to only display multiple form sections if a multi-section form is actually defined. In the event that there were customers who relied on this old behavior, they may see forms that previously had multiple sections missing all but one section. To restore old behavior, a multi-section form can be defined in System UI --> Forms. If help is needed, Technical Support can assist, if needed. For assistance, please open an incident with Technical Support.

23 Email Stationery capability removed

Email Stationery was a capability introduced (BETA) in the Summer 2008 Stable 2 build, but it was removed for the Winter 2009 release. While it may return in a different form at some point, it was not ready for prime-time.

24 Journal fields are not filterable

Journal fields (Additional comments, Work notes, etc.) do not appear in the filter widget. If a filter needs to query on text being in or not in a journal field, we recommend a "for text" or Keyword condition be added.

25 Personalize List "cog" might not show any fields in Available or Selected

The list_mechanic UI Macro was modified for the Winter 2009 release, and customers who had previously customized that UI Macro in their instance will not get the modified version. This may result in the Personalize List capability not showing any fields in the Available or Selected sides of the slushbucket. In this event, you can look to /demo for the list_mechanic UI Macro, and copy its contents to your list_mechanic UI Macro to resolve the issue.

Doing this will overwrite any customization, so if the customization was critical it will need to be added to the new version of the UI Macro.
