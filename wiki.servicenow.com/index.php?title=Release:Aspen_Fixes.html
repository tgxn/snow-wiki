PRB564742 The BSM map display entered an infinite loop when clusters contained circular references, causing the server to crash.
PRB446497 BSM map nodes with excessively long CI names improperly rendered with a scroll bar in Microsoft Internet Explorer and partially obscured the node beneath it. Number Details
PRB561392 The Windows classify sensor generated the following TypeError: Cannot convert null to an object.
PRB562905 Application dependencies might not map correctly for Application Mapping with Discovery (Oracle and Tibco).
PRB562582 Cluster node discovery could incorrectly see nodes as clusters. A script was added to the release to delete all nodes that are not correctly associated with a cluster.
View the steps to reproduce
Discover the subnet with the cluster and cluster nodes.
PRB561484 The serial number identifier could perform poorly. The identifier no longer tries to query the serial number table if there is no serial number to match against initially.
PRB561520 Duplicate entries in the dscy_credientials_affinity table were causing the table to grow at a faster than normal rate.
PRB562907 SQL Server instances on clusters were not properly discovered, because virtual IP addresses were not included in the list of CMDB CI IP addresses. This caused incomplete tcp half-connections on Windows.
PRB563411 In Discovery, when the Enforce Synching of IP Addresses option is set to Yes and the CI has multiple NICs, one of the IP addresses associated with the NICs is used as the IP Address field of the CI.
PRB563870 Discovery was producing a null pointer exception warning when running an SNMP classifier. The method used by the platform to download the Management Information Base (MIB) records was changed, which fixed this issue.
PRB563983 SSH Discovery probes could not authenticate with the private key credential for some devices on the network.
PRB564381 Not all networks were being added to the Network IP table in Discovery.
PRB564420 In some cases, Discovery missed discovering software packages on 64-bit Windows systems.
PRB257992 Users with the discovery_admin role did not have access to Discovery schedules or credential forms.
View the steps to reproduce
  1. Grant an ITIL user the customer_admin role.
  2. Attempt to create a Discovery schedule.
   The user sees an empty form.
PRB068711 Running SSHCommand probes against "non-standard" network hardware (switches, PDUs, etc.) caused the probe to hang.
PRB559138 Discovery was populating the Manufacturer field incorrectly in software package records.
PRB563412 Any Discovery schedule that included a Debian Linux server hung the worker processes on the instance when Discovery attempted to parse the result of the Linux – Installed Software probe. Once all worker processes were hung, Discovery was unable to run any schedules or discoveries.
PRB561520 The Credential Affinity [dscy_credentials_affinity] table was growing to an unmanageable size from duplicate entries appearing in the table. Additionally, the instance was attempting to download the entire table for the MID Server. Both circumstances could cause the instance to run out of memory. The duplication issue has been resolved, and the instance now transfers the affinity information to the probes rather than downloading the entire table for the MID Server.
PRB561767 Windows Cluster information was not being populated in the Discovery results. The cluster information was returned during classification, but the cluster probe was not triggered. The issue was fixed by matching the case of the node name returned from the cluster to a host name.
PRB562904 Discovery could not detect all the IP addresses assigned to a single network interface card (NIC) on a Windows Server. This resulted in classified process connections being missed and an incomplete list of IP addresses in the configuration item (CI) record.
PRB563788 Discovery was relying on detecting persistent TCP connections between an Oracle instance and its listener to create the proper relationship. Discovery could not reliably detect the TCP connections and defined these relationships incorrectly. Discovery now uses the listener configuration to establish the correct relationship with Oracle instances regardless of any TCP connection information it detects.
