1 Overview

Commonly used methods for controlling field behavior and automatically setting values are Client Scripts and Business Rules. Each method has its own distinct advantages and are best used together to facilitate safe, easy record updating. This page details the benefits of using both methods to set values, but is not meant to be a comprehensive discussion of every possible method.

2 Client Scripts

Client scripts are written in JavaScript and execute within the browser only (NOT on the server). Examples are: Client Scripts, Catalog Client Scripts, Wizard Client Scripts, UI Policy (both script and filter-based actions), and UI Scripts.

2.1 Advantages

Values set in the browser by a Client Script are dynamic, and the change to the record can be seen immediately on the form by a user. An example of this can be seen in an Incident record with the dynamic interaction between the Impact, Urgency, and Priority fields.

2.2 Disadvantages

  • Client Scripts do not run on lists. Field values edited in a list will not be set with the same interaction as those values on a form controlled by a Client Script. In our example from the Incident table, editing the Impact, Urgency, and Priority fields in the list view does not retain the relationship between the fields that is defined for the form by the Client Script and will introduce errors into the record.
  • Security measures implemented in Client Script are easily subverted. Security implemented by a client-side script should be supplemented with security on the server provided by Business Rules or Access Control.

3 Business Rules

Business Rules refers to any JavaScript that is executed on the server. This includes Business Rules, Access Control, Script Includes and much more.

3.1 Advantages

The same changes can be made to a record from a form or from a list. List edits, form edits, or even most updates created by a script will trigger a Business Rule, making the nature of these edits universal.

3.2 Disadvantages

Edits to forms controlled by Business Rules are not dynamic. A user must update the record before the changes become visible.

4 Using Business Rules and Client Scripts

The best practice is to implement both Business Rules and Client Scripts to enable users to set record values properly using both forms and lists, and to see immediate changes to the values in forms as these edits are made.

4.1 Advantages

  • The update behavior in the system is universal whether a user is list editing or form editing.
  • The changes to a record are dynamic and visible immediately when editing a form.

4.2 Disadvantages

The same functionality must be implemented twice: once in a Client Script or UI Policy, and once in a Business Rule or Access Control.

5 Use Cases

5.1 Example 1

An organization has a Client Script that sets the email address for a user to first.last@company.com. The administrator likes this because he can see the email address immediately when he enters the user's information. The administrator then performs a bulk import of users from a spreadsheet containing the users' first and last names. His expectation is that each user's email address will be set automatically, as they are when he edits the form. Since the Client Script runs only on the form (the interface to the record), it has no effect on data imported into the record from outside that interface, and no email addresses are created. To solve this problem, the administrator implements a Business Rule that runs when the import occurs and creates the email addresses.

5.2 Example 2

An organization wants to hide the Priority field on an Incident form if the assignment group is Development. They create a UI Policy on the Incident form to do this, but their users can still see and edit the Priority field using the list editor. To rectify this, the ServiceNow administrator applies Access Control to prevent read access to the Priority field when the assignment group is Development.
