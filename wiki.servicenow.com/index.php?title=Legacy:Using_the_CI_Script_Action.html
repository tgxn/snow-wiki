1 Overview

Script actions enable you to perform a wide range of tasks, such as modify a configuration item (CI), change which data is being merged into a CI, or abort a consolidation.

2 Creating or Modifying Delete Actions

To create or modify a delete action, such as aborting a consolidator, the script action looks like this:



The fields on this form:

  • Summary. Unique title for the action. The script action does not use the value in this field.  This summary title appears in the list of actions on the consolidator page.
  • Order. An integer value that controls the order in which consolidator actions run.  Actions with smaller order values run earlier, actions with larger order values run later.
  • Active. If checked, this script action will run when the consolidator runs.  If not checked, the script action will not run.
  • Description.  Brief description of this script action. This description is not used anywhere else.
  • Script.  The JavaScript code that implements your script action.  You must use an anonymous function with exactly the signature shown, but what happens inside that function is entirely up to you.  This function does not return any value.

3 Understanding the Script Action Script

The script action shown above, as well as in the other sample script actions supplied with this plug-in, serve as working examples of how to build a script action.  The first thing to understand are the function parameters:

  • source: A hashmap (in other words, a JavaScript object) of the just-discovered CI's record. For fields in the CI's form (as opposed to a related list record), you can get the value simply by writing source.name, where “name” is the name of any field. The value returned is always a string. The example script uses this twice: once to get the sys_id value, and again to get the serial_number value. Records from any related list specified in Discovery Definition → CI Related Lists are also available, though the syntax is a little more complicated. For example, suppose your finder needed to get access to all the serial numbers captured for a computer. If you've specified the serial number related list in CI Related Lists, you'll see that its related list name is cmdb_serial_number:cmdb_ci. This becomes a field name on the source object, so you can write source.cmdb_serial_number in your script, which will return an instance of CIRelatedList for that list.That class has several properties and methods (see Discovery Definition → Script Includes, under CI Consolidation, for details). One example: in your script, you could write source.cmdb_serial_number:cmdb_ci.records, which would return an array of hashmaps, each of which represents one record in the serial number table.
  • target. A string containing the sys_id of the target (the duplicate CI found by a finder).
  • results. This is another hashmap, but this time it's for your own use. At minimum it will contain one property: “finder”, which contains the name of the finder that located the target CI. If your finders stored other properties in this hashmap, those properties will be present as well.
  • consolidator. This contains an instance of CIConsolidator, which has some useful methods and properties (see the code in Discovery Definition → Script Includes for details). Probably the most useful method is abort(), which aborts the consolidator at the point when it was invoked. This method is used in the example above.
  • log. This is an object that allows you to write messages to the CI Consolidation log (see Discovery Definition → CI Consolidation Log). This object has three methods (info, warn, and error), each of which has a single parameter for the message. You can see this object in use in the example script, on the line just above the return statement.
  • action. This is an instance of CICScriptAction that represents this action. It has several methods and properties (see the Script Includes to see its code), but probably the most useful one is debug. This method allows you to write code like action.debug("my debugging message"), which will messages to the CI Consolidation log only if the CI Consolidation Debugging property is active (see Discovery Definition → Properties).

The rest of what happens inside your script action is entirely up to you. The example above was used during testing, to abort the consolidator before it ran the delete action (note that it is inactive now). A more interesting example of a script action might be arranged to run just before a merge action to modify the source data before the merge. For example, suppose you needed to merge the source to the target, but you wanted to retain whatever name the target already had. You could do this with a script action that ran just before the merge, by reading the target's record, grabbing the name field, then storing that value in the source. The subsequent merge would leave the target's name value unchanged.
