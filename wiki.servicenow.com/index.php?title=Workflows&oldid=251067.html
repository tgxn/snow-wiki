1 Overview

The workflow engine automates multi-step processes and is used throughout the ServiceNow system. Each workflow has a sequence of activities, such as generating records, notifying users of pending approvals, or running scripts. The transitions between these activities are based on configurable conditions.

At each step in a workflow:

  • An activity is processed, and an action defined by that activity occurs.
  • At the completion of an action by an activity, the workflow checks the activity's conditions.
  • For each matching condition, the workflow follows the transition to the next activity.

When the workflow runs out of activities, the workflow is complete.

For more information and for instructions on using the graphical workflow editor, refer to the following pages:

  • Using the Workflow Editor

2 Roles

Name Description
activity_creator Creates and edits custom workflow activities, reuses custom activity data, and manages activity packs downloaded from the ServiceNow Store.
web_service_admin Accesses and uses REST and SOAP messages in the activity designer. Creates and edits custom activities that use the REST web service and SOAP web service templates.
workflow_admin Creates, edits, publishes, and deletes graphical workflows.
workflow_creator Creates new graphical workflows.
workflow_publisher Publishes graphical workflows.

3 Menus and Modules

The following modules are available in the Workflow application:

  • Workflow Editor: Create and edit workflows using the graphical Workflow Editor.
  • Scheduled Workflows: Create a schedule on which to run a workflow.
  • Stage Sets: Organize workflow stages using stage sets.
  • Default Stages (by table): View the list of default fields to use as a stage field, per table.
  • Live Workflows
    • Active Context: View currently running workflows.
    • All Contexts: View currently running or previously completed workflows.
    • Executing Activities: View a list of activities currently running within an active workflow.
    • History: View the list of completed workflows.
  • Administration
    • Activity Definitions: View all available predefined activities.
    • Workflow Versions: View all versions of workflows.
    • Activities: View records for all workflow activities.
    • Transitions: View records for all workflow transitions.
    • Properties: Configure properties for workflow behavior.
  • Operations
    • Workflow Operations Dashboard: Access the dashboard for viewing workflow performance. This module is available starting with the Fuji release.

4 Enhancements

4.1 Fuji

  • Displays workflow performance metrics on the Workflow Operations Dashboard.
  • Collects workflow run time metrics.
  • Manages workflow updates from the ServiceNow Store.
  • Offers a new workflow editor UI.

4.2 Eureka

  • You can cancel a running workflow.
  • The Parallel Flow Launcher activity is available for running and managing multiple workflows simultaneously.
  • Users cannot check out or delete workflows that are associated with a read-only application file.
  • A workflow author can provide language-specific stages.
  • A new workflow stage renderer is available to display the progress of a workflow in a workflow field.
  • Workflows belong to a domain when Domain Separation is enabled.
  • A single workflow can now use multiple Timer activities.
  • Workflows must use a unique name.
  • The workflow engine can cache workflows to improve performance.
  • The Rollback activity is deprecated. Use the Rollback To activity instead.

4.3 Dublin

  • PowerShell framework scripts now synchronize from the instance to a MID server. This change makes it unnecessary to unblock PowerShell scripts.
  • Workflow creators can set the Stage Order field on a workflow's properties. This field controls the order in which workflow stages appear when viewed as a glide list.
  • Workflow activities now display new inline descriptions and context-sensitive help.
  • Workflows now retain data from multiple parallel events when subsequent threads complete before previous threads.
  • A new stage management mechanism automatically populates workflow stage icons, such as those used by service catalog workflows, as a workflow progresses. Previously, workflows that updated stage icons each required a unique business rule. Workflows started prior to upgrading to Dublin do not use the new mechanism.
  • Workflow stage values are automatically updated to conform to localization standards.
