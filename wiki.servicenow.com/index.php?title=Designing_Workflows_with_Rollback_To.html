1 Overview

 Note: Before reading this page, be familiar with the ServiceNow rollback activities detailed on the Approval and Rollback Activities page.


When conditions in a workflow trigger a Rollback or Rollback To activity, the workflow moves processing backward to a specified activity in the workflow and resets certain activities that have already executed back to their original state. This is useful when handling an unexpected failure or as part of a programmed logical flow. When an activity is reset during a workflow rollback, the following things happen:

  • Approvals are reset to Not Requested.
  • Tasks are reset to either Open or Pending. A rollback workflow path cannot create new tasks.

Activities that perform external system operations such as deleting a file or sending an email are not rolled back. Only approval and task activity states are reset.

The Rollback activity is deprecated starting with the Eureka release. Use the Rollback To activity instead.

2 Rollback Issues

When designing a Workflow that employs a rollback, design the transition activities to prevent reset tasks and approvals from being orphaned by alternate execution paths or to ensure that the workflow does not fail from a broken execution path. These examples show how to resolve two common rollback issues: a workflow that fails when a Rollback To occurs, and another that finishes with pending tasks and approval requests left in an unresolved state.

2.1 Example 1: Rollback Fails

In this example, an If activity decides whether to create Tasks 1 and 2 and Approval 1 or create only Task 3 and continue to the end. In either case, the workflow can execute properly, but only if the rollback does not occur. If an approver rejects the request, the rollback attempts to connect to Task 3, which does not exist in this execution path. The rollback cannot connect to Task 3 (or create a task) and cannot reset the states of Tasks 1 and 2 and Approval 1. The workflow ultimately fails.


2.1.1 Redirect the Rollback

One solution to the problems with this workflow is to redirect the Rollback To activity to the If activity. This resets the approval and the tasks already created and allows reevaluation of the If activity. From this point, the workflow can execute successfully down either path.


2.1.2 Remove the Rollback

If you do not need to reset the approval and tasks when the request is rejected, remove the Rollback To activity and connect the approval directly to Task 3 using a Go To connector. When connected in this way, a rejected approval resets Task 3, if it already exists, or creates Task 3 if it does not exist, and then continues to the end.


2.2 Example 2: Unresolved Tasks and Approvals

In this example, the rollback resets Tasks 1 & 2 and Approval 1 and begins processing again at Task 1. However, if execution takes a different path at the If activity, the workflow can run to the end successfully leaving Task 2 with a state of Pending and Approval 1 at Not Yet Requested. The only way to resolve the states of the orphaned activities in this workflow is to deal with each of them manually. This issue can occur with either the Rollback or Rollback To activity.


2.2.1 Testing with a Script

To avoid performing manual resolution of tasks after the workflow has ended, insert a script between the If and Log Message activities to test the states of tasks and approvals in the workflow. If the script determines that a task or an approval was left in an undesirable state, the script can take appropriate action, such as to close Pending or Open tasks and to revert an approval to Rejected.
