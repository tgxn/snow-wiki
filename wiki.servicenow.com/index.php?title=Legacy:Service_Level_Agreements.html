1 Overview

Service Level Agreements (SLAs) are agreements between a Service Provider and customers to measure a certain level of service. The system provides a mechanism for defining SLAs and managing them using escalations. Escalations are indications that a task in the system has gone for a period of time without being closed. Escalations can be defined for Incident, Problem, and Change management.

The SLA is not a method of tracking inactivity for a ticket. To track inactivity for a ticket, use an Inactivity Monitor.

Modifying an SLA does not update existing escalations that are already scheduled to fire. Only new escalations set going forward from that point will reflect the updated SLA. One way around this is to update the relevant task records, so the updated SLA will be applied. This will work if the System Properties are set to recompute escalations at every save/update (which is the system default).

Once the record becomes inactive, the Made SLA field becomes set. The duration is calculated from the date and time field on the record defined by the sla_basis field. By default, the record's date and time basis is set to opened_at. For tables which do not have the Opened At field, the SLA basis must be changed. The calculated duration is compared to the summed escalation intervals to determine with the SLA was made or not.

2 Defining an SLA

To define an SLA, from the left navigation pane, select System Policy > SLA Management and click New and populate the form. The following fields are crucial to defining the SLA:

  • The Table field defines what type of ticket will be monitored.
  • The Evaluation Order value is used if more than one SLA apply to the same ticket. In such a case, SLAs with lower evaluation order values will take precedence over SLAs with higher evaluation values.
  • The Conditions fields narrow the type of ticket will be monitored. If conditions are specified, the SLA will apply only to tickets that meet the conditions, but if no conditions are specified the SLA will apply to all events from the table.
  • The Pause Conditions field defines SLA will be paused and will not escalate. As soon as the ticket is updated and no longer meets the pause conditions, the SLA will unpause and resume counting where it left off.
  • The sla_basis field defines what date and time field the SLA uses to base its counter. By default, the SLA basis will be set to the Opened At field. Some tables do not use the Opened At field, so adjust the sla_basis field to match the appropriate date and time field. For instance, a Change Request form does not have Opened At by default, so use the Work Started field as a basis for the SLA instead.

An escalation is defined using a level and a duration of time; if the time has transpired without the task moving to a closed state, the escalation level is raised. When the escalation level is raised, actions such as notifications or additional assignments can occur.

4 Defining an SLA for a New Table

In order to define an SLA for a new table, the new table must extend the base table Task. Once the new table is created, some alterations to the form for a new table must be made to accommodate the new table. This requires selecting a basis/start point for your SLA. The basis for an SLA within incident management is the Opened field which means that the SLA will start its calculation based on the date and time the incident was opened.

To define an SLA for a new table:

  1. Create a new date/time field and place it on the form for the new table
  2. Right click and select Personalize Dictionary on your new field
  3. Copy and paste the following into the Default Value:
javascript:gs.nowDateTime()

    4.  Copy, replacing yourtablenamehere with your new table name, and paste the following into the Attributes: 

sla_basis=yourtablenamehere,timeDimension

        If you intend to use the Opened field as is used on incident, then paste the following into the Attributes instead: 

sla_basis=incident;yournewtablename,timeDimension=true

    5.  Save.
