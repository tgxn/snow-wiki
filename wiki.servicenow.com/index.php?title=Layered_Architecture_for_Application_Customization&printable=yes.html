Layered Architecture for Application Customization

1 Overview

Layered application architecture allows customers to customize ServiceNow applications and still inherit new features from future ServiceNow upgrades.

Layered application architecture is available starting with the Dublin release and is implemented in the HR Service Management application.

For non-layered application architecture, if a customer changes application files, then upgrades that application to receive a new feature, the upgrade may not provide the expected feature, because an upgrade does not overwrite customer-modified files. So while all customer-specific changes are retained, the full upgrade features may not be available.

A layered approach avoids these issues.

2 The Layered Approach

This architecture is based upon these layers:

  • The ServiceNow layer: Should not be changed by customers. Future code changes to the application made by ServiceNow will be made in this layer.
  • The customer layer: Can be changed by customers. Allows customers to either inherit the default functionality or define custom functionality appropriate to their organizations.

2.1 The ServiceNow Layer

The ServiceNow layer contains a factory class (<application prefix>_BaseFactory), which returns one or more wrapper classes.

A wrapper class contains the Business Logic associated to a table in the application and wraps a GlideRecord object. The various parts of the application that need business logic, such as business rules, make use of these wrappers.

Typically an application uses one wrapper per table and contains methods that only act on a single record. If logic is needed that applies to the whole table but not a specific record, such as returning a count of all active HR cases, ServiceNow adds a class method to the wrapper.

2.2 The Customer Layer

The customer layer also contains a factory class (<application prefix>_Factory). By default, this factory provides the wrappers registered in the ServiceNow layer.

To customise the application, a customer should define a new wrapper class, then register the wrapper, against a table name, in the customer factory. The factory will now return the customised class rather than deferring to the ServiceNow factory.

Note: The recommended labeling for wrapper classes is <customer prefix>_<application prefix>_<classname>.


3 Example: Human Resources

The Human Resources application adds events in the ECC Queue when an HR case record is created or updated.

The code that does this is stored in the Send hr_case events business rule:

If customers want to customize this application, for example to dispatch additional events when HR cases are created or updated, they can do this without interfering with future plugin upgrades.

First, implement a wrapper with the new behavior. 

var acme_hr_Case = Class.create ();
 
acme_hr_Case.prototype = Object.extendsObject (hr_Case, {
    sendEvents: function () {
        //Call the original sendEvents method if needed
        hr_Case.prototype.sendEvents.call(this);

        // revised logic should go in here
    },
    type: "acme_hr_Case"
});
Note: If the ServiceNow layer wrapper has class methods defined you will need to reference these in the customised class. This can be done with a simple assignment for each function. E.g. acme_hr_Case.aFunction = hr_case.aFunction;


Next, register the new class against the table name so the factory to returns the new wrapper. 

hr_Factory.prototype = Object.extendsObject(hr_BaseFactory,{
...
});


hr_Factory.getWrapperType = function(tableName) {
    switch (tableName + "") {
        case hr.TABLE_CASE:
            return acme_hr_Case // name of the new script include that extends hr_case
        default:
            return hr_BaseFactory.getWrapperType(tableName);
};
