1 Overview

Web services allow diverse applications to talk to each other. ServiceNow supports both inbound (provider) and outbound web service consumer services. The web services provided are WS-I compliant outlined in the WS-I Basic Profile 1.0.

2 Concepts and Terminology

  • Provider - publishes web service for clients to invoke (consume)
  • Consumer - invokes / consumes published web service
  • Standards
    • WSDL
      • Web Service Description Language
      • XML document describing functions, arguments, data schema, and endpoint (where / how to invoke the service, URL)
      • WSDL only necessary when generating SOAP envelope programmatically
    • SOAP
      • Simple Object Access Protocol
      • XML document usually HTTP posted to web service endpoint described in WSDL
      • SOAP:Envelope / SOAP:Header / SOAP:Body
    • HTTP
      • Hyper-Text Transfer Protocol
      • POST vs GET - Web Service is POSTed
      • XML vs. Form POST - Web Service is XML

3 Web Service Provider

As a web service provider, the ServiceNow platform supports publishing access (input and query) to its underlying table structures and their associated data via direct web services. A less direct, input only mechanism is available via web service import sets which invokes data mapping using transform maps. Lastly, scripted web services is available for developing custom web services that are executed in Javascript.

3.1 Direct Web Services

A direct web service is available for any table in the system provided the correct access control is setup. The supported format of the incoming message is document style literal XML SOAP documents (Document/Literal). To retrieve the direct web service WSDL description and XML schema, point to the relative URL of <tablename>.do?WSDL. For example, to retrieve the WSDL for the Incident table, use the following URL:

https://<instance name>.service-now.com/incident.do?WSDL

3.2 Web Service Import Sets

Web Service Import Sets complement Direct Web Services and Scripted Web Services to provide a web service interface to Import Sets tables. This type of web service will transform the incoming data synchronously based on the associated Transform Maps by default.

3.3 Security

Web service security is enforced using the combination of basic authentication challenge/response for the HTTP protocol, as well as system level access control using the Contextual Security Manager. Additionally, there is a set of web service specific roles that may be granted to the web service user.

3.3.1 Basic Authentication

To enforce basic authentication on each request for a WSDL document or posting of SOAP messages, you may set the property glide.basicauth.required to true. If you do so, each WSDL or SOAP request would have to contain the "Authorization" header as specified in the Basic Authentication protocol. Because the request is non-interactive, we always require the Authorization header during a request.

Supplying basic authentication information whether or not it is required has the added advantage that the data created or updated as a result of the Web Service invocation is done on behalf of the user supplied in the basic authentication credentials. As an example, when creating an Incident record, the journal fields will have the user id of basic authenticated user, instead of the default "Guest" user.

3.3.2 Web Service User Roles

When basic authentication is required for web services, enabling the glide.soap.strict_security property validates the authenticated SOAP user with Contextual Security Manager as well as vaidates the SOAP action with the user assigned roles. The following roles are available for securing any inbound SOAP requests:


Role Description
soap Full query, create, update, and delete access to all tables, as well as execute scripts
soap_query Query only access to all tables and columns
soap_create New record creation access to all tables and columns
soap_update Update only access to existing tables and columns
soap_query_update Query and update access to all tables and columns
soap_delete Record deletion access to existing records in all tables and columns
soap_ecc Query, create and update access to the ECC Queue table only
soap_script Access to execute scripted web services

3.3.3 Enforcing Strict Security

By default, the user ID used in the basic authentication header for web services is an ID used for authentication only, the user roles are not used for access authorization. To enforce user role access, check the Enforce strict security property in System Properties -> Web Services.


3.4 Overriding the SOAP Endpoint

 Release: Functionality described here is available with the Winter 2009 release.

Since the Winter 2009 release, the SOAP End-point address (where the SOAP message is posted) is generated based on the WSDL. In some cases, however, this may generate an incorrect URL. If this happens, it is necessary to over-ride the generated URL by changing the property com.glide.soap_address_base_url. The value of this property will change the base URL.

For instance, for the SOAP Endpoint: 

https://instance.service-now.com/incident.do?SOAP

The property will override: 

https://instance.service-now.com/

Note that the ending slash is part of the override.

4 Web Service Consumer

Consuming external web services is achieved using Javascript objects that represent the web service SOAP envelope and the subsequent SOAP HTTP request that submits the request. The Web Service Consumer Plugin article documents these programmatic constructs as well as examples of how to invoke web services.
