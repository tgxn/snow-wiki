3 General

  • Legacy Certification app renamed to Record Certification:
Applied to All Instances
Since the legacy Certification application only allowed for certification of records (vs. elements of records), it has been renamed to Record Certification.
  • assigned_to reference qualifier behaves differently when assignment_group is populated:
Applied to All Instances
In List v1, if the assignment_group field had a value, the reference qualifier on assigned_to (roles=itil) was deliberately ignored. More generally, the reference qualifier was ignored for any sys_user reference dependent on another field that had group in its name, if that field was populated.
With List v2, this unexpected behavior has been removed, and reference qualifiers are now always honored. A symptom might be that an incident cannot be assigned to a member of the Assignment group if the member doesn’t have the itil role.
There are three options in this case:
  • Give the user the ITIL role
  • Remove the reference qualifier from the assigned_to field
  • Change the reference qualifier to an advanced ref qual that emulates the old behavior. e.g., roles=itil can be changed to the following: 
javascript:if (current.assignment_group.nil()) “roles=itil”;

4 Incident Management

  • Incident Overview homepage reworked for new instances:
Applied to New Instances
The Incident Overview homepage has been modified for consistency and relevancy for new instances. Existing instances will not see a change in behavior.
  • Email client activated by default:
Applied to All Instances
The email_client=true attribute has been added to the Incident table's dictionary record, activating the email client by default for all instances. It can be deactivated by setting the attribute to false.
  • Incident state changes to Active when assigned for new instances:
Applied to New Instances
An Incident in the New state will be changed to the Active state when assigned. This happens with the "Incident State Active on Assignment" business rule for new instances. Existing instances will not see a change in behavior.
  • Reopen count integer field added for new instances:
Applied to New Instances
A new integer field, Reopen count, has been added to the Incident table for new instances. The "Reopen Count" business rule increments this value when the Incident's state is changed from Closed or Resolved to another state. Existing instances will not see a change in behavior.
  • Parent Incident reference field added for new instances:
Applied to New Instances
A new reference field, Parent Incident, has been added to the Incident table. It can be used to group Incidents. Additional comments and Work notes to a parent incident are copied to its unresolved child incidents. Resolving a parent incident automatically resolves its unresolved children. Close notes and Close code is copied from a parent incident to its children if these fields are empty on the child. Existing instances will not see a change in behavior.
  • Child Incidents integer field added for new instances:
Applied to New Instances
A new integer field, Child Incidents, has been added to the Incident table for new instances. When the Parent Incident field value changes for an incident, a business rule automatically updates the Child Incidents value. Existing instances will not see a change in behavior.
  • VIP indicator for Caller field in lists for the Best Practice - Incident Resolution Workflow plugin:
Applied to All Instances
The field style on incident.caller_id has been modified in the Best Practice - Incident Resolution Workflow plugin to add a small VIP indicator to the Caller field in a list if the referenced user is a VIP. This indicator is consistent with the one already added to the Caller field label on the form by the Highlight VIP Caller client script. Customers with the Best Practice - Incident Resolution Workflow plugin activated who have not customized this field style will see this change.
  • Knowledge article Source field populated:
Applied to All Instances
The Incident Create Knowledge and Problem Create Knowledge business rules have been modified to populate the Source reference field of a new article when it is automatically created from an Incident or Problem. Customers who have not customized these rules will see this change.
  • Self Service users can resolve/close their own Incidents for the Best Practice - Incident Resolution Workflow plugin:
Applied to All Instances
The Close Incident and Resolve Incident UI Actions have been modified in the Best Practice - Incident Resolution Workflow plugin to allow Self Service users to resolve and close Incidents where they are the Caller. Customers with the Best Practice - Incident Resolution Workflow plugin activated who have not customized these UI Actions will see this change.
  • Incident form reworked for new instances:
Applied to New Instances
The Incident form has been modified for new instances. It now has multiple sections (main, Notes, Related Records, and Closure Information), and fields have been added/removed for relevance and consistency. Existing instances will not see a change in behavior.
  • Email Client active for Incidents:
Applied to All Instances
The email_client=true attribute has been added to the Incident table, enabling the email client for Incidents. Customers who do not wish to use the Email Client should set the attribute to false.

12 Scripting

  • Script Logs module added:
Applied to All Instances
The Script Logs module (System Logs > Script Logs) was added, showing the list of log entries created by gs.log(message) statements in scripts.
  • enableOption and disableOption functions removed:
Applied to All Instances
The enableOption and disableOption functions in g_form were not cross-browser compatible, so they have been removed. Their use may cause a JavaScript error.
  • Read-only multi-line fields can use g_form.setValue():
Applied to All Instances
Read-only multi-line fields can now use g_form.setValue().
  • Disabled scripts' ability to access java.* and getClass:
Applied to All Instances
Disabled scripts' ability to access java.* and getClass directly. Must use Packages call instead.
  • GlideRecord.getPlural() method added:
Applied to All Instances
Similar to getLabel(), which returns the label of a GlideRecord's table, getPlural() will return the plural.
  • jQuery dependencies and library removed:
Applied to All Instances
All jQuery dependencies have been removed as well as the jQuery library. Instead, a custom Prototype Effects library has been created and included for all pages. This significantly improves page load time and JavaScript parse time. If for any reason an admin must use jQuery for any non-effects based classes/methods, inject the library manually by using the <g:requires src='jquery.min' /> tag where appropriate. The jQuery namespace can then be referenced in its entirety using the "jQuery" or "$j" globals.
  • Transform Script to deactivate LDAP users for new instances:
Applied to All Instances
An inactive onBefore Transform Script has been added to the LDAP User Import transform map, which can be used to deactivate users imported from LDAP when appropriate. Existing instances will not see a change in behavior.
  • Group import transform script removed for new instances:
Applied to New Instances
An inactive onAfter transform script that imported groups from LDAP based on group membership has been removed from the LDAP User Import transform map. Almost without exception, it resulted in numerous undesired groups being created. Existing instances will not see a change in behavior.
  • g_form.enableAttachments():
Applied to All Instances
g_form.disableAttachments() has been available to disable attachments on a form. Now g_form.enableAttachments() will do the reverse.
  • Logging with gs.log(message, source):
Applied to All Instances
gs.log(msg) had been enhanced to support gs.log(msg, source) that will set the Source field in the resultant syslog record to the specified value. This gives customers the ability to log from within their scripts and easily filter by the source provided.
  • jQuery UI Script:
Applied to All Instances
jQuery has been removed from the included scripts, and added as its own UI Script that can be included in Jelly scripts as required.

Applied to All Instances
Specific roles are required for certain access levels. This syntax displays Table name > Access type (R=read, W=write, D=delete, C=create) > Required roles.
  • sys_user_preference > users can RWD their own preferences

  • Role-based ACLs for import set tables:
Applied to All Instances
Specific roles are required for certain access levels. This syntax displays Table name > Access type (R=read, W=write, D=delete, C=create) > Required roles. These ACLs are inherited by the platform-generated, temporary staging tables, which are intended to hold imported data until it is transformed into other tables in the system. Staging tables are given a u_ prefix, and are only accessible to users with the import_admin and import_transformer roles. These ACLs protect the staging tables and control access to records imported with an Import Set until the data is moved to destination tables that have their own ACLs.
sys_import_set_run/sys_import_set_row/sys_import_set_row_error/sys_transform_map/sys_transform_entry/sys_transform_script/sys_progress_worker > RWCD > admin/import_admin/import_transformer
  • Write access to five task fields restricted to admin for new instances:
Applied to New Instances
When a new instance is created, ACLs are automatically created restricting write access to admin for the following task fields: Number, Opened, Opened by, Closed, Closed by. Existing instances will not see a change in behavior.
  • Only admins can delete tasks for new instances:
Applied to New Instances
A row-level Delete ACL has been added to the task table requiring the admin role to delete task records. Existing instances will not see a change in behavior.
  • High Security Plugin ACLs:
Applied to All Instances
When the High Security plugin is installed, admin access to ACLs on the sys_security_acl table is removed. security_admin role is required instead.
  • Security ACL Detail database view:
Applied to All Instances
A new database view has been created, "Security ACL Detail", joining sys_security_acl and sys_security_acl_role for ease of reporting.
  • ACLs for Project tables:
Applied to All Instances
New ACLs have been added to Project tables as follows:
  • pm_project (write) Project Manager and Project User roles
  • pm_project (delete) Project Manager role
  • pm_project_task (write) Project Managers and Project User roles
  • pm_project_task (delete) Project Managers role
  • Revised ACLs and UI Actions for sys_email:
Applied to All Instances
Revised ACLs for sys_email (requires Security Jump Start plugin). Admin is required to write/delete (that this does not prevent use of the email client by non-admins). Updated three sys_email UI Actions to require admin (resend, reprocess email).
  • Revised ACLs for sys_attachment:
Applied to All Instances
  • A new Write ACL for sys_attachment.* has been added, returning false. There is no need to edit fields on sys_attachment records.
  • Read ACL for sys_attachment modified. If not attached to a GlideRecord, everyone can read. If attached to a GlideRecord, users who can read that record can read attachments.
  • Delete ACL for sys_attachment modified. Users can delete attachments they created. Otherwise, users can delete attachments from valid GlideRecords they can write to.
  • Write ACL for sys_attachment reformatted to match better logic of other ACLs.
  • ACLs added for sys_attachment_doc
  • ACLs to protect any page:
Applied to All Instances
An ACL can protect any page served by ServiceNow (lists, forms, UI Pages, hardcoded pages, etc.). When going after a page, a check is made to see if there is are ACLs of type "ui_page" for that page. If the ACL denies access, a message is displayed instead of the page.
  • Use case: prevent access to report_home.do, which is not a UI Page and thus could not be secured previously.
  • Use case: since no one should go to sys_attachment_list.do, could create an ACL of type ui_page where name = sys_attachment_list, and script is "false;". Now the system doesn't have to evaluate ACLs for 100 rows when a user tries to go to the list, a single ACL check denies access.
  • Use case: only admins should even go to sys_properties_list.do. So could create an ACL of type ui_page where name = sys_properties_list, and Required Role is admin. Now one ACL can prevent list access vs. evaluating against 100 rows at a time.

  • Dictionary related lists:
Applied to All Instances
Two Relationship related lists have been added to the Dictionary form, Choices and Labels. They show any choice list options for a field and any labels for a field or table. Lists are omitted if empty. New button is omitted. Customers who have customized their Dictionary form in any way may not see this change.
  • Client Transaction Timings plugin active for new instances:
Applied to New Instances
New instances have the Client Transaction Timings plugin activated by default.
  • Austere and Visionary CMS theme plugins no longer available:
Applied to All Instances
The legacy Austere and Visionary CMS theme plugins have been removed from the codebase and are no longer available for installation. Customers who have already installed these plugins will not notice any change.
  • Field styles apply to multi-line fields in forms:
Applied to All Instances
Field styles are now applied to multi-line fields in forms, consistent with single line fields.
  • Windows Mobile and Windows Phone supported versions:
Applied to All Instances
Added support for Windows Mobile 6.5 and Windows Phone 7
  • Variable types hidden for Survey questions:
Applied to All Instances
A new client script hides Container Start and Container End variable types for the survey_question_new form, since those types aren't supported for surveys.
  • glide_list display change in lists:
Applied to All Instances
When displaying a glide_list field in a list, a space has been added between a separating comma and the next display value. e.g., instead of "Beth Anglin,Don Goodliffe,David Loo", display "Beth Anglin, Don Goodliffe, David Loo".
  • List cell title attribute (tooltip) change:
Applied to All Instances
If the entire content of a field is displayed in a list cell, it is no longer duplicated in a tooltip of the cell. The tooltip is only included if it is different than what is already displayed in the cell (i.e., if it's been truncated).
  • List Edit for Workflow fields:
Applied to All Instances
Fields of type "workflow" can now be list edited.
  • Index for document_id fields:
Applied to All Instances
Fields of type "document_id" now get a database index by default, just like reference fields.
  • No sorting for glide_list:
Applied to All Instances
Fields of type "glide_list" can no longer be sorted, since sorting has never been supported for that field type and led to unexpected results.
  • Removed support for Choice type 4 (pick list external):
Applied to All Instances
With List v2, support has been removed for Dictionary choice type '4' (pick list external).
  • List v1 is no longer available:
Applied to All Instances
The boats have been burned. Long live List v2!
  • Impersonator role:
Applied to All Instances
New "impersonator" role. A user with this role can impersonate other users. If the impersonator does not have the admin role, they are not allowed to impersonate someone who does. The impersonation dialog's reference field filters out admin users if the user doesn't have admin.
  • Friendlier NULL:
Applied to All Instances
In a grouped list, if the group value is "NULL", "(empty)" is shown instead. NULL can be scary.
  • Display field magic:
Applied to All Instances
If no display field is specified for a table, and a "name" field doesn't exist, now the "u_name" will be used as the display field if it exists before we punt and use sys_id.
  • List DOM element and attribute changes:
Applied to All Instances
Significant changes have been made to how lists are built in HTML and JavaScript. Scripts that inadvisedly rely on specific DOM elements or attributes of a list may be thwarted with this release.
  • Insert and Insert and Stay UI Actions generate new number:
Applied to All Instances
Use of the Insert or 'Insert and Stay' UI Action will blank out the number field, ensuring a new number is generated for the new record.
  • Wikitext field attributedual_mode:
Applied to All Instances
Wikitext field optional attribute "dual_mode=true" displays both editor and preview simultaneous when in edit mode. Refresh preview with refresh icon, similar to MediaWiki preview.
  • Many-to-many autocomplete behavior:
Applied to All Instances
Autocomplete in a many-to-many slushbucket now honors Customized Autocomplete Search Behavior user preferences.
  • Impersonation Events:
Applied to All Instances
Two new events, "impersonation.start" and "impersonation.end", are triggered as a result of an admin impersonating a user. They can be reacted to as desired with an Email Notification or Script Action.
  • Field Normalization changes:
Applied to All Instances
Made insert/update interception for field normalization independent of the GlideRecord engine mechanism, so that it is possible for field normalization to work even when Workflow is turned off. Also created two new GlideRecord API elements: setFieldNormalization(boolean) and isFieldNormalization() that can be used with insert, update, and query operations.
  • Global Text Search Form View:
Applied to All Instances
A new property, glide.ui.text_search.match_view, has been added to specify the form view used for a Global Text Search exact match. Default is the "Default" form view.
  • Numeric (Likert) Variable Max Scale:
Applied to All Instances
The max scale on numeric (Likert) variables has been limited to 1000.
  • Active field for Public pages:
Applied to All Instances
A new field, sys_public.active, has been added to public pages. Allows deactivating of a public page's sys_public record without having to delete it.
  • Disabled List Editing indicator:
Applied to All Instances
If a user has disabled List Editing for their user record, and subsequently tries to double-click in a field to trigger list editing, they'll see a small indicator telling them what has happened.
  • List Mechanic gear removed from Global Text Search:
Applied to All Instances
The list mechanic (personalize list) gear has been removed from Global Text Search results lists.
  • Reference Styles:
Applied to All Instances
A new table, sys_ui_reference_style, can be used to define a style for all references to a table. Two fields, name (table) and style. e.g., creating a Reference Style of width:240px; for the cmdb_ci table will have that width apply to all reference fields to cmdb_ci, with no need to create (at last count in /demo) 66 individual Field Styles to accomplish the same thing. Reference Styles can be overridden by Field Styles, since they are sequential in the style attribute of an element. A Reference Style on cmdb_ci applies to all extension tables unless a lower table has its own Reference Style. e.g., for a reference to cmdb_ci_server, first the system looks for a Reference Style on that table, then on cmdb_ci_computer if none exists, then on cmdb_ci.

18 Task Management

  • Simultaneous update alert for new instances:
Applied to New Instances
A new client script "Simultaneous Update Alert" has been added to the task table for new instances. It alerts a user, when submitting a form, if the record has been updated since it was opened in their browser. The user is given the opportunity to cancel or proceed with the form submission. Existing instances will not see a change in behavior.
  • User reference autocomplete uses contains query for new instances:
Applied to New Instances
A new user preference named "sys_user.autocomplete.contains" has been added to new instances setting the autocomplete query to a "contains" query (vs. a "starts with" query) for reference fields against the User table. e.g., typing "Good" will return "Don Goodliffe" in the autocomplete matches. Existing instances will not see a change in behavior.
  • Reassignment count field and business rule moved from Incident to Task:
The reassignment_count field is promoted to the task table from the incident table. All instances will see this change. The "reassignment count" Business Rule has had the Table value changed from task to incident. Customers who have not customized this rule will see this change.
  • Field Styles for Additional comments and Work notes journal fields for new instances:
Applied to New Instances
To distinguish them from each other, Field Styles have been added to the Additional comments and Work notes journal fields for new instances, to color the inputs the same as they are colored by default in the activity formatter. Existing instances will not see a change in behavior.
  • Additional comments field label change for ITIL users for new instances:
Applied to New Instances
Users with the itil role will see "(Customer visible)" appended to the Additional comments field label. This is done with the "Modify Comments Label" client script for new instances. Existing instances will not see a change in behavior.
  • New button removed from task form related lists for new instances:
Applied to New Instances
Where it didn't make sense to have them, the New and Edit buttons have been removed from related lists for various task forms for new instances. This is done using List Control and can be customized as desired. Existing instances will not see a change in behavior.
  • closed and closed_at field populated for all tasks:
Applied to All Instances
Added a high order Business Rule to the task table, "Set Closure Fields", that sets closed and closed_at fields when active changes to false if not already populated.
  • Calculated Planned Task field decoration:
Applied to All Instances
Planned task fields that are automatically calculated from values in child records are identified with a field decoration:
  • Real time planned start calculation:
Applied to All Instances
Update to planned task and project date logic. Primary change is to always calculate Planned end and successor Planned start dates from any actuals (work_start, work_end) that have values. This makes for a real time project instead of calculating everything on original planned values even when actuals exist and are different. Also display actuals instead of planned in project timelines.
  • Certification Task group assignment:
Applied to All Instances
Certification Tasks can now be assigned to groups and users automatically. Requires the Certification plugin.
  • Parent Breadcrumbs Formatter:
Applied to All Instances
A new Task Parent Breadcrumbs Formatter is available to show the parents of a record in breadcrumb format on the form. This is most useful in Project Management and Release Management, where tasks exist in multi-level hierarchies.
  • New planned_task.rollup field:
Applied to All Instances
The new planned_task.rollup field is used to identify a task with children, more efficient than frequent queries for children. Locked down more fields on rollup tasks since they are calculated from children (work_cost, effort, rollup, percent complete, remaining duration, remaining effort, actual_duration, work_effort, work_end, work_start).

19 Web Services

  • Scripted Web Services:
Applied to All Instances
The Web Services Provider - Scripted plugin is now installed automatically on all instances.
  • ODBC driver performance improvements:
Applied to All Instances
ODBC driver performance has been improved, preventing out-of-memory errors when a query produces an extremely large result set.
  • SOAP response formatting changes:
Applied to All Instances
The SOAP response now returns with the envelope and body tags in upper case, and the header tag omitted.

20 Workflow/Runbook

  • Workflow-specific Workflow Stages cannot be added from the Stages reference icon list view:
Applied to All Instances
To add new stage, navigate to the Graphical Workflow Editor and select the Workflow Actions menu and select Edit Stages.
  • Workflow field type filter operators:
Applied to All Instances
Workflow fields now get the same filter operators as choice fields.
  • Workflow properties module:
Applied to All Instances
A Properties module has been added to the Workflow application for easy access to relevant properties.
  • Approvals in Workflows for Non-Task Tables:
Applied to All Instances
The Approval - User and Approval Action workflow activities are now available for non-task tables.
  • Transform Map runs with Workflow:
Applied to All Instances
For additional comments to get updated in a transform map, Workflow must run to generate audit records for the journal formatter to pick it up. Workflow is now turned on in the transform map, and isInteractive() is used to prevent loops instead.
