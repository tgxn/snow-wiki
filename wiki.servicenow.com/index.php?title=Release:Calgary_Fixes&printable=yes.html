1 Overview

The initial Calgary release includes fixes for several known issues. This release is no longer recommended for upgrades. If you want to upgrade an instance, consider a current version. For more information about how to upgrade your instance, see Requesting an Upgrade.

For the initial Calgary release:

Build date: 02-28-2013_0948
Build tag: glide-calgary-02-15-2013
 Note: See how to determine the version on your instance.

PRB573235 The Discovery identify sensor was producing an error when attempting to discover Mac OS X devices. A script include could not parse the data returned from the device, because it was not valid XML.
Click the plus for steps to reproduce
  1. On a Mac OS computer, create a user called "discovery.user".
  2. Open a terminal on that computer and enter "sudo mv /Users/discovery.user /Users/discovery.user_".
  3. Run a discovery against this computer, using credentials for the user you created.
    The sudo command causes discovery to run for a user who does not have a proper home directory. If the result from the command "system_profiler SPSoftwareDataType -xml" contains the following message, the sensor produces an error:
    CFPreferences: user home directory at file://localhost/Users/discovery.user/ is unavailable. User domains will be volatile.

PRB576127 The "Virtualized by" relationship between a VM guest and the ESX host was not reconciled properly by Discovery after the VM guest was migrated between ESX hosts. The result was that two "Virtualized by" relationships were registered for the VM guest.
View the steps to reproduce

  2. Within the vSphere Client, shutdown the VM.
  3. Migrate the VM to another host.

    2. Select Change host.
    3. Select a different host than the current host.
    4. Use the Summary tab to verify the current configuration.
  4. Run the discovery again.
  5. Verify that two Virtualized By relationships now exist on the VM guest's CI (not the VM instance CI).
PRB573019 The starting time could not be set for a Discovery Schedule configured to run Once or Periodic, because the "run_start" (Starting) field was not visible on the form. To use the field, a user had to personalize the form. The "run_start" field now appears on the form in the base ServiceNow system.
View the steps to reproduce
  1. Navigate to Discovery > Discovery Schedules.
  2. Create a new schedule or open an existing schedule.
  3. Set the "run_type" field to "Once" or "Periodic."
    The "run_start" field isn't shown, so the starting time cannot be set.
PRB570516 The process of cloning one instance over another replaced the MID Server records in the MID Server [ecc_agent] table on the overwritten instance. Although the MID Servers were able to authenticate on the overwritten instance, they could not register with the ECC Queue because their records were missing. A cloned instance can now recreate records in the MID Server [ecc_agent] table for any MID Server that is able to access the instance.
Click the plus for steps to reproduce
  1. Install two MID Servers.
  2. Configure one to point to Instance A and one to point to Instance B.
  3. Make sure the MID Server user name and password are the same for both instances.
  4. Clone Instance A over Instance B.
  5. View the MID Server list in Instance B.
    Note that the MID Server record is from the clone source instance (Instance A) and not from Instance B. Look at the ecc_queue and MID Server logs. The Instance B MID Server is still running and sending payloads to the instance.
PRB570645 The Discover Now process did not check the value in the Status field in the network adapter record. If an IP address was no longer being used on an adapter (status = Absent), and had been assigned to an adapter on a different CI, then Discovery found both CIs at that IP address.
Click the plus for steps to reproduce
  1. Find a configuration item (CI) with multiple network adapter records, some with status = Absent.
  2. Click the Discover Now UI action in Related Links.
  3. Navigate to Discovery > Status and view the IP address you are trying to discover.
    Note that iIn some cases, Discover Now finds a CI with an IP address that is listed as Absent in the network adapter record for that CI.

The relationship to the original zone persisted in the CMDB after the move. If the server was moved multiple times, each of the previous relationships were retained, until a user could no longer determine the correct zone from the record.

PRB577005 Discovery for NetApp devices returned device IDs for the array, pools, volumes, and disks instead of a human-readable label. Also, Discovery did not return the following NetApp device information:
  • Serial number

Run a discovery against NetApp devices on SNIA (ONTap in the name). Observe that the names used for the server, pools, and volumes are device IDs, not the assigned human readable labels. Also observe that the Make and Model are not filled in for NetApp after a discovery.
PRB576055 Discoveries using external credential storage could not discover SNMP devices using credentials stored on the external repository.
PRB577606 The VMware - vCenter probe could not connect to vCenter. The cause was a trailing space in the user name of the vCenter credentials. The platform now displays a warning when it detects a leading or trailing space in the User name field in the Credentials [discovery_credentials] table.
PRB576995 Discovery could not produce an accurate CPU core count on Solaris machines with multiple cores.
PRB577190 Discovery could not produce an accurate CPU core count on Linux machines with hyperthreading enbabled.
PRB576957 When the platform ran the Help the Help Desk login script, it did so with the user's own credentials (cookie-based authentication) rather than the configured Basic Authentication. Because the ecc_soap role was not present on the user's account, Help the Help Desk could not update the CMDB.
PRB575807 The TCP Connection [cmdb_tcp_connection] table was storing unused connections and could grow to enormous size. The Running Process [cmdb_running_process] table was storing unused processes and also became unnecessarily large. The size of these tables affected Discovery performance, which improved when the tables were cleaned. These tables were added to the Table Cleaner with a default interval of 30 days.
PRB571515 Discovery created duplicate configuration items (CI) for Websphere and vCenter servers during process classification, due to the way correlation IDs were generated. Process classification relied on a parameter whose value changed whenever the Websphere server was restarted. A new module called Process Handler was added to the platform to filter out these parameters prior to classification. For more information, see the link for Process Handler in the New in This Release table.
PRB578911 Storage configuration items (CI) were registered in the CMDB without location information, although the management server was correctly registered under the proper location.
PRB573400 The following message, warning that a cluster resource could not be found, was logged many times during Discovery: Warning: Unable to connect to path: root\MSCluster. Error: Invalid namespace. This message, generated by the WMI engine, was issued when a server did not have cluster resources installed.
