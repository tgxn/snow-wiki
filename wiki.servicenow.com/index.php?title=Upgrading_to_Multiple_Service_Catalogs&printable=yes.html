Upgrading to Multiple Service Catalogs

1 Overview

Administrators and catalog administrators can manage multiple service catalogs, starting with the Eureka release.

Before upgrading, you should be aware of changes made to the underlying service catalog data model. These changes are made to implement multiple catalogs and should not affect a standard upgrade. However, the changes may impact the instance if you have made customizations, such as changes to the data model.

2 Tables

The following tables are provided with multiple service catalogs:

Table [name] Summary
Catalog Site [sc_catalog_site] Associates catalogs to sites.
Catalog Portal Page [sc_catalog_view_mtom] Links catalogs to CMS portal pages. Defines the default catalog to portal page combination used by default links.
Catalog Items Catalog [sc_cat_item_catalog] Stores the catalogs an item is available in.
Catalog Item Category [sc_cat_item_category] Stores the categories an item is available in.

3 Using Catalog Site Records

The Catalog Site [sc_catalog_site] table associates catalogs with CMS web sites. Catalog site records appear in the Sites related list on the Catalog form.

Several service catalog system properties are deprecated and replaced with fields on the catalog site record, starting with the Eureka release. This allows you to specify values for different sites used by different catalogs.

Property Catalog site field [name]
glide.sc.home.cms_page CMS homepage [cms_home]
glide.sc.search.cms_page CMS search page [cms_search]
glide.sc.continue.shopping.target CMS 'Continue Shopping' page [cms_continue_shopping]

When you upgrade, a new default catalog site record is automatically created. It is populated with the default system properties from the previous version, which associate the Employee Self-Service site to the default service catalog. The 'Continue Shopping' page field on the Catalog form is also populated with the previous version's glide.sc.continue.shopping.target property value.

4 Setting Catalog Portal Pages

The Catalog Portal Page [sc_catalog_view_mtom] table:

  • Links catalogs to CMS portal pages.
  • Defines the default catalog to portal page combination used by URLs. For example, the default portal page for the service catalog is the catalog_default page.

Catalog portal page records appear in the Catalog Portal Pages related list on the Catalog form.

The Catalog Portal Page table generates two values in a URL: sysparm_catalog_view and sysparam_catalog. Use the Catalog Portal Page form to enter a custom link, if required.

http://<instance name>.service-now.com/catalog_home.do?v=1&sysparm_catalog=742ce428d7211100f2d224837e61036d&sysparm_catalog_view=catalog_technical_catalog

4.1 Resolving Missing Catalog Portal Pages

If a catalog record is not associated to a catalog portal page record, ServiceNow attempts to present a view based on a set prefix of: catalog_<catalog name>.

For example, if the default service catalog has no catalog portal page record, the system sets sysparm_view to catalog_Service_Catalog. If this happens, when you navigate to Self-Service > Service Catalog, you see the new and apparently blank catalog_Service_Catalog portal page, instead of the existing and populated catalog_default portal page.

To correct this, go to the 'Sites related list on the Catalog form and add a new catalog portal page record manually.

For example, for the default service catalog, add a record with values of:

  • Catalog: Service Catalog
  • Portal Page: Catalog (catalog_default)
  • Default: true

5 Setting Catalogs and Categories for Catalog Items

A new data model is provided for the catalogs and categories associated to an item, as shown in the Catalogs and Categories related lists. This model uses the Catalog Items Catalog [sc_cat_item_catalog] and Catalog Item Category [sc_cat_item_category] tables.

For both catalogs and categories, the related list values are the primary definitions for these relationships. Functionality is driven by these values, not by the values in Catalogs and Category fields on the Catalog Item form.

5.1 Migrating Items to the New Data Model

The fix_multi_catalog.xml script helps ensure a seamless upgrade by automatically migrating records on the Catalog Items [sc_cat_item] table and its child tables to the new data model.

If you import catalog item records into your ServiceNow instance, import the corresponding records for the sc_cat_item_catalog and sc_cat_item_category tables to ensure the imported item records have the correct data model. If you do not have these records for items, you can run this script at any time to populate them.

If you are moving or modifying plugins, you should also put this script into any plugin that loads catalog items.

The script performs the following actions:

  1. Associates all categories and items to the default service catalog if they are not associated to any catalog in the new data model. The sys_id used is that of the default service catalog.
    GlideappCatalogItem.associateToCatalog('e0d08b13c3330100c8b837659bba8fb4');
  2. Populates the Catalog Item Category [sc_cat_item_category] table with the values from the catalog items' Category field. If a catalog item belongs to a category which is associated with a non-default catalog, associates the item to the catalog related to the one the category is in.
    SNC.CatalogItemCategory.populateFromItemCategory();
    Note: To associate a catalog item to multiple categories, you must either create the additional category associations manually, or import them.
  3. Populates the Catalog Items Catalog [sc_cat_item_catalogs] table with the values from the catalog item's Catalogs field.
    SNC.CatalogItemCatalog.populateFromItemCatalogs()
  4. Populates the Continue Shopping fields on both the default service catalog and on the default catalog site record for the default service catalog, with the value of the URL to redirect to upon clicking of the "Continue Shopping" button inside of a cart property (glide.sc.continue.shopping.target).
    SNC.Catalog.get('e0d08b13c3330100c8b837659bba8fb4').populateContinueShoppingPage();
