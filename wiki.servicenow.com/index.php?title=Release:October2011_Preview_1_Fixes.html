Release:October2011 Preview 1 Fixes

From ServiceNow Wiki
Home > Release Notes > Legacy Versions > October2011 Preview 1 > October2011 Preview 1 Fixes
Jump to: navigation, search

The following is a list of issues fixed in the October2011 Preview 1 release. To sort by descending or ascending order, click the sorting icon in the column heading.


Number Category Details Steps to Reproduce
PRB561374 Approvals Approving a task's approval record sometimes did not carry over to the task (introduced with the June 2011 release).
  1. Create a change.
  2. Create or use a change task.
  3. Add an approver to the change task.
  4. Have the approver approve the task.
    Notice the change task does not change to "Approved".
PRB561138 Approvals Approvals sometimes did not work for some tasks on iPhones (introduced with the June 2011 release). The issue occurred with tasks using the default approval summarizer.
  1. In an instance at the June 2011 release, create an approval on a Change Request.
  2. Login using an iPhone.
  3. Go to My Approval.
  4. Open an Approval and approve the request.
    This action takes you to another screen, indicating that the request is Not Yet Requested.
PRB561340 Auditing Catalog Tasks might have duplicate History entries.
  1. Navigate to Service Catalog > Open Records > Tasks.
  2. Open any task.
  3. Enter some Work Notes and save the record.
  4. Enter more Work Notes and save.
    Each time you enter new Work Notes and save the record, the previous Work Notes are duplicated.
PRB446497 CMDB Business Service Map nodes with excessively long CI names improperly rendered with a scroll bar in MIcrosoft Internet Explorer. Nodes now simply truncate excessively long CI names in the BSM map.
  1. Rename any CI to: testing_display_issue_with_bms_map.
  2. Add the CI by relationship creator to any other CI.
  3. Click Show BSM Map.
    On Internet Explorer 8 the map is displayed with a scroll bar, not allowing user to see the CI's name.
PRB561278 CMDB Setting the cmdb_ci.name field to Unique in the Dictionary did not work as expected. The cmdb_ci.name field can now be set to Unique in the Dictionary if no duplicates exist.
  1. Log into an instance at the Winter 2011 release.
  2. Navigate to the Dictionary and drill into the Name field of the cmdb_ci table.
  3. Check the Unique field and save.
  4. Navigate to any cmdb_ci record and try an Insert and stay from the context menu.
    No error appears, but a duplicate record is created with the same name .
PRB561392 Discovery The Windows classify sensor generated the following TypeError: Cannot convert null to an object.
PRB561484 Discovery The serial number identifier could perform poorly. The identifier no longer tries to query the serial number table if there is no serial number to match against initially.
PRB561810 Importing Scheduled Imports could generate the following error: java.io.IOException: SSHClient() getFile(): temp file does not exist.
PRB561416 Importing Transform Maps could run Business Rules when they were configured to not do so. This issue could occur if the sys_class_name of the imported record was changed in an onBefore script in the Tansform Map.
  1. Set up a Web Service import set via the instructions here: http://wiki.service-now.com/index.php?title=Web_Service_Import_Sets.
  2. Create before insert Business Rules on Task and Incident that use the following: current.description += "\n This is a business rule that is running on incident or task";
  3. Change the text to reflect the table on which the business rule is running.
  4. Load soapUI and use the WSDL for your Web Service import set.
  5. Perform an insert on the task table.
  6. Make sure that you have the Business Rule to run set to false for the Transform Map.
  7. Create a before transform script that sets the following:
if (action == 'Insert') {
  target.sys_class_name = "Incident";
}
This runs the business rules, even though the check box indicates that the business rules are disabled.
PRB561330 Knowledge Base Knowledge Base search results order preference was honored but not reflected in the drop down list when searching from an Incident.
  1. Go to Knowledge Base on demo.service-now.com and select View.
  2. Search for manage.
  3. Note the order in which the documents are returned.
    Although the order is sorted by Relevancy, the value in the Sort By drop down list is dependent on last option selected. If the user previously had brought up the search results and then changed the sorting value to Number of View, the records for the next search appear in Relevancy order, but the Sort By drop down has the value Number of Views.
  4. In demo.service-now.com, select Incident > Create New.
  5. In the Short Description, type manage.
  6. Select the blue, Search Knowledge Icon.
    The Knowledge Search Results window appears. The records again appear in the correct Relevancy Sort order, but the Sort By drop down is set to Number of Views. If the user selects the Search magnifying glass icon, the Sort By drop down changes to the defined value from the Knowledge search result order.
    The expectation is that the value should automatically correspond to the value specified in the Knowledge Base Properties file.
PRB195131 Platform Administration - Fields Mandatory fields could be bypassed for new records if no data had been modified on the form. This issue happened when the user clicked through to a referenced record without modifying anything on the form.
  1. From an existing Incident, create an RTASK. Manually set the Parent to the Incident you're spawning from.
  2. Save the RTASK.
  3. Open the RTASK and blank out the Short Description field.
  4. Add something to Work Notes and Description field.
  5. Hover over the information icon on the Parent field.
  6. When the parent window opens, click on the information icon.
  7. You are taken to the Parent Incident form.
  8. Click on the RTASK you just navigated away from.
    The required Short Description field is still blank, but the Work Notes and Description fields were updated, proving that the record was saved without a required field.

Steps to reproduce from duplicate problem PRB237187

  1. Create a new change request.
  2. Save the change request by right-clicking on the top bar and then choosing Save.
  3. Scroll to the bottom and click on the new button under Task.
  4. Make sure there is a mandatory field on this form. If there is not, create one.
  5. Click on the hover icon next to the change request text box.
    This action bypasses the the mandatory field and saves the task.
  6. Add a client script that sets the modified flag to true for the change_task table
    This stops part of this behavior.
  7. With the script in place, users are prevented from bypassing the mandatory fields by clicking on the hover icon.
  8. Hover the mouse over the hover icon and wait until a pop-up appears.
  9. Click the hover icon, and it will bypass the mandatory fields and save the change task.
PRB559962 Platform Administration- Interceptor Using an Interceptor from a related list of Tasks could wrongly create an extraneous Task record.

From related Incident:

  1. Navigate to any problem record.
  2. Click the New Child Task tab.
  3. Select New ticket type record.
  4. Attempt to begin filling information in various area.
  5. Click on the parent icon to return to the problem ticket.
    Note: User intends to cancel this child ticket creation.
    On the parent problem ticket, notice two new child records created (one ticket type record and one task type record). Child ticket task should NOT be created, because user didn't save this child ticket record explicitly. Also one extra task type record is created.
PRB561661 Platform Administration - Queries Invalid query was generated for an extended field when using ISEMPTY or ISNOTEMPTY as an OR query condition. ISEMPTY or ISNOTEMPTY as an OR query condition using an extended field (e.g., a task list but using incident_state in the query) now generates the correct query.
  1. In a task list, add a list aggregate.
  2. Add a filter to the list that is incident_state NOT IN (1,2) OR incident_state ISEMPTY.
  3. Execute the filter.
    Notice that the SQL query generated to get the aggregation is bad.
  4. Change the query in step 3 to be: incident_state NOT IN (1,2) OR incident_state is.
  5. This is basically the same query in that it checks for a null value in the incident state.
  6. The query works, and the aggregation also works.
PRB560652 Platform Administration - Database There was a simple method to rebuild jRobin databases through the UI when there was a problem with them. A new Rebuild JRobin Databases module has been added to the Performance Definition application.
PRB561198 Platform Administration - Reference Qualifier Read-Only field values were not available to Advanced Reference Qualifiers during new record creation. Disabled records are now serialized properly to make their values available to Advanced Reference Qualifiers.
  1. Select Incident > Create New.
  2. Set a value for the Required User field
    Note that it becomes read-only now.
  3. Click the reference lookup for the User field.
  4. Notice that no users are returned.
  5. Turn off the Turn ReadOnly User Field if Empty UI Policy.
  6. Repeat steps 1-4.
    Notice that now only the user specified by Required User field is returned.
PRB560874 Platform Administration - UI Policy With multiple URL fields on a form, a UI Policy making both read-only does not hide the lock icon for the second one.
  1. Personalized the Incident form and added two additional fields of Type URL.
  2. Create a UI Policy with no conditions.
  3. Create two UI Actions that set the URL fields to read-only.
PRB561527 Platform Administration - UI Policy A UI Policy with a condition on a numeric field could fail for numbers greater than 1000.
PRB561566 Platform Administration - Zing Text Search Potential indexing error occurs in the System Error Log if a record was inserted and then updated in the same second. Add an after insert business rule on an incident that changes the Short description and does an update. This inserts and updates the incident in the same second. When the events are processed, the update event may be processed first and if so, generates the error about a duplicate in the index.
PRB117919 Process Applications - Change Management Save Proposed Changes could forget previously proposed changes when proposing new changes. Previous proposal is now remembered and augmented as expected.
  1. Open the Change Task, and add a server to the list of Affected Ci's.
  2. Right-click and select Proposed Changes.
  3. Update the RAM (MB) value and click Save Proposed Changes.
  4. Repeat the previous step, but this time change the CPU speed(MHZ) value and click Save Proposed Changes again.
  5. Repeat the process for the third time, and on the Server form, you see that the RAM (MB) resets itself back to the original value.
PRB561581 Process Applications - Incident Management Changing the SLA Condition Class property did not affect which Script Include was called to process SLA conditions
  1. Navigate to the Service Level Management Preferences module.
  2. Change the value for the default condition class.
    Note that it doesn't have any effect. The name of the property (com.snc.sla.conditionclass) does not match the name referred to in all the scripts (com.snc.sla.default_conditionclass).
PRB561159 Process Applications - Project Management Lag time in relationships of Project Tasks did not change as expected if user manipulated the Gantt Chart improperly. The Gantt Chart no longer allows dragging of task start if it has already started or if start type is ASAP. The end date can no longer be dragged if the task has ended.
  1. Log into an instance at the Winter 2011 release.
  2. In Project Management, open PRJ0010001.
  3. Click Gantt Chart.
  4. Double-click the relationship between test2 and test5.
    The Planned Task Relationship shows a lag of 1 day.
  5. Move test5 to the right.
    The lag stays at one day even though the task start time has been modified.
PRB561508 Process Applications - Project Management The Planned task rollup field was not properly copied when using the UI Action to copy a partial Project.
  1. Use the Copy partial project context menu option to copy a project that has more than 2 layers of parent > child relationships.
  2. Drill down to a lower level project task with children.
    The rollup field is not set to true and so the Duration field is writeable.
PRB560161 Process Applications - Project Management After re-opening a Project, closing any subtask could close the entire Project.

This procedure is not the only way to reproduce the problem, but demonstrates the issue clearly. Creation of the various elements only needs the Short description field to be filled in.

  1. Create a new project.
  2. Add two new project tasks below the project.
  3. Add two new project tasks below the top two project tasks.
    This results in the following structure:
    project
    - task 1
    - - task 1-1
    - - task 1-2
    - task 2
    - - task 2-1
    - - task 2-2
  4. Go to the project and mark the state as Work-in-progress.
    All tasks in the project are marked Work-in-progress.
  5. Change each sub-task to a state of Closed complete.
    As each sub-task is closed, the relevant Percent complete fields are updated appropriately until all sub-tasks, tasks, and project reach 100% and are closed. This is the normal behavior.
  6. Return to the project and make the state Work-in-progress again.
    All tasks in the project are marked Work-in-progress.
  7. Mark any sub-task as Closed complete.
    The sub-task, task, and the project are marked as Closed complete. This should behave the same as the first time.
PRB561483 Reporting Running a Histogram report could display a broken image (introduced with the June 2011 release).
PRB559620 Reporting Bar Charts and other report types might not display all Chinese characters properly.
  1. Enable Chinese Language (i18n) plugin.
  2. Open the base system bar report All Incidents By State.
    Note that the y-axis label correctly displays Incidents in the y-axis label.
  3. Change to Chinese language.
    Note that squares are displayed instead.
PRB561393 Security Users are unable to authenticate via LDAP. This issue could happen if an inactive OU Definition had the same RDN value as the active one. The inactive OU Definition might be wrongly used in that case.
PRB561360 Security The Encryption Wrapper Plugin might generate an error and fail. This issue could occur when encrypting a password2 type field containing undecryptable (invalid) data. Now the fact is logged and processing continues.
PRB561109 Service Catalog Updating a Catalog Item with a video URL might result in duplication of the video.
  1. Create a Catalog item.
  2. Add a video URL to the body of the item.
  3. Save the item.
  4. Preview the item.
    Notice that it adds the video twice to the body. This has been seen in June 2011 Preview 3 release.
PRB561209 Service Catalog Currency conversion might not be accurate on Catalog Variables. This could occur if a user had a Country Code but no Language specified on their user record.
  1. Create two users, one with Country Code US and the other with UK.
  2. Create a Catalog item, such as a Blackberry or computer, and set the price to any price, such as 1000.00
  3. Open that item in the Catalog with both users and note the prices.
    The conversion rate is inaccurate.
PRB560465 Service Catalog There is a link error after cloning a Service Catalog Request. The link to return to the Request after cloning a Request in the Service Catalog now works as expected.
  1. Log in' as an admin user.
  2. Turn on clone from checkout from Service Catalog > Properties.
  3. Select My Requests (assuming you have created some requests).
  4. Open one of the Requests.
    The checkout summary page appears.
  5. Add a user to one of the clone fields.
  6. Click Order Now.
  7. Once the clone completes, click Request Return to Request: REQXXXXX.
  8. The screen returns to the checkout summary page.
    There are no details in the request list (e.g. Description, Delivery Date, Stage...).
PRB559129 Scripting Hiding a related lists using a Client Script might fail in List v2 (g_form.hideRelatedList('table_name')).
  1. Open an existing incident.
  2. Add the Metrics Related List.
  3. Create an onLoad Client Script on the Incident table that executes the following line: 
    g_form.hideRelatedList("metric_instance");
  4. Reload the incident form.
    Notice that the Metrics related list is not concealed.
PRB561406 SLA Timezones were not being properly associated with SLAs (Introduced with the June 2011 Release).
PRB560995 Surveys Survey Wizards did not properly update the relevant Task Survey record with the name of the user who took the Survey.
  1. Create a survey using the wizard (http://wiki.service-now.com/index.php?title=Creating_a_Survey_Wizard_%28Demo%29).
  2. Configure the survey such that it is triggered by a task, such as closing an incident.
  3. Trigger the survey by closing an incident.
  4. Complete the survey.
    Observe the missing updates in Survey Instances and Task Surveys.
PRB561320 Templates Templates could not properly set trhe Duration fields (introduced with the June 2011 release).
PRB561425 Upgrade Older Service Catalog-related sys_update_xml records might be recreated with the same information after the June 2011 release. Subsequent customers upgrading to the June 2011 release will not see this issue. Upgrade to June 2011 and look at the sys_update_xml table around the time that you started the upgrade.
PRB561315 Web Services An instance could run out of SOAP semaphores, preventing further SOAP requests until the instance was restarted (introduced with the June 2011 release).
