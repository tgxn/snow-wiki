1 Amazon Resource Optimization

Amazon Resource Optimization allows you to track Amazon resource usage through the Amazon CloudWatch API. Resource optimization allows you to generate reports which help you better manage or utilize your resources. The reports show which resources are being under-utilized, stranded, or unattached. You can configure rules to determine when a report is generated for your resources. The resource optimization feature requires Amazon resources to already be discovered and imported into the ServiceNow instance. Discovery is accomplished via the Amazon Web Services account record form.

The application supports optimization of the following resources:

  • EC2 Instances
  • Elastic Block Store (EBS)
  • Elastic Load Balancing (ELB)
 Note: There is a potential charge to use this feature.

2 Resource Optimization Reports

Reports are generated for pre-defined optimization categories. In order for a report to be generated, optimization rules are used to establish conditions that must be met. The Amazon CloudWatch API fetches the necessary metrics to check against these conditions. If the conditions are met, then the report is generated for that category. The following table describes the optimization category, CloudWatch metrics, resource rules, and conditions that must be met.

Optimization Category CloudWatch Metrics Resource Rule Fields Conditions
EC2 - Underutilized CPU Utilization, NetworkIn, NetworkOut CPU Usage and Network IO CPU utilization < CPU Usage and NetworkIn + NetworkOut < NetworkIO
EC2 - Stranded Instances StatusCheckFailed N/A StatusCheckFailed = 1
EBS - Underutilized VolumeReadOps, VolumeWriteOps IOPS VolumeReadOps + VolumeWriteOps < IOPS
EBS - Unattached N/A N/A cmdb's volume_status field = unattached
ELB - Underutilized RequestCount Number of request RequestCount < Number of request
ELB - Unattached HealthyHostCount, UnHealthyHostCount No instance attached HealthyHostCount and UnHealthyHostCount < 1 and No instance attached is selected.
ELB - Attached HealthyHostCount Number of attached healthy instances HealthyHostCount < Number of attached healthy instances

2.1 Generating Reports

To generate reports on-demand, navigate to Amazon Web Services > Configuration >Resource Optimization Rules and click on Generate Report.

After you click on the link, the calculations for the latest reports are made. The latest report is saved in the Resource Optimization Report table with the last executed report time. Once the report generation completes, the latest rows are marked with Current=true' and all prior records are updated to have Current=false. The reports only show resources you can access from the My Amazon Assets and My Team's Amazon Assets portals.

You can view the resource optimization reports on the following dashboards:

  • Cloud Admin Portal
  • My Amazon Assets
  • My Team's Amazon Assets
 Note: The dashboards only show the current records. If you want to view the previous reports, you can modify the gauge to show a previously executed report.


2.2 Scheduling Reports

  1. Navigate to Amazon Web Services > Configuration >Resource Optimization Rules and click on Report Schedule.
  2. Fill in the fields.
  3. Click Update.
Field Description
Name Name of the scheduled report.
Active If selected the scheduled report is active.
Run Frequency of the scheduled report.
Conditional If selected, conditions must be met before the report is run.
Run this script Script used to set conditions for the report to run.

By default, the report scheduler, which runs every Monday, is disabled. To enable it, navigate to Amazon Web Services > Management > Properties and enable the following property: Enable Resource Optimization Report scheduled job.

The scheduled job stops after 24 hours (86400 seconds) or after the amount of time you enter in the following property: The maximum time (in seconds) a resource optimization analysis is allowed to run, before termination.

2.3 Configuring Resource Optimization Rules

The resource optimization rules can be configured as follows:

  1. Navigate to Amazon Web Services > Configuration > Resource Optimization Rules.
  2. Click on a rule.
  3. Make changes as necessary and click Update.
 Note: You cannot create new rules, but you can deactivate an existing rule if you do not want to track the metrics for that particular resource.


For the default EBS Rule:

Field Description
Name Name of the rule.
Volume IOPS < Threshold value that is triggered once the volume IOPS is below this number.
Over a period of days Number of days that the rule will run.
Include unattached volumes If selected, unattached volumes will be included.
Active If selected, the rule is active.
Workflow The workflow that is associated with the rule.

For the default EC2 Rule:

Field Description
Name Name of the rule.
CPU Usage (%) < Threshold for percentage of CPU usage.
and Network IO (MB) < Threshold for size of the Network IO.
For number of days Number of days to use for the rule
and over a period of days Number of days that the rule will run.
Active If selected, the rule is active
Workflow The workflow that is associated with the rule.

For the default ELB Rule

Field Description
Name Name of the rule.
Number of attached healthy instances Threshold for the number of attached healthy instances.
No instance attached If selected, a report will be generated for the ELB - Unattached category.
Number of requests Threshold for the number of requests.
Over a period of days Number of days that the rule will run.
Active If selected, the rule is active.
Workflow The workflow that is associated with the rule.

2.4 Amazon CloudWatch API Limits

See the following link to view the CloudWatch API request limits:

http://docs.aws.amazon.com/AmazonCloudWatch/latest/DeveloperGuide/cloudwatch_limits.html.

CloudWatch metric data is kept for two weeks. Modifying the rules and trying to fetch data older than two weeks will not provide you with any data. The maximum recommended period for any of the rules is 14 days.
