1 Overview

The Verizon eBonding Integration plugin provides Web Service interfaces for both outgoing and incoming messages between the ServiceNow platform and Verizon's Business eBonding Services (VBES). From these interfaces, the base platform provides a Web Service that maps Verizon's trouble ticket to ServiceNow Incidents bi-directionally.

Note: This integration is not a standard plugin that you can enable and configure on-the-fly. You must work jointly with your Verizon Business account representative to set up, configure, and test this integration. We recommend that you initiate contact with Verizon and indicate interest in the Verizon to ServiceNow eBonding integration. Typically, this integration requires a higher level of ServiceNow expertise than that required by a traditional system administrator. A sound knowledge of certificates, business rules, script includes, and integration practices is necessary to complete this integration. Due to the expertise required, we recommend that you engage Professional Services or one of our certified partners to implement this plugin.

2 Architecture

2.1 Inbound to ServiceNow

Web Service import set tables are used to stage incoming SOAP requests to enqueue trouble tickets. The staging data is then transformed into insert or update events against ServiceNow incidents based on coalescing on a unique identifier between the two systems. The correlation_id field in incident is used to contain a unique identifier from the Verizon trouble ticket. This is the field value that is used to coalesce incident records during data transformation.



Function Description
publishETMSTicket When a Verizon trouble ticket is created, the integration creates a ServiceNow incident record by inserting into the ETMS Publish Ticket (vz_etmspublish) import set. The ETMSTicketNumber field is used to update the incident record's correlation_id field and used as the ID for linking the Verizon ticket with the ServiceNow incident record. Multiple posts with the same ETMSTicketNumber value result in the same incident being updated.
publishETMSActivity When an existing Verizon trouble ticket is updated, this Web Service request is posted to ServiceNow, causing a row to be inserted into the ETMS Publish Activity (vz_etmsactivity) import set. The data is then transformed into comments that are added to the e-bonded incident. The incident state is also mapped by a transform script according to this table:
Verizon ResultingState ServiceNow Incident State
RESOLVED Resolved by Verizon
CLOSED Closed by Verizon
DEFERRED Deferred by Verizon
OPEN Active
testInterface A service that is available for Verizon to consume and test connectivity between the two servers.

2.2 Outbound to Verizon eBonding

A collection of SOAP message objects is available to invoke Verizon's Business eBonding Services (VBES) Web Services. These messages are triggered by an update of an existing incident that is linked to a Verizon trouble ticket. The correlation_id field value in incident is used as the unique identifier to update a Verizon trouble ticket.



The following is a list of Verizon Web Services that ServiceNow supports. The Web Services are posted based on the insert or update of an incident and the conditions on the priority or incident_state fields in the record. The invocation of these Web Services are only triggered when the incident Category field is currently set to VZ eBonding.

Function Description Triggering Condition
createTicket When an Incident is created for Verizon, creates a Verizon Trouble Ticket.

There are variations to the createTicket SOAP messages in the form of createTicket(proudct type) that creates messages specifically for the identified product types. For example, createTicketHosting creates a message specifically for the HOSTING product type.

When a new incident is submitted, and its category is set to VZ eBonding
modifyTicket When an existing incident linked to Verizon is updated, modify the Verizon Trouble Ticket. Automatic on incident creation posted from Verizon (publishETMSTicket). Currently only used to update Verizon with a newly created incident number, when an incoming publishETMSTicket creates an incident.
addComment When a comment is added to the incident, update the Verizon ticket with the comment. An additional comment was added, but the priority and incident state values have not changed.
reopen Reopen a Verizon ticket that was closed by Verizon. An incident state changes from a closed state (Closed by Verizon).
requestEscalation Request that a Verizon ticket is escalated. When the priority field of an active incident is increased.
requestCloseCancel Request that the Verizon ticket is closed or cancelled. The incident is updated with the incident state changed to Close Cancel, and Close Code and Close Notes are completed.
requestCloseFixVerified Request that the Verizon ticket is closed because the incident or issue is no longer present. Normally this is in reaction to Closed by Verizon status update. You are required to supply the Close Code and Close Notes. The incident is updated with the incident state changed to Close Fix Verified, and Close Code and Close Notes are completed.
testConnection This Web Service is used for testing connectivity of the Verizon service. This Web Service is triggered by clicking the Test link of the SOAP Message definition.


The following is a list of new incident states that are included in the integration. These values are only available when the incident category is VZ eBonding.

Incident State Description
Close Cancel The incident is closed in a "cancel" mode with one of the following close_code:
Close Code

Problem found on our side, please close

Close Code

Closed by Verizon When Verizon determines that there is no longer an issue, it will invoke the publishETMSActivity Web Service to close the incident. Setting the state to any other non-closed state and updating the incident causes a reopen request to go back to Verizon.
Resolved by Verizon When Verizon determines that the issue is resolved and should be verified as closed, it will invoke the publishETMSActivity Web Service to set this state in the incident. You have the choice to set the state to Close Fix Verified or any other state to reopen the incident.
Deferred by Verizon Verizon is marking this incident as deferred by calling the publishETMSActivity Web Service to update the incident. If you wish to reopen the incident, simply set it back to the Active state. If there is no longer an issue, close the incident by updating it with Close Cancel or Close Fix Verified states.

3 Setup

The first step to integrating with Verizon eBonding is to contact Verizon to begin the setup process. You must acquire eBonding group and service credentials as well as generate a certificate through their system. After completing these steps, proceed with the setup on the ServiceNow side.

3.1 Requesting the Plugin

Click the plus to expand instructions for activating a plugin.

If you have the admin role, use the following steps to activate the plugin.

  1. Navigate to System Definition > Plugins.
  2. Right-click the plugin name on the list and select Activate/Upgrade.
    If the plugin depends on other plugins, these plugins are listed along with their activation status.
  3. [Optional] If available, select the Load demo data check box.
    Some plugins include demo data—sample records that are designed to illustrate plugin features for common use cases. Loading demo data is a good policy when you first activate the plugin on a development or test instance. You can load demo data after the plugin is activated by repeating this process and selecting the check box.
  4. Click Activate.

3.2 Application and Modules

Activate the plugin to install the Verizon eBonding application on your ServiceNow instance. This application is where the integration is configured, and customized.

  • Properties: Configure various properties used by the integration. The "Interface Group ID, User name and Password" is used for communicating to Verizon's eBroker service.
  • Certificate and Keystore: The public certificate and provate key store used to sign and identify web services invocations to Verizon's eBroker service.
  • Client scripts: Scripts that run during Incident management that controls how required fields behave when the Incident category is VZ eBonding
  • Inbound: This section groups web services, import sets and script objects that implement providing the ETMS web service for Verizon to consume.
  • Outbound: This section groups together the outbound web service definitions as well as the script objects that processes Incident creation and update, to invoke Verizon's eBroker web services
  • fMCI Product Types: This section groups together the fMCI or ETMS product types supported by default. These objects are extensions of the Configuration Item
  • fVZ Product Types: This section groups the fVZ or ILEC product types supported by default. These objects are extensions of the Configuration Item

3.4 Certificate and Keystore

This module contains the certificate and keystore entries used for sending secure SOAP messages to Verizon. The public certificate / private key pair is requested and generated with instructions from Verizon at the time of customer on-boarding and provided for uploading. ServiceNow requires the keystore format to be PKCS#12.


3.4.1 Setting up the VZ eBonding Certificate

  1. Contact Verizon for instructions on how to request a certificate for your connection to Verizon.
  2. Generate a base64 encoding of the Verizon certificate. Copy and paste the string into the "PEM Certificate" field.
    The string should begin with -----BEGIN CERTIFICATE----- and end with -----END CERTIFICATE-----.
  3. Save the certificate record in ServiceNow.
    The other fields in the record are generated automatically.

3.4.2 Setting up the VZ eBonding Keystore

PKCS#12 is a binary format used to store server certificates and private keys in a single file that can be encrypted. Because the keystore file generated by the Verizon Certificate issuer only functions in Windows environments, we must covert the keystore information into a PKCS#12 file so that it can be used in our java system.

Follow these steps on a Windows based computer:

3.4.2.1 Step 1: Import the .pfx certificate using a Mozilla-based browser

  1. In FF4.0 go to Options > Options.
  2. In the Advanced section, select the Encryption tab.
  3. Click View Certificates.
  4. In the Certificate Manager window, select the Your Certificates tab.
  5. Click Import.
    A prompt appears, asking you to select an existing file.
  6. Select the certificate file from the location in which you saved it when you exported it.
    A prompt appears, asking for the Master Password.
  7. Enter this password if you have set one.
    A prompt appears, asking for the password used to encrypt the certificate backup.
  8. Enter this password.
    The following prompt appears: Successfully restored your certificate(s) and private key(s).
  9. Click OK.

3.4.2.2 Step 2: Export the certificate as p12 format

  1. In FF4.0 go to Options > Options.
  2. In the Advanced section, select the Encryption tab.
  3. Click View Certificates.
  4. In the Certificate Manager window, select the Your Certificates tab.
  5. To export the certificate, select it, and then click Backup.
    A prompt appears, asking for a filename and location for the PKCS#12-format certificate file.
    The file extension will be .p12.
  6. Provide the requested authentication, and then click OK.
    A dialog box appears requesting the Master Password (the password and certificate database).
  7. If you have set a Master Password, provide it. If not, you can make one up and provide it (optional).
    Remember this password.
  8. If prompted, make up and enter a second password.
    This password is for restoring this particular backup of this certificate. Remember this password.
  9. When the system indicates it has successfully backed up your certificate and private key, click OK.

3.4.2.3 Step 3: Attach the file to the VZ eBonding Keystore record

  1. Removing any existing attachments from the record.
  2. Type in the keystore password that you created when generating the file.
 Note: When you install the ServiceNow/Verizon eBonding plugin, all the corresponding SOAP Message Functions are created by default with the Keystore Alias set to verizon as well as the Keystore Password. You can either set up the keystore so that it matches these default values or set them up according to your own specifications. Remember to change the keystore alias and password values in each SOAP Message Function in the integration.



3.5 Static WSDL (Mandatory)

ServiceNow uses a Static WSDL to connect to Verizon. The Verizon eBonding plugin creates this WSDL, but is set to use localhost rather than the instance in which the plugin was installed.

To change the instance name:

  1. Navigate to Integration - Verizon eBonding > Inbound > WSDL.
  2. Search for localhost in the WSDL.
    It is near the bottom of the WSDL and looks like this:


  3. Change the highlighted portion of the URL to reflect your instance.
    For example, if the instance is mycompany.service-now.com, then the URL would look like:
    https://mycompany.service-now.com/ETMSPublish.do?SOAP

3.10 Common Customizations

3.10.1 Priority

If your incident Priority values are different than what ships out of the box, you will want to change the mapping that takes place to the Priority field in the "PublishTicket" transform map so that Verizon's values map properly to your instance. Likewise, you will need to modify the VZEbonding Script in the getVZPriority function to map your priorities over to Verizon's priority mapping as well.

KNOWN ISSUE: Currently, there is no Priority field mapping in the "PublishTicket" web service transform map. In order to get priority information from Verizon to your instance, you will need to add a "Priority" field in the transform map.

3.10.2 Symptom Codes

Out of the box, the Verizon eBonding integration stores the Verizon-specific Symptom Codes in the "Subcategory" field on the incident form. If your organization wishes to customize this feature and place it in a different field, or place it in the comments, etc, you will need to modify the appropriate transform map to handle the Verizon "Symptom Codes' coming over. Click here to see the Verizon eBonding Symptom Codes.

4 Creating a Reactive Verizon Ticket

When an incident related to a Verizon service or managed asset is created in ServiceNow and transmitted to Verizon for creation is called a "Reactive" ticket. The criteriea for creating a "Reactive" ticket is by setting the Incident's category to VZ eBonding. By setting the category value and selecting the Configuration item from an ETMS or ILEC product type, the ticket will be sent to Verizon. When successful, the Work notes field will contain a log of the progress.

7 Recommended Implementation Steps

7.1 Connectivity

Since connectivity issues require communication from various groups within ServiceNow, Verizon, and the customer, it is a good practice to start with a connectivity plan between your ServiceNow instance and the Verizon Business Web Service.

In most cases, the communication between ServiceNow and Verizon will take place via SOAP Messages that are mutually authenticated and mutually signed and sent over the Internet.

You will need to request a certificate from Verizon, sign that certificate, and then generate a PKCS12 keystore file that you will set up in the Certificate / Keystore module in the Verizon eBonding application of ServiceNow.

7.2 Verizon Service Access

While setting up and testing your Verizon eBonding integration, you will want to be communicating with a Verizon "Test" system. Once you have finalized your integration and moved it to your production ServiceNow instance, and done some basic sanity tests, you will want to switch over to the Live/Production Verizon eBonding service. This will require a different set of Verizon credentials as well as a different certificate and PKCS12 keystore.

You will also need to change your SOAP Message endpoint fields to be the SOAP Endpoints for the production Verizon eBonding service. (By default these fields are set to the Test system at Verizon).

7.3 Moving from Test to Production

Once your integration has been installed, customized, and tested, you will need to migrate over to both the production Verizon eBonding system as well as to your production ServiceNow instance.

For a list of steps you should consider in making that transition, please refer to the ServiceNow/Verizon eBonding Go Live plan.

8 Troubleshooting

8.1 "WSSecurityException: Signature creation failed" in System Log when creating a ticket in SN

If you create a VZ Ebonding ticket in your SN instance and no SOAP Envelope is ever getting sent to Verizon (nor does it even appear in the ECC Queue), then it is best to go to the System Log. Once there, see if you find an exception that looks like the following: 

org.apache.ws.security.WSSecurityException: Signature creation failed; nested exception is:
java.security.spec.InvalidKeySpecException: java.security.InvalidKeyException: IOException : DerInputStream.getLength(): lengthTag=127, too big.
Caused by error in Script Include: 'SOAPMessage' at line 231

If you see this type of exception, it is likely that either your "Keystore Password" and/or "Keystore Alias" are not the same as the values set on each of your "Verizon eBroker" SOAP Message Functions.

To check the values of your Keystore Password and Keystore Alias, you can use the "keytool" command line utility (available on some OS's) to verify your Keystore Password and Alias.

If you have keytool installed, download your PKCS12 file from your instance. Then use the following command line parameters: 

keytool -list -keystore <PUT-YOUR-KEYSTORE-FILE-NAME-HERE> -storetype pkcs12 -storepass <PUT-YOUR-KEYSTORE-PASSWORD-HERE>

The output for this command should like like: 

Keystore type: PKCS12
Keystore provider: SunJSSE

Your keystore contains 1 entry

<YOUR-ALIAS-WILL-BE-LISTED-HERE>, Jun 17, 2011, PrivateKeyEntry, 
Certificate fingerprint (MD5): 12:DD:3E:AE:35:4C:53:56:AF:2B:62:64:D3:44:C5:85
