1 Frequently Asked Questions

1.1 What is a ServiceNow integration?

Integration is the exchange of information between company applications and a ServiceNow instance. Data and process integrations can be categorized as either event-driven or batch-loaded. An event driven integration is implemented for processes with on-demand data requirements and smaller payloads so records can be created to trigger additional work-flow in the system. Event driven integration records usually go through an external queuing table called the ECC Queue. Batch or a data-load based integration is typically done with a scheduled Import Set, where mappings and scripts are defined to gather data from specified sources and transform them into table records.

Processing is triggered by a variety of methods and can occur at different points in the workflow. Integration is achieved using a variety of protocols: SOAP, JSON, JDBC, and ODBC. The ServiceNow platform supports programmatic retrieval of Excel binary format, HTP, XML, and PDF data through an HTTP GET request.

Successful integrations allow ServiceNow customers to:

  • Provide their corporate LDAP directories for user authentication
  • Use ServiceNow data to trigger alerts and notifications
  • Export data for reporting and asset control purposes
  • Work with existing configuration management databases (CMDB) to import CI's into ServiceNow's CMDB

1.2 Do integrations need to be done before the ServiceNow implementation goes live?

You can establish integrations before or after the ServiceNow implementation goes live, however the best practice is to enable integrations before your implementation goes live. Enabling integrations prior to going live allows you to test the integration. This is especially important for integrations to user lists or single sign-on applications since a problem in one of these integrations can prevent users from logging on.

1.3 What technologies are used for integration?

External Authentication technologies can be implemented out of the box to allow fast integration with your portal. Customized scripts are also used for flexibility in the different environments our customers have.

Because of the SOA platform, bi-directional data level integration using Web Services is available to all data objects naturally. The interface is also direct and dynamic because all modifications to existing object, and all new objects are automatically published as a Direct Web Service. A more indirect web service creation and usage can be achieved through Mapped Web Service where a transform map is used to marshal incoming web service data into the final targeted tables. Also, an advanced Scripted Web Service technique is available for defining process based web services, where data is irrelevant, but serves more as a trigger for a process or a composite of actions that execute at the server. Incident Ticketing Integrations contains examples using various methodologies for uni-directional and bi-directional ticketing integrations.

The ServiceNow platform offers a rich interface for loading external data using Import Sets. Using this feature, you can load from various data sources such as HTTP/s, FTP/s, and SCP using file formats such as XML, CSV, and Excel XLS files. A direct JDBC connection to your database is also available as a data source, provided the network connectivity is allowed. Exporting, or exporting data can be done in a variety of formats. See Exporting Data.

Forms, lists, as well as reports on the ServiceNow platform can be accessed directly using a URL, which facilitates integration on the UI level between two or more web applications.

1.5 What are import sets?

The Import Sets workspace provides tables that are a staging area for your data. Data in multiple formats is accepted and transformations can be done before passing the data to your ServiceNow instance.

1.6 How are integrations secured?

Security is one of the first concerns mentioned in doing an integration project. These mechanisms contribute to a secure integration:

  • A ServiceNow implementation connects from a single machine using a fixed IP address and through a specific port on your firewall. This allows firewalls to limit all traffic to a particular IP address and communications port.
  • Many integrations communicate over the HTTPS protocol. This ensures that all communications are encrypted.
  • Mutual Authentication (Single sign-on (SSO)) and other login techniques provide user verification and role assignment.
  • Access Control Lists (ACLs) provide protection of assets that do not need to be exposed in integration processing.
  • The High Security Plugin provides additional security controls.

1.8 What are the performance issues involved with integration?

Overhead costs must be carefully considered as part of an integration project. For example, LDAP refreshes to ServiceNow should be done during off-peak hours to maintain system availability.

1.9 How is integration between ServiceNow instances achieved?

There are times when you find you need to perform a specific integration between your instance and another ServiceNow instance. Instance-to-Instance integrations are a snap because all of the integration points exist between the two instances.

Update Sets also provide exchanges between instances. For more information, see Getting Started with Update Sets.

1.10 What application attributes are required for a ServiceNow instance?

Attributes are defined for the ServiceNow user database. We cannot answer the question of which specific attributes are needed because this is determined by the Business Rules, the scope of the project, and the business requirements that are being solved.

1.11 Do integrations provide real time data?

It depends on the application being integrated. Most integrations use events to trigger processing to happen within seconds. For example, an LDAP integration uses an LDAP Listener to poll for pending work every 15 seconds. Scheduled data exports and scheduled data imports can be configured to run for batch processing or for as-needed individual jobs.

1.12 What integrations support user logins?

Integrations that support a user logging into ServiceNow include techniques used for Mutual Authentication (Single sign-on (SSO)), dynamic user record creation, as well as loading users, groups and location data.

1.13 What is a Single sign-on (SSO) integration?

An Mutual Authentication (Single sign-on (SSO)) integration bypasses the ServiceNow login authentication by allowing a user of your corporate portal to automatically login to the system with centrally managed roles and authorization. An LDAP integration is the most common SSO integration.

1.14 How is user data handled in an LDAP integration?

Using our integration with LDAP or secure LDAP, we can dynamically create user accounts by importing from your Active Directory or OpenLDAP implementation. You can do this securely through a point-to-point VPN. Additionally, you can use the dynamic mapping of data sources such as files or databases using JDBC to import user, groups, and location data into ServiceNow using Import Sets. See Creating a Transform Map.

1.15 What is a user interface integration?

Integration at the user interface level allows standard, seamless embedding of ServiceNow charts and reports into portals, exposure of knowledge base articles into web sites, and the embedding of in-context URLs into other web enabled applications.

Technologies include:

  • Integrating ServiceNow with your Intranet using an IFrame (Web IFrame). For more information on IFrames, see Using Content Blocks.
  • Launch in-context
