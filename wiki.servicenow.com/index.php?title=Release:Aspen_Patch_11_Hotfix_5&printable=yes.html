1 Overview

The Aspen Patch 11 Hotfix 5 release provides fixes for the Aspen release. This release is in a state of controlled availability (see release terminology). For more information about how to upgrade your instance, see Requesting an Upgrade.

For the Aspen Patch 11 Hotfix 5 release:

Build date: 06-28-2013_1809
Build tag: glide-october2011-10-28-2011__patch11-hotfix5-06-27-2013
 Note: See how to determine the version on your instance.

4 Summary

These release notes describe changes and features that are included in the Aspen release. The Aspen release is focused on system scalability and resilience. Enhancements include:

  • Managed Documents: provides new electronic signature support and enhanced knowledge base integration.
  • IT Governance, Risk & Compliance: links risks and controls to entities, tracks authoritative source requirements in audits.
  • Project & Portfolio Management: offers new templates, task calculation and rollup, improved Microsoft Project import, and a new portfolio Gantt chart capability.
  • Zing Text Search: supports wildcards, Boolean operators, and improved international language searches.
  • Workflow: provides a timeline for workflow execution, the ability to roll back to a point in the workflow, and a new debug assistant.
  • Orchestration Automation: offers enhanced process packs for VMWare and Amazon EC2, Windows and Linux password management, and Active Directory user provisioning.

This release is no longer recommended for upgrades. If you want to upgrade an instance, consider a current version. For the Aspen release:

Build date: 12-01-2011_1423
Build tag: glide-october2011-10-28-2011
 Note: See how to determine the version on your instance.


4.1 Release Name & Upgrade Process

The Aspen release introduces a new naming convention based on an alphabetical system (rather than seasons or months). For example, this release is Aspen and the next release name will begin with B.

ServiceNow introduces a new upgrade process with this release. For the first time, early adopters have an opportunity to install this release before it becomes generally available. Also, upgrades will occur over a window of several weeks, rather than all at once. Customers will be notified of their time window prior to any upgrades.

4.2 High Security and Access Control

With the Aspen release, a significant number of new ACLs were added to tables throughout the system. The vast majority of these new ACLs were added in support of the optional Platform Security Settings - High plugin, which includes a property to enforce a default deny security model. In the default deny model, if there is no ACL that grants access to a record, the user does not get access. On the other hand, default allow (the standard for instances created before this release) grants access if there are no ACLs denying access.

Instances using a default deny model need additional ACLs where none existed before. Without them, users might not be able to access required records.

In all application plugins in the Aspen release, we have taken care to ensure that any new ACLs specific to the default deny model are added to an instance only if the Platform Security Settings - High plugin is activated. This plugin is optional for instances created on the June 2011 release and prior.
