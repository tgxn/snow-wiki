1 Overview

You can manage the alert life cycle. This involves managing the alerts as they are received by event management, acknowledging the issues, making work notes, creating associated incidents if necessary, and eventually closing alerts when their associated issues have been resolved.

The evt_mgmt_user role is required to manage the alert life cycle.

2 Managing the Alert Life Cycle

The alert life cycle describes how users manage alerts from when they are created until they are closed. As alerts arrive, users acknowledge them to indicate that they are aware of them and are taking action to resolve them. As users work to resolve the issues, they can communicate with each other and provide a history for the alert by adding work notes. If normal service operation is interrupted due to an issue, a user can create an incident. When the issue is resolved the user can close the alert. The alert closes automatically when the associated incident is closed. The acknowledge alert and generate incident tasks can also be automated. Closed alerts that are not modified for 90 days are deleted by the table cleaner.

To manage the alert life cycle:

  1. Navigate to Event Management > Active Alerts.
  2. Click the alert number.
  3. Edit the alert fields as necessary (see table).
  4. Click the appropriate button:
  • Update: Updates the alert with any changes made on the Alert form.
  • Delete: Deletes the alert from the Alert [em_alert] table.
  • Acknowledge: Indicates that the user is aware of the issue. If an alert is reopened, the previously set acknowledge flag is cleared.
  • Create Incident: Creates an incident for the alert and opens it in the Incident form. If any rules, with or without the auto open flag, match the condition on the alert, the rule is used to create the incident. If nothing matches, an empty incident is created with the short description, description, CMDB ci, and severity copied from the alert. Event management also can be configured to automatically create incidents from alerts that meet defined criteria.
  • Maintenance: Sets an alert to be in maintenance. This Maintenance button is not visible if the alert is in the Closed state.
  • Close: Closes the alert when the issue that caused the associated event has been resolved. Resolving or closing an associated incident also closes the alert.
Alert form

Field Description
Number The unique, automatically generated number of the alert.
Source The source that generated the event, such as Icinga or Nagios.
Node The node associated with the alert.
Resource The affected resource, such as RAM.
Type The event type, such as High CPU.
Category The type of rule that generated the alert. Choices are:
  • Threshold: a threshold alert rule.
  • Regular: a regular alert rule.

This field is available starting with the Fuji release. Upgrading updates existing alerts to have the Regular category. See Alerts.

Severity The severity of the alert. The options are typically interpreted as follows:
  • Critical: Immediate action is required. The resource is either not functional or critical problems are imminent.
  • Major: Major functionality is severely impaired or performance has degraded significantly.
  • Minor: A partial, non-critical loss of functionality or performance degradation has occurred.
  • Warning: Attention is required, but the resource is still functional.
  • Clear: The conditions that caused the alert have been resolved.

The value for this field is copied from the event unless the event closes the alert, in which case the previous severity is retained for reporting.

State The state of the alert: Open, Reopen, Flapping, or Closed.
Acknowledged An indicator that shows whether the alert has been acknowledged.
Maintenance An indicator that shows whether the resource affected by the alert is in maintenance.
Configuration Item The configuration item ID associated with the alert, if any.
Incident The incident number associated with the alert, if any.
Knowledge Article The knowledge article associated with the alert, if any.
Parent The parent alert, if any—that is, any related alerts that have occurred earlier.
Initial event time The time the event that generated the alert first occurred. This time is the ServiceNow processing time, not the source system time.
Last event time The last time the event that is linked to the alert occurred. This time is the ServiceNow processing time, not the source system time.
Initial remote time The time the event that generated the alert first occurred. This time is the source system time and matches the time from the associated event.
Last remote time The last time the event that is linked to the alert occurred. This time is the source system time and matches the time from the associated event.
Created The alert creation time.
Updated The most recent time that the alert information was updated.
Short description A short description of the alert.
Description A description of the alert. This field is copied from the event.
Work notes Work notes describing the alert history.
Flap count The number of times the alert has flapped—that is, has fluctuated between a closed and a non-closed state—within the flap interval since the start time in the Flap start window.
Flap start window The initial start time to measure the flapping occurrences.
Flap last update time The last time flapping occurred. This time is the ServiceNow processing time, not the source system time.
Flap last state The state before the alert entered the flapping state.

3 Maintenance Mode

Alerts have a flag to indicate if the related configuration item is in maintenance. This filters out items that might be undergoing maintenance or upgrades. The Event Management Overview page does not show data related to alerts that are marked as in maintenance.

An alert is also set to be in maintenance if all previous events associated with the alert are in maintenance. If an alert was created with the maintenance mode set to off and the configuration item was later set to maintenance, any subsequent events do not put the alert into maintenance mode. For example, if an alert is created for a router and an hour later the router is taken down for maintenance, it is still assumed there was a problem before maintenance started and the issue should not be ignored.

An alert's maintenance status is reevaluated with each event and is reset if the configuration item status changes.

4 Managing Incidents Associated with Alerts

An alert can be linked to an incident. This link can be created automatically by an alert rule using the incident template, or you can manually create an incident with the Create Incident button. Changes to the alert states can impact the incident states. You can customize these relationships using the system properties.

4.1 Closing an Alert

Closing an alert closes any related incident that is not already resolved or closed. When an alert is closed by an event or a user clicking the Close button, you can control what happens to the related incident:

  • If the incident is resolved or closed, a work note is added to the incident indicating that the related alert was closed.
  • If the incident is not resolved or closed, the system verifies that the incident is not related to any other alerts that are not closed, and resolves the incident, closes the incident, or does nothing, depending on the evt_mgmt.alert_closes_incident system property.

4.2 Reopening an Alert

Reopening an alert reopens an associated incident that is in a resolved or closed state or creates a new incident. When an alert is reopened by an event or a user changing the State to Reopen, the system processes the related incident as follows:

  • If the incident is not resolved or closed, the system adds a work note to the incident indicating that the related alert was reopened.
  • If the incident is resolved or closed, the system checks the evt_mgmt.alert_reopens_incident system property to determine whether to reopen the incident, create a new one, or do nothing:
    • If the incident is reopened, the system adds work notes to the incident.
    • If a new incident is created, the alert is used to find a matching alert rule and create the incident using the template defined. If there is no matching alert rule, the system clones the existing incident and copies the fields into the new incident.

4.3 Resolving or Closing an Incident

Resolving or closing an incident also closes the associated alerts. A background job checks the evt_mgmt.incident_closes_alert system property and closes all alerts related to the incident.
