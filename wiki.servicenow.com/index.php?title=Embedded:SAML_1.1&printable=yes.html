Embedded:SAML 1.1

From ServiceNow Wiki
Embedded:SAML 1.1
Jump to: navigation, search
 Functionality described here requires the SAML 1.1 Single Sign-On plugin.
  • Browser/POST Profile
  • OASIS sstc-saml-tech-overview

Security Assertion Markup Language (SAML) is an XML-based standard for exchanging authentication and authorization data between security domains, that is, between an identity provider (a producer of assertions) and a service provider (a consumer of assertions). SAML is a product of the OASIS Security Services Technical Committee. When implemented correctly, SAML is one of the most secure methods of single single sign-on available. However, it is also one of the most difficult to implement. Those wishing to use SAML should understand the standard as defined by OASIS. More information regarding SAML 1.1 can be found here http://en.wikipedia.org/wiki/SAML_1.1

ServiceNow has created a plugin that enables a ServiceNow instance to accept SAML 1.1 responses. At this time, we support the receiving of SAML responses and not SAML artifacts.



To enable this plugin, please contact support and request the 'SAML 1.1 Single Sign-On' plugin be activated.
Once the plugin is active, a SAML Single Sign-On application will appear. You will need to follow these steps in order to configure the plugin:

  • Click on the x509 Certificate module. Enter the x509 certificate in PEM format in the PEM Certificate field. The other fields should auto-populate based off of the certificate. The certificate entered MUST match the one recieved in the SAML response.
  • Provide your SAML service with a Post Consumer Url which will be the URL to your Instance (eg. https://<instance name>.service-now.com).
  • NOTE: Do not change the SAML 1.1 name for the X.509 certificate. This certificate is referenced by name by the SAML 1.1 integration.



  • Next, configure the properties at SAML Single Sign-on -> Properties.


Property Description
Enable external authentication Uncheck the value to disable SAMLSingle Sign-on.
HTTP header name that contains the base64 encoded SAML value Typically, this property does not need to be changed unless your SAML server posts the response under a different POST variable name.
The name of the XML element that contains the login information Describes the XML element that contains the login information. For example, if the XML contained <SM:Username>ajandersen </SM:Username>, then SM:Username would be the XML element. If the XML contains <saml:Attribute>AttributeName="SMContent"AttributeNamespace=...</saml:attribute>, then the XML element would be saml:Attribute.
If the XML element appears more than once, this value represents which of these XML elements to use. This is a 0-based index so the first occurrence would be offset 0, the second occurrence is 1, etc. If there is more than one XML element that has the name as defined in the last property, then you can define which one to look at using this property. The value of 0 means that the first occurrence of the XML tag is the one to use.
What ServiceNow user field to coalesce against the XML value corresponding to the XML key. Enter the name of the ServiceNow user field that should be used to log in the user. Typically, this is either user_name or email.
Enable replay cache. Activating this property will ensure that the same assertion cannot be used more than one time. For security purposes, this should remain checked (Yes).
Validate assertion conditions. Disabling this property is a security risk and should only be disabled for debugging purposes. Disable this property when debugging SAML or when configuring it for the first time. This setting makes the ServiceNow security check ignore the reply cache, valid time window, etc. Even with this unchecked, the security check still evaluates the signature to make sure it matches what you have uploaded to ServiceNow in the x509 Certificate module.
Enable verbose logging Select this check box to enable expanded logging (larger, more informative log records).


  • Finally, click on the 'Login Script' module and make sure that both External Authentication and SAMLEndPoint are active.


 Note: The difficulty/time involved in enabling an instance to accept SAML 1.1 responses depends on your company's current utilization of SAML for Single Sign-On with other products. If a SAML provider is not currently configured, it typically takes 6-8 weeks for a developer within your organization to configure one. If you do not currently implement SAML, it is often wise to implement one of the aforementioned SSO methods until a SAML solution can be developed.
