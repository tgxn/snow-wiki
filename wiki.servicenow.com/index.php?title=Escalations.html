1 Overview

The out-of-box SLA engine uses Escalations to keep track of SLAs attached to tasks. An escalation is a timer attached to an SLA. As time progresses, the Escalation moves the escalation level higher and higher, until it becomes overdue. Conditions can be set to define when the Escalation starts, when it completes, and when it pauses.

Note: Instances that use the more current Service Level Agreements (SLA) Plugin do not use escalations. Instead, they use SLA Workflows to track the SLA's state.

2 Pausing Escalators

Previously, you couldn't pause an escalator when, for example, a task was waiting on a third party. With the new escalation engine, you are able to define pause conditions for each escalator.

2.1 How to do it

Each escalator now has a new field on it called Pause Condition. Simply fill in a condition there, and update the escalator. Whenever that condition is met, the escalator will be paused. As soon as the ticket changes such that the condition is no longer met, the escalator will unpause and resume counting where it left off.

2.2 Changing Escalators

Before, the escalator that got assigned at task creation stayed with the task until it closed. This could be problematic if, for example, a ticket was initially assigned incorrectly. A ticket which arrives as Priority 4 and gets a (rather slow) Priority 4 escalation shouldn't stay on that escalation path if the ticket is subsequently set to Priority 1.

With the new escalation engine, you have the option of enabling dynamic escalation allocation, which will automatically recalculate the appropriate escalator for you every time a ticket is updated.

2.2.1 How to turn it on

Go into System Properties -> System and find the property titled Assign escalations only at ticket creation (if unchecked, escalations are recomputed at every save/update). (older instances may have the property Should escalations be assigned once at ticket creation (true) or recomputed on every save (false).

Make sure the checkbox is unchecked, and click http://wiki.service-now.com/images/2/2d/NewSave.png at the bottom of the screen.

2.2.2 How Exactly Does Changing an Escalator Work?

Basically it works just like the initial escalator assignment works. Every time you update the ticket, we go out and look at your current record and apply the following steps.

  1. We get all escalators for the appropriate table
  2. We sort them by evaluation order
  3. We compare each one's condition to the state of the current record
  4. As soon as one of the conditions is met, that escalator becomes our escalator

2.2.3 Does it recalculate my existing escalations when this happens?

Yes, it does. So if you change your task from a slow escalator to a fast escalator, it'll be rescheduled based on the fast escalation schedule as soon as the escalator changes.

2.2.4 Does it recalculate my existing escalation level when this happens?

What if I start out on a very fast escalation schedule, which escalates me to level 2, then subsequently discover the task isn't high priority after all and set it to a lower priority (and hence a slower escalation level)? What happens if, under this new, slower escalation level, I should only be level 1?

The short answer is that we set you back to escalation level 1, because that's where you ought to be according to your escalation rules.

2.2.5 Can I use both an SLA and an Inactivity Monitor at the Same Time?

Yes, you can. When a task has both an activity monitor and an SLA on it, the SLA will control the escalation of the ticket just like it always does, but the activity monitor will provide in with additional inactivity events in the event a ticket is inactive for too long.

3 Event Types

3.1 Set

You will get an escalation in the form <tablename>.escalation.set whenever a new escalator is associated with a particular document. This event will provide you with four pieces of information.

  1. Event Name -- <tablename>.escalation.set e.g. incident.escalation.set
  2. Event Instance -- sys_id of the document that this escalation was set to
  3. Event Parameter 1 -- sys_id of the current escalator
  4. Event Parameter 2 -- sys_id of the previous escalator (may be null)

3.2 Pause

You will get an escalation in the form <tablename>.escalation.pause whenever an escalator is paused.

  1. Event Name -- <tablename>.escalation.pause e.g. incident.escalation.pause
  2. Event Instance -- sys_id of the document that this escalation was set to
  3. Event Parameter 1 -- sys_id of the current escalator
  4. Event Parameter 2 -- sys_id of the previous escalator (may be null)

3.3 Resume

You will get an escalation in the form <tablename>.escalation.resume whenever an escalator is taken out of the paused state.

  1. Event Name -- <tablename>.escalation.resume e.g. incident.escalation.resume
  2. Event Instance -- sys_id of the document that this escalation was set to
  3. Event Parameter 1 -- sys_id of the current escalator
  4. Event Parameter 2 -- sys_id of the previous escalator (may be null)

3.4 Fire

You will get an escalation in the form <tablename>.escalation.fired whenever an escalator actually fires and bumps up (or down) the escalation level of your document.

  1. Event Name -- <tablename>.escalation.fire e.g. incident.escalation.fire
  2. Event Instance -- sys_id of the document that this escalation was set to
  3. Event Parameter 1 -- The previous escalation level e.g. 1
  4. Event Parameter 2 -- The current escalation level e.g. 2

3.5 Inactivity

You will get an event in the form <tablename>.inactivity whenever a simple inactivity timer goes off.

  1. Event Name -- <tablename>.inactivity e.g. incident.inactivity
  2. Event Instance -- sys_id of the document that this escalation was set to
  3. Event Parameter 1 -- sys_id of the inactivity monitor that caused this event to fire
  4. Event Parameter 2 -- not specified
