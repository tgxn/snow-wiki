1 Overview

The service management application designer allows you to create custom applications that use a similar service management workflow. Three options are offered for creating these applications--Basic, Intermediate, and Advanced--each with a different set of capabilities. And each of these three methods allow you to create the application with the click of a button or by first tailoring the output to your own particular needs.

The service management application designer is available starting with the Fuji release.

After a service management application has been generated, the following types of files are automatically created, based on the application level and configuration settings selected:

  • Tables
    • Request and task tables (with number prefixes)
    • State flow tables (populated with state flows, and associated with UI actions and business rules)
    • Request and task template tables (and model categories)
  • Roles — copied from the service management roles (admin, basic, initiator, qualifier, dispatcher, agent, approver), prefixed with the name of the service management application
  • Properties and properties categories (same properties as Facilities Management and Facilities Management Geolocation, with unique prefixes)
  • Access Control Lists — copied from the service management ACLs, prefixed with the name of the service management application
  • Service catalog — with a default category
  • Knowledge base
  • Templates (created under the Product Catalog menu)
  • Navigation Menus
  • Service Management CMS application: An icon and text description is added to the CMS site.
 Note: The service management application designer is not supported for the Internet Explorer 8 and 9 browsers. To create custom applications, you must upgrade to at least IE10.


2 Creating a Service Management Application

The service management application designer allows you to create custom applications that use a similar service management workflow. Three options are offered for creating these applications, each with a different set of capabilities:

  • Basic: The basic request management and tracking application employs a request driven process. This allows parent records (that is, work orders, facilities requests, HR cases, etc.) that contain a single task to streamline the request process by displaying information from parent and task records on the same form. The basic application also includes a set of configuration options that complement the request driven process, including approvals, automatically-assigned requests, and catalogs for creating requests.
  • Intermediate: The intermediate application uses a task driven process. This is useful for requests that contain multiple tasks. Intermediate applications include all of the additional functionality inherent in the basic application plus document management.
  • Advanced: Advanced applications are also task driven. They contain all of the capabilities of the basic and intermediate applications, as well as other configurable capabilities, such as part requirements, tracking of travel time, and more.
 Note: Each of these application types includes default settings that you can modify prior to the application generation process. It should be noted, however, that after an application is generated, you can access the Configuration menu option for the application and add or remove capabilities. In essence, you can use the configuration options to convert a basic application to an intermediate or advanced application (or a hybrid). So you are not limited by the default configuration settings.


3 Elements in a Generated Service Management Application

The following elements are created when you use the Service Management Application Designer.

3.1 Tables and Form Layouts

The following diagram illustrates the main table structure of your generated SM application.

The following tables and form layouts are generated:

  • Primary tables
    • Request table
    • Supporting task table
Note: Auditing is turned on for both the request and supporting task tables.
  • Default form layouts
  • Number prefixes for request and supporting task table
  • Process flows provide a visual subway bar to identify the order or request's current state. Completed tasks appear in green.
The states that display depend on your settings on the application's Configuration screen. For example, if you set the Approval slider to OFF, the states relating to approval are not displayed in the process flow formatter, and the request moves to the next state.
Note: The workflow identified in the subway bar represents the default approval workflow. If needed, administrators can create and use their own approval workflows.


3.2 Template Tables

The following diagram illustrates the template table structure of your generated SM application.

The following template tables are generated:

  • Request template table
  • Supporting task template table

3.3 Model Categories

The following model categories are generated:

  • Request model category: This sets the default for the cmdb_model_category field in all request templates.
  • Supporting task model category: This sets the default for the cmdb_model_category field in all supporting task templates.

3.4 State Flows

The following state flow tables and flows are generated:

  • Request flow table and flows
  • Supporting task flow table and flows

3.5 Roles

The following roles are generated. All roles are prefixed with the role prefix provided in the designer.

  • Basic: Reads and creates requests, and follows up on those requests (for example, legal_basic).
  • Initiator: Grants UI access, as well as performing the same functions as Basic (for example, legal_initiator).
  • Agent : Performs work on a task (for example, legal_agent).
  • Dispatcher: Schedules and assigns tasks to agents (for example, legal_dispatcher).
  • Approver user: Approves requests (for example, legal_approver_user).
  • Admin: Controls all data (for example, legal_admin).

3.6 Properties

The following properties are generated:

  • {property_prefix}.autodispatch.skills: Indicates whether skills should be used for auto-dispatch.
  • {property_prefix}.allow.toll.roads: Indicates whether toll roads should be allowed when routing.
  • {property_prefix}.rush.travel.buffer: Percentage to add to all rush hour travel.
  • {property_prefix}.limit.location: Limits the dispatch groups available based on the location of the task.
  • {property_prefix}.evening.rush.hours: Indicates the evening rush hour span.
  • {property_prefix}.autodispatch.location: Indicates whether location should be used for auto-dispatch.
  • {property_prefix}.morning.rush.hours: Indicates the morning rush hour span.
  • {property_prefix}.default.start.time: Identifies the default start time for agents when no schedule is provided.
  • {property_prefix}.autodispatch.geolocation: Indicates whether geolocation should be used to update the estimated travel time for a task.
  • {property_prefix}.work.spacing: Identifies the amount of time (in minutes) to add between tasks.
  • {property_prefix}.default.end.time: Identifies the default end time for agents when no schedule is provided.
  • {property_prefix}.travel.buffer: Identifies the percentage to add to all travel times.
  • {property_prefix}.max.agents.processed: Identifies the maximum number of agents to be processed at a time using auto-dispatch.

3.6.1 Property Categories

The following property categories are generated:

  • Main category: The name of this category is the same as the application label (for example, Legal). This category includes these properties:
    • {property_prefix}.default.end.time
    • {property_prefix}.default.start.time
  • Dispatch category: The name of this category will be a variation of the application label (for example, Legal Dispatch). This category includes these properties:
    • {property_prefix}.max.agents.processed
    • {property_prefix}.work.spacing
    • {property_prefix}.autodispatch.location
    • {property_prefix}.autodispatch.skills
    • {property_prefix}.limit.location
  • Geolocation category: The name of this category will be a variation of the application label (for example, Legal Geolocation). This category includes these properties:
    • {property_prefix}.rush.travel.buffer
    • {property_prefix}.travel.buffer
    • {property_prefix}.allow.toll.roads
    • {property_prefix}.morning.rush.hours
    • {property_prefix}.evening.rush.hours

3.7 Application Menus and Modules

The following application menus and modules are generated:

  • A main application navigation menu (for example, Legal).
  • Overview: The overview screen with default reports.
  • Requests
    • Create New: For creating new requests.
    • Created by Me: For viewing all requests created by the logged-in user.
    • All {Main table label plural}: For viewing the list of all requests (for example, All Legal Requests).
    • Assigned to me: For viewing the list of supporting requests assigned to the logged-in user.
    • Opened - Unassigned: For viewing all open or unassigned requests.
  • Tasks
    • All {Task table label plural}: For viewing the list of all supporting tasks (for example, All Legal Request Tasks).
    • Assigned to me: For viewing the list of supporting tasks assigned to the logged-in user.
    • Opened - Unassigned: For viewing all open or unassigned tasks.
  • Administration
    • Properties: For modifying auto-dispatch properties.
    • Configuration: For modifying the process configuration.
    • {Main table label} Templates: For making requests based on a template.
    • Maintain Catalog Items: For making new requests via the service catalog.
    • Knowledge: For seting up a knowledge base for the application. You can also browse and search for articles that are configured to grant you access, and submit feedback on those articles.

3.7.1 Additional Modules for SM Applications

Depending on the logged-in user's role and configuration settings in the application designer, certain other modules may be displayed in SM applications.

  • If the logged-in user has the approver role, and the Approval is required for new requests option is set, the SM application menu will contain My Approvals.
  • If the logged-in user has the qualifier role, and the Qualification is required for new requests option is set, the SM application menu will contain Awaiting Qualification.
  • If the logged-in user has the dispatcher role, and the Tasks are dispatched via auto-assignment option is set, the SM application menu will contain My Dispatch Queue.

3.8 Other Files

  • A Service Management configuration record (sm_config): Defines all of the relevant information about the application (for example, the main and supporting task table names and the role and property prefixes).
  • Service catalog: If no name is provided, this field defaults to a variation of the application name (for example, Legal Catalog).
    • Default category: If Catalog item on is set to true, any new request template will automatically be associated with a new service catalog item in the default category.
  • Knowledge base with the same name as the application (for example, Legal).
  • Publications
    • Article (for example, Legal Article)
    • Blog (for example, Legal Blog)
    • Wiki (for example, Legal Wiki)
 Note: If any choice of parameters causes duplicate navigation menus, tables, roles, etc. to be created, the SM generator will show an error.
