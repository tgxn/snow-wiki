1 Overview

Discovery CI Consolidation (DCIC) uses Duplicate CI Finders (or “finders” for short) to locate existing CIs in the CMDB that are duplicates of the CI just discovered.  This is the first step in CI consolidation: finding two CIs that look like they should be the same one.

2 How Finders Work

Each time Discovery completes exploration of a CI, it runs the active finders (i.e., those with the “active” checked), in order, until either:

  • One of the finders returns a sys_id, indicating that it has found a duplicate CI
  • All of the finders return null, indicating that none of the finders located a duplicate

This process makes it easy for you to create any number of finders, each with a specific purpose. You might have one finder that locates duplicate Windows Servers using the host name, another finder that locates duplicate Linux servers by their serial numbers, and yet another finder that locates duplicate routers by their short descriptions. There's no limit to how many finders you can have.

Here are a few things to keep in mind about finders:

  • Finders run in order, starting with the finder that has the smallest order value and continuing through the finder with the highest order value. If you have multiple finders with the same order value, then those finders will run in an unpredictable order (with respect to each other).
  • Finders run only if they are marked active.

When a finder returns a sys_id, the duplicate CI consolidator specified in the finder (see below) runs. Its job is to do something (and lots of things are possible) to ensure that the just-discovered CI and the just-found duplicate CI don't remain as duplicates. When the consolidator is finished, the finders run again, starting back at the beginning, on the just-discovered CI – just in case there was more than one duplicate.

3 Creating or Modifying Finders

You can see a list of all the finders in your system by clicking on Discovery Definition → Duplicate CI Finders. If you click on the name of any finder, you'll see a form much like the example below:


The fields:

  • Name.  This is a name you give the finder.  It has no meaning to CI Consolidation.
  • Consolidator.  The Duplicate CI Consolidator that will run if this finder locates a duplicate CI.
  • Order. An integer value that determines the order in which this finder will run.  Smaller values run earlier, larger values later.
  • Active. If this box is checked, the finder will run during CI Consolidation.  If it is unchecked, the finder will not run.
  • Description. The human-readable description of this finder.  It no meaning to CI Consolidation.
  • Script. The JavaScript code that implements your finder.  You must use an anonymous function with exactly the signature shown, but what happens inside that function is entirely up to you.  The value you return determines what happens when your finder finishes.  If your function returns a null, CI Consolidation will know that it did not find a duplicate. On the other hand, if your function returns a CI's sys_id, then CI Consolidation will treat that CI as a duplicate of the just-discovered CI, and will invoke the Duplicate CI Consolidator to take care of it.

4 Understanding the Finder Script

The finder shown above, as well as in the other sample finders supplied with this plug-in, serve as working examples of how to build a finder.  The first thing to understand are the function parameters:

  • source.  This is a hashmap (in other words, a JavaScript object) of the just-discovered CI's record. For fields in the CI's form (as opposed to a related list record), you can get the value simply by writing source.name, where “name” is the name of any field. The value returned is always a string. The example script uses this twice: once to get the sys_id value, and again to get the serial_number value. Records from any related list specified in Discovery Definition → CI Related Lists are also available, though the syntax is a little more complicated. For example, suppose your finder needed to get access to all the serial numbers captured for a computer. If you've specified the serial number related list in CI Related Lists, you'll see that its related list name is cmdb_serial_number:cmdb_ci.  This becomes a field name on the source object, so you can write source.cmdb_serial_number in your script, which will return an instance of CIRelatedList for that list.  That class has several properties and methods (see Discovery Definition → Script Includes, under CI Consolidation, for details).  One example: in your script, you could write source.cmdb_serial_number:cmdb_ci.records, which would return an array of hashmaps, each of which represents one record in the serial number table.
  • results. This is another hashmap, but this time it's for your own use.  There is only one value in the hashmap when your finder runs: “finder”, which contains the name of your finder. This hashmap is here for you to use to pass values from one finder to another, or to the CI Consolidator. The same hashmap is used for all finders, so you can add values to this hashmap in one finder, and read or modify them in another. Likewise, this same hashmap is used for the CI Consolidator, which can then also read or modify these values. You may find this useful for optimization, or for implementing complex finder schemes.
  • log. This is an object that allows you to write messages to the CI Consolidation log (see Discovery Definition → CI Consolidation Log). This object has three methods (info, warn, and error), each of which has a single parameter for the message. You can see this object in use in the example script, on the line just above the return statement.
  • finder.  This is the instance of CIFinder that represents this finder.  It has several methods and properties (see the Script Includes to see its code), including debug() andstop().  The debug() method allows you to write code like finder.debug("my debugging message"), which will write messages to the CI Consolidation log only if the CI Consolidation Debugging property is active (see Discovery Definition → Properties).  The stop() method allows your script to stop all finders (for this CI) after the current finder completes.  Once the current finder finishes (whether with a match or not), no other finders will be run for this CI.

The rest of what happens inside your finder script is entirely up to you.  Typically the script will grab some identifying information from the source object, then use that information with database queries (using the ubiquitous GlideRecord class) to locate any duplicate CIs.  That's exactly what the example script above does.  About the only thing you'll need to be careful about is writing a script that takes a long time to execute, because the finders will run quite frequently during a discovery.
