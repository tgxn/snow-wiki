1 Overview

Administrators in the global domain can modify various aspects of the human resources application. You can configure human resources configurations to:

  • Include or exclude portions of the case process. For example, you can require a case to be approved before continuing to the next stage.
  • Automatically assign tasks using auto-assignment, a custom-defined workflow, or leave it up to dispatchers to manually assign tasks.
  • Track agent travel time.
 Note: Administrators in domains lower than the global domain can view the Human Resources Configurations screen, but cannot modify the settings.


2 Setting Up the Human Resources Configuration

  1. Navigate to Human Resources > Administration > Configuration.
  2. Fill in the fields, as appropriate (see table).
    If you are using a version of Internet Explorer earlier than IE 10, the appearance of the configuration screen will resemble pre-Fuji forms.
  3. Click Save.
 Note: After clicking Save, you must wait until the updated settings have taken effect before submitting another set of changes. To submit more changes, wait a few minutes, reload the page, and try again.

Approval is not required for new requests. Select is required to require cases to be approved before moving to the next state.

If this option set to is required, new cases go from Draft to Awaiting Approval, and when approval is given, cases are set to Ready. If this option is set to not required, new cases go to Ready.

Note: The approval of cases is set to not required by default; however, the approval process for employee change is active by default. This option allows you to make the approval of cases active.

Use this workflow for approvals: select a workflow. This field is displayed if you select Approval is required for new requests. Click the select a workflow link and select the approval workflow you want to use for cases.

If you do not select a workflow, the default workflow, Approval Workflow for SM, is used. This workflow requests an approval from all users with the approver role.

An approver must provide one of the following approval responses before cases can move to the next state:

  • Approved - moves the case to the next state
  • Rejected - cancels the case
  • More information required - moves the case back to the draft state

If you do not select a workflow, the default workflow,Dispatch Workflow for SM is used. This example workflow evaluates each individual in the assignment group for the task and assigns the task to the individual that has the most matching skills.

Additionally, the Estimated work duration field is not mandatory when this configuration option is set to will not.
Auto-selection of agents will consider agent locations Select will to use the agent and location when determining who to assign the task to. Agents closer to the task location will get preference.
Agent auto-selection requires agents to have all of the required skills for the task This option determines the degree to which skills must be matched to a task when determining auto-assignment.
  • Select all to require that an assigned agent must have all of the skills to perform the task. An agent who lacks one skill will be eliminated.

3 Task- vs. Request-Driven Processing

All Service Management (SM) applications, such as Work Management and Facilities Service Management, employ one of two types of processes for handling tasks:

  • task-driven
  • request-driven

Each application defaults to one or the other of these processing types, but you can switch between them as needed.

3.1 Task-Driven Processing

Task-driven processing means that the order or request is simply a container for a list of tasks that need to be completed in order for the overall work to be considered complete. When a work order is created, an associated task record is also automatically created. A request must have at least one task, and additional tasks can be defined to handle all aspects of the request. As tasks are performed and completed, the request transitions through a series of states. After the last task is closed, the request automatically transitions to closed.

3.2 Request-Driven Processing

Request-driven processing means that tasks can still be assigned to a request; however, closing all tasks will not automatically close the request. A request does not require any sub-tasks and can be opened and closed independently. Any sub-tasks added can be transitioned and assigned independently and to different agents than specified on the request. Even if all tasks are closed, the request can remain open and continued to be worked on. However, the request cannot be closed until all tasks are also closed. In request-driven processing, state transitions are based solely on the request.
