1 Overview

Administrators can create a workflow field to display the progress of a workflow. In a list, the workflow field displays stage icons that represent the series of states, stages, phases, or tasks within a workflow. In a form, the workflow field displays the current stage as a choice list value.

When updating a workflow field, always work through the associated workflow. Any other updates made to the workflow field, such as updates from business rules or other scripts, can interfere with displaying workflow stages.

2 Creating a Workflow Field

Administrators can select the workflow type when creating a new field, starting with the Dublin release. If you are using an earlier version, see the previous version information.

  1. Create a field with a type of Workflow.
  2. Right-click the field label and select Personalize Dictionary.
  3. Change the Choice field to Dropdown with --None--.
  4. Click Update.
    Change the Choice field

    On a list, ServiceNow automatically populates the stage icons using the activity stages from the associated workflow. To display workflow stages on a form, see Displaying Stages on a Form.
  5. Add the field to a list layout to see the widget.
    Workflow field on a list

Click the plus to expand previous version information
  1. Create a field with a type of Choice.
  2. Right-click the field label and select Personalize Dictionary.
  3. Change the Type field to workflow, then click Update.
    Change the field type to workflow

  4. Add the choices for your workflow.
    Add one choice for each workflow stage that the workflow field can display.
  5. Add the field to a list layout to see the widget.

2.1 Displaying Stages on a Form

Workflow fields on a form appear as choice lists. To ensure that a workflow field on a form displays the stage properly:

  • Define a choice for each workflow stage. The Value of each choice must match the Value of the corresponding workflow stage.
  • When creating workflow stages to use with a choice list that already has choices, from the graphical workflow editor select the gear icon, select Edit Stages and use the Import Stages from Choice List related link. Selecting this link creates a stage for each choice specified in the Stage field defined for the workflow.
  • Make the choice list read-only. If a user changes the stage value for a record from the choice list rather than allowing the workflow to control the value, the workflow-driven, legacy, and linear renders may not work as intended. You can also use business rules or events to ensure the workflow progresses accordingly.
  • Do not use spaces in the Value field for either choices or stages. Use underscores in place of spaces.

3 Selecting a Stage Field

Selecting a Stage field in a workflow's properties causes the workflow to update that field whenever the workflow reaches an activity that has an associated stage.

To configure a workflow to use a new stage field:

  1. Create a new workflow field.
  2. Navigate to Workflow > Workflow Editor.
  3. Click New or open an existing workflow.
    For an existing workflow, click the gear icon and select Checkout. Then, click the gear icon again and select Properties.
  4. In the Table choice list, select the table that contains the new workflow field.
  5. In the Stage field choice list, select the new workflow field.

4 Displaying Workflow Field Icons

ServiceNow provides several icons to display different stages in a workflow field. Based on the stage renderer selected for a workflow, these icons may display a tooltip with additional information.

4.1 What Each Icon Means

  • Current active step
  • Visited
  • Approval pending
  • Approval approved
  • Approval rejected
  • Complete
  • Late (Change/Request) or Canceled (Catalog)
  • Skipped (Catalog only)
  • On hold
  • Has not yet begun

4.2 Viewing Stage Tooltips

When viewing a stage field, point to the stage icon to see more detailed information about the stage. The behavior of this tooltip depends on the stage renderer selected for the workflow.

  • Legacy Renderer: When a user points to a Waiting for Approval stage, the tooltip displays the name of the approver. A tooltip does not appear when a user points to other stage icons.


  • Other Renderers: When a user points to the active stage on the stage field, the tooltip shows the Name of the activity the workflow is waiting on, for example, Waiting for Approval by Fred Luddy or Waiting for Catalog Task : TASK0000002.

If you do not want the approver's name included, navigate to System Properties > Service Catalog and clear the check box for Show the current pending approver's name in the stage widget mouseover.

5 Rendering Workflow Stages

Workflow renderers allow a workflow to display stages visually in a workflow field. ServiceNow provides multiple rendering options starting with the Dublin Release.These workflow stage renderers are available:

  • Workflow-driven: displays icons for stages using the stage state controlled by the workflow. This renderer shows icons in a way that is meaningful for many situations. This renderer can display stages from a main workflow and subflows. The order of the icons is determined by the expected path of the executing workflow. As the workflow progresses, stages on paths that the workflow did not take are removed from the display. Stages from paths other than the expected path are not included unless the workflow follows that path. Most workflows should use this renderer. New instances running Dublin or later use this renderer by default for all workflows. Only use a different stage renderer to satisfy specific requirements for how the stages appear.
  • Main flow: displays icons for stages defined in the main workflow only. Use this renderer when you do not want to expose the details of the subflows. For example, a single main workflow may run several subflows to handle implementation details. The stages in these subflows do not provide useful information for the user who starts the workflow, but are useful when editing the subflow. In this example, using the main only renderer leads to the best user experience. The stage field displays the high-level process of the workflow without exposing unnecessary details.
  • Linear: displays icons for stages in a linear sequence regardless of the paths the workflow follows as it executes. This renderer uses stages defined in both the main workflow and any subflows that the main flow launches. The icons appear in the user-specified order. Skipped stages do not appear. Use this renderer when the workflow stages must display in a consistent order and the actual details of how the workflow runs are less important. For example, a workflow may revisit or revert to previous stages based on one or more condition activities. Displaying these loops in the stage field does not provide useful information to the user. In this example, using the linear renderer leads to the best user experience. The stage field displays the predefined process, including stages from subflows, in a consistent order.
  • Progress bar: displays a single progress bar instead of a sequence of stage icons. This renderer is particularly useful when you want to display the general progress of the workflow as a percent. Each stage in the workflow controls an equal percentage of the progress bar. For example, if a workflow has 10 stages, reaching each stage causes the workflow field to display an additional 10% of the progress bar. Stage names do not appear in the progress bar. This renderer is available starting with the Eureka release.
  • Legacy: displays stages in the same way as Calgary and earlier releases, and sometimes produces unexpected behavior. When upgrading from a release prior to Dublin, ServiceNow sets all existing workflows to use the legacy renderer. Whenever possible, use a different stage renderer instead.

5.1 Selecting a Stage Icon Renderer

  1. Navigate to Workflow > Workflow Editor.
  2. Click Open and select an existing workflow.
  3. Click the gear icon and select Checkout.
  4. Click the gear icon and select Properties.
  5. Select a renderer from the Stage Rendering choice list.

5.2 Using the Linear Renderer

Linear rendering displays all stage icons from the main workflow and all subflows within a single workflow field on a list or form. The Workflow Stages list for the parent flow and all subflows must contain the same stages. Use a workflow stage set to ensure the parent flow and all subflows have the same stages.

On the workflow canvas, each workflow needs only the stages used directly by that workflow. It is not necessary to add stages from subflows to activities on the parent flow, or stages from the parent flow to activities on the subflows.

To use linear rendering, satisfy these requirements:

  • The parent workflow and all subflows must modify the same current record.
  • The Stage rendering value for the parent workflow and all subflows must be Linear.
  • The Stage field value for all subflows must match that of the parent workflow.

For example, you can create a service catalog workflow that uses a subflow to complete delivery of the requested item. The following image illustrates ordering corporate-branded supplies.

5.3 Using the Legacy Renderer

Workflows that use legacy rendering rely on script includes to populate stages. All workflows on instances prior to the Dublin release use legacy rendering. The Legacy renderer preserves this behavior for the Dublin release and later. Starting with the Dublin release, use an alternative renderer to control the display of workflow icons whenever possible.

When using the legacy renderer, a workflow author can create and reference a custom workflow field icon set by setting the icons attribute to a new script include.

For example, to use the WorkflowIconsSCR script include to define which icons to use, add the attribute icons=WorkflowIconsSCR to the Attributes field of the dictionary entry for the workflow field. To use default icon display behavior, use the attribute icons=WorkflowIconsStages.

5.4 Using the Progress Bar Renderer

The progress bar renderer displays a progress bar instead of stage icons. To correctly display the progress bar, the workflow must have a Stage ordering value of User-defined and workflow stages within the workflow. See Using Workflow Stage Sets for more information about adding stages to a workflow.

As stage names do not appear in a workflow field when using the progress bar renderer, you can simplify stage names to represent a percentage of the workflow. For example, if a process has four main steps, name the workflow stages 25%, 50%, 75%, and 100%. Enter the numerical value of each stage, such as 25, in the Value and Order fields.

The progress bar renderer provides a way to display the status of a workflow that has a large number of stages. Other rendering options may provide a better user experience when rendering fewer than four stages in a single workflow.

5.5 Rendering Multiple Workflow Columns

When using two workflows to update two unique workflow fields on a single record, both workflows must use a non-legacy renderer. The Stage rendering value of each workflow can be different, but do not set either one to Legacy.

6 Displaying Workflow Fields with Deleted Records

Workflow fields may indicate when a user deletes a record required by the workflow starting with the Dublin release. When a user deletes a referenced record, the reference in the primary record is no longer valid. If the stage renderer detects a reference that is no longer valid, the stage field displays a message about the deleted record. Administrators can restore deleted records.
